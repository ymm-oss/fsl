Functional DDD / async effects

domain でドメイン遷移と非同期 effect を仕様として確認する

fsl-domain は、aggregate の所有範囲、command の意図、受理された event、純粋な decide/evolve、非同期 effect lifecycle、saga/process-manager を1つの仕様にまとめる dialect です。仕様は既存 kernel へ lower されて形式検証され、構造的なリスクは安定 JSON の finding として、runtime の実挙動は replay の観測証拠として返ります。

設計の詳細は DESIGN-domain.md と DESIGN-effect.md にあります。

何を防ぐか

ドメイン境界と非同期の破れをリリース前に見つける

実装では、command の受理判定に I/O が混ざる、届いた完了イベントがどのリクエストの完了か分からない、決済のような取り消せない処理が再送で二重に走る、といった破れが起きます。fsl-domain はドメインを有限の遷移系として lower し、こうしたリスクを検証と finding で機械的に返します。

純粋な core の完全性

すべての command に decide が、すべての event に evolve があること、evolve が aggregate 外の状態を書かないことを検査します。

correlation

async effect の完了は、correlation_id で結ばれた pending なリクエストが先に存在することを要求します。宣言が欠ければ uncorrelated_async_completion です。

idempotency

irreversible な effect には idempotency_key が必須です。欠けていれば error finding が形式検証そのものを止めます。

止まらない process

pending のまま離脱手段がない effect、完了観測のない saga step、循環する wait を warning / error として報告します。

モデル

domain は既存カーネルへ lower される

domain は新しい verifier kernel ではありません。宣言を typed IR に読み込み、既存の状態機械 kernel へ展開します。展開結果は fslc domain expand でそのまま FSL として確認できます。

domain の要素展開後の形実際の例（fslc domain expand の出力）
aggregate の state fieldsnake_case の kernel state 変数。Order.status → order_status: OrderStatus
command + decide + evolve1つの kernel action。requires は guard に、rejects E when C は requires not (C) になります。RequestPaymentCapture → order_request_payment_capture(payment_request_id)
event の発生step ごとの Bool flag。直前 step でどの event が起きたかを表します。event_PaymentCaptured: Bool
enum membernamespace 化された member。別の enum が同じ語を使っても衝突しません。Pending → OrderStatus_Pending
aggregate invariant / can(Command)kernel invariant。can(C) は C の decide 前提条件（すべての requires と各 rejects ... when の否定）に展開されます。noDuplicateCapture → Order_noDuplicateCapture
effect の lifecyclecorrelation ごとの有限 map と、completion / retry の kernel action。capture_payment_status: Map<PaymentRequestId, CapturePaymentEffectStatus>、capture_payment_attempts（0..3）
saga の step / compensationevent flag で guard された kernel action と、完了を観測する observe action。saga_order_fulfillment_capture_payment、saga_order_fulfillment_observe_payment_captured

effect の lifecycle は correlation_id を key とする有限状態機械 NotStarted -> Pending -> Succeeded | Failed | TimedOut | Cancelled | Compensated です。completion action は該当 correlation が Pending であることを要求し、retry action は Failed / TimedOut かつ attempts < max_attempts を要求します。Succeeded は sticky で、生成 kernel の transition 検査 CapturePayment_SuccessSticky がそれを担保します。

基本形

aggregate と effect を同じ仕様に書く

decide は state と command から「受理 event か、名前つき reject か」を決める純粋関数、evolve は受理 event を state に適用する純粋関数です。I/O は effect の側だけに置きます。

domain OrderAsyncEffect {
  implementation_profile functional_ddd

  enum OrderStatus { Pending, Approved, Cancelled }
  enum PaymentStatus { NotRequested, PaymentPending, Captured, Failed, TimedOut }

  aggregate Order {
    id OrderId
    state {
      status: OrderStatus = Pending;
      payment_status: PaymentStatus = NotRequested;
    }

    command ApproveOrder {}
    command CancelOrder {}
    command RequestPaymentCapture { input payment_request_id: PaymentRequestId }

    event OrderApproved {}
    event OrderCancelled {}
    event PaymentCaptureRequested { payment_request_id: PaymentRequestId }
    event PaymentCaptured { payment_request_id: PaymentRequestId }
    event PaymentFailed { payment_request_id: PaymentRequestId }
    event PaymentCaptureTimedOut { payment_request_id: PaymentRequestId }

    error CannotCancelCapturedOrder
    error CannotCaptureUnapprovedOrder
    error DuplicatePaymentCapture

    decide ApproveOrder {
      requires status == Pending
      emits OrderApproved
    }

    decide CancelOrder {
      rejects CannotCancelCapturedOrder when payment_status == Captured
      requires status in [Pending, Approved]
      emits OrderCancelled
    }

    decide RequestPaymentCapture {
      rejects CannotCaptureUnapprovedOrder when status != Approved
      rejects DuplicatePaymentCapture when payment_status in [PaymentPending, Captured]
      emits PaymentCaptureRequested
    }

    evolve OrderApproved { status = Approved }
    evolve OrderCancelled { status = Cancelled }
    evolve PaymentCaptureRequested { payment_status = PaymentPending }
    evolve PaymentCaptured {
      requires status != Cancelled
      payment_status = Captured
    }
    evolve PaymentFailed { payment_status = Failed }
    evolve PaymentCaptureTimedOut { payment_status = TimedOut }

    invariant noDuplicateCapture {
      payment_status == Captured -> not can(RequestPaymentCapture)
    }
  }

  effect CapturePayment {
    async
    irreversible
    idempotency_key Order.id
    correlation_id PaymentCaptureRequested.payment_request_id
    handles PaymentCaptureRequested
    emits one_of [PaymentCaptured, PaymentFailed, PaymentCaptureTimedOut]
    retry {
      max_attempts 3
      backoff exponential
    }
    timeout after 10m emits PaymentCaptureTimedOut
    compensation {
      emits PaymentFailed
    }
  }
}
effect の宣言意味
async完了が別の step で届く非同期 effect。correlation_id が必須になります。
irreversible取り消せない副作用（決済など）。idempotency_key が必須で、compensation がなければ warning です。
correlation_id Event.fieldリクエストと完了を結ぶ key。lifecycle map の添字になります。
idempotency_key Order.id二重コミットを防ぐための安定 key の宣言。
handles / emits one_of [...]受けるリクエスト event と、返しうる完了 event の集合。
retry { max_attempts 3 }失敗・タイムアウト後の再試行上限。kernel では attempts < 3 guard になります。
timeout after 10m emits ...pending からの離脱。10m は wall-clock ではなくモデル上の event です。
reliable + outbox配送保証を主張する effect は outbox 境界とペアで書きます。なければ reliable_effect_without_outbox_boundary warning です。

saga

saga で境界越しの process を調整する

複数 step の eventual な process は saga で書きます。starts_on の event で開始し、各 step は emits でリクエストを出し、awaits one_of [...] で完了を待ち、timeout after で離脱します。compensation は「X が Y の後に起きたら Z を出す」という補償を宣言します。

saga OrderFulfillment {
  starts_on OrderApproved
  outbox OrderOutbox
  inbox FulfillmentInbox

  step ReserveInventory {
    async
    emits InventoryReservationRequested
    awaits one_of [InventoryReserved, InventoryReservationFailed]
    timeout after 5m emits InventoryReservationFailed
  }

  step CapturePayment {
    async
    requires InventoryReserved
    emits PaymentCaptureRequested
    awaits one_of [PaymentCaptured, PaymentFailed, PaymentCaptureTimedOut]
    timeout after 10m emits PaymentCaptureTimedOut
  }

  step ShipOrder {
    requires PaymentCaptured
    emits ShipmentRequested
  }

  compensation {
    when PaymentFailed after InventoryReserved {
      emits InventoryReleaseRequested
    }
  }
}

saga の awaits / after は、step ごとの event 観測 action（saga_order_fulfillment_observe_payment_captured のような生成 action）として kernel に入り、結果 JSON に DOMAIN-ASSUME-SAGA-OBSERVED-HISTORY という assumption が付きます。durable な process history は無限の kernel proof としては扱わず、fslc domain replay の有限ログ観測で確認する境界です。

実行

kernel 検証と domain 専用コマンドを使う

domain ファイルは kernel へ lower されるため、通常の fslc check / fslc verify もそのまま通ります。dialect の境界（finding、scaffold、replay）は fslc domain サブコマンドが担います。

fslc check  examples/domain/order_async_effect.fsl
fslc verify examples/domain/order_async_effect.fsl --depth 8
fslc domain check   examples/domain/order_async_effect.fsl
fslc domain analyze examples/domain/order_fulfillment_saga.fsl
fslc domain expand  examples/domain/order_async_effect.fsl
fslc domain generate examples/domain/order_functional_ddd.fsl --target typescript -o src/domain
fslc domain testgen  examples/domain/order_functional_ddd.fsl --target vitest -o order.domain.test.ts
fslc domain replay   examples/domain/order_async_effect.fsl --logs examples/domain/order_async_effect_replay.jsonl
結果読み方
verified_under_assumptionsdomain check の成功。formal_result: "verified" で、kernel の詳細（invariants_checked、action_coverage など）は kernel にネストされ、generated_actions と assumption 一覧が並びます。
violated + formal_result: "not_run"severity error の finding が形式検証の前に止めました。exit code は 1 です。
analyzeddomain analyze。aggregate / effect / saga の所有関係（state、commands、events、steps、compensations、outbox/inbox）の要約 JSON です。
generateddomain generate / domain testgen。生成された files の一覧と出力先を返します。
conformance_checked / nonconformantdomain replay。有限ログの観測結果で、guarantee_kind: "runtime_observed" が付きます。形式証明ではありません。

finding

error は検証を止め、warning は設計リスクを知らせる

finding は fsl-domain-finding.v0 schema の安定 JSON です。severity error は「このまま kernel を回すと保証を過大に主張してしまう」構造的欠陥で、kernel を実行せずに violated を返します。severity warning は設計レビュー上のリスクで、それ単体では kernel の実行を止めません。

findingseverity意味
missing_decide_for_commanderrorcommand に対応する decide がありません。
missing_evolve_for_eventerrorevent に対応する evolve がありません。
aggregate_boundary_violationerrorevolve が aggregate の外の状態を直接書いています。
uncorrelated_async_completionerrorasync effect に correlation_id がありません。replay では「先行リクエストのない完了」も同じ kind で報告されます。
irreversible_effect_without_idempotency_keyerrorirreversible な effect に idempotency_key がありません。
process_wait_cycleerrorsaga の wait graph が循環していて、外部の完了なしには進めません。
pending_effect_without_timeout_or_fallbackwarningpending からの離脱手段（timeout / retry）がありません。
late_completion_without_stale_policywarningキャンセル後などに遅れて届く成功完了への stale policy がありません。
missing_compensation_for_irreversible_effectwarningirreversible な effect に compensation（effect 直付けまたは saga 側）がありません。
reliable_effect_without_outbox_boundarywarningreliable 宣言に outbox 境界が伴っておらず、runtime の配送証拠を過大に主張しています。
saga_dead_endwarningstep のない saga、または完了観測（awaits / timeout）のない async step です。

実例: examples/domain/unsafe_irreversible_effect_without_idempotency.fsl に fslc domain check を実行すると、error 1件（irreversible_effect_without_idempotency_key）と warning 2件（pending_effect_without_timeout_or_fallback、missing_compensation_for_irreversible_effect）が返り、error があるため result: "violated"、formal_result: "not_run" になります。各 finding には repair_candidates（例: add idempotency_key <stable key> to irreversible effect CapturePayment）が付き、LLM の write→verify→repair ループにそのまま渡せます。

scaffold と replay

実装 scaffold と runtime 観測へつなぐ

domain generate

--target typescript が最も充実しています: types.ts、<aggregate>/decide.ts、<aggregate>/evolve.ts、<aggregate>/adapter.ts、effect があれば effects.ts、saga があれば process-manager.ts。python | kotlin | swift | rust は pure-domain の簡易 scaffold です（python は domain_scaffold.py 1ファイル）。

domain testgen

vitest の conformance test scaffold を生成します。Monitor（具象 evaluator）が固定 seed で焼き込んだ trace を replay する形で、adapter（makeAdapter()）を実装に接続するまで各テストは skip されます。既存 testgen と同じ adapter boundary に繋がります。

境界

証明できることと、できないことを分ける

kernel が証明すること

bounded な aggregate invariant、rejected な command が no-op であること（reject 条件は guard に lower されるため状態を変えません）、completion-requires-request、retry bound、success status の stickiness。

有限モデル仮定

DOMAIN-ASSUME-FINITE-DOMAIN-MODEL: 明示宣言のない ID・scalar 入力型は 0..1 の有限範囲としてモデル化されます。無限の id 空間への証明ではありません。

saga history

DOMAIN-ASSUME-SAGA-OBSERVED-HISTORY: awaits / after は step ごとの観測 flag です。durable な process history は replay evidence で確認します。

外部世界

実際の payment gateway、queue delivery、wall-clock time（10m はモデル上の event です）、無限 key 空間にわたる本番 idempotency は証明しません。

生成コード

DOMAIN-ASSUME-GENERATED-SCAFFOLD: 生成物は scaffold であり、本番アーキテクチャの証明ではありません。runtime conformance には adapter / replay の evidence 境界が必要です。

replay は観測証拠

conformance_checked は「この有限ログがモデルと一致した」という観測であり、形式証明ではありません。ログの不在は非発生の証明にはなりません。
