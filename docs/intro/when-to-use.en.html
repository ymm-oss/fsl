When to use it

Whether to use FSL
isn't about how many states you have.

It's about how it breaks. Can some order of operations or combination of flags
slip into a state that must never happen? That's the whole test.

FSL is not a tool to apply to everything.
This page is the line you take home: where it pays off, and where you don't need it. (10 min, no jargon)

▽ Scroll

The one test

Decide by how it breaks, not how many states

The number of states is irrelevant. Ask one thing —
can some order of operations or combination of flags slip into a state that can't legitimately exist?

❌ Not "many states → use it"

A hundred screens on a single linear browse flow rarely hide loopholes. Sheer count is not a reason to reach for FSL.

✅ "Breakable by interaction → use it"

Even 3 states and 2 flags hide loopholes once back, cancel, retry, or permission branching get involved. That's where it pays off.

So the same test fits a tiny confirmation modal and a huge payment flow alike.
The difference isn't scale — only "can order or flags break it?"

Decision flow

Pass three gates, top to bottom

Work down in order. Wherever you get stuck — that's your answer.

① Is there a state machine?

Can you draw boxes (states) and arrows (operations)?
No → static display, simple CRUD, decoration. Out of scope.

↓ Yes

② Can interaction reach a "bad" state?

Do order, flags, permissions, async, or retry combine to slip into a state that must never happen?
No → linear path. Tests usually suffice (low priority).

↓ Yes

③ Is it expressible as finite & discrete?

Are real-time values, probability, continuous quantities, or free text not the core?
No → the core of the model won't fit. Out of scope (table below).

↓ Yes

✅ A candidate for FSL

What remains is whether you can realize the value — covered next.

Where it fits

High-yield, low-priority, and out of scope

The crucial split: "possible but thin (low priority)" versus "not expressible at all (out of scope)." Don't conflate them.

✅ High-yield

  • Payments, refunds, billing
  • Applications, approvals, send-backs
  • Reservations, inventory, allocation
  • Permissions, authorization, audit logs
  • Queues, jobs, async processing
  • SLA, timeout, retry
  • Contract states, plan changes, cancellation
  • Screen transitions / wizards / double-submit guards / unsaved-changes / disable-while-loading

△ Low priority (possible but thin)

  • Static-display-only screens
  • Simple CRUD
  • UI decoration with few constraints
  • Processing with almost no transitions

On a linear path, existing tests usually suffice.

⛔ Out of scope (not expressible)

  • Real-time values, continuous amounts of money
  • Probability, expected value, statistical properties
  • Semantic correctness of free text / prompts
  • Continuous quantities, physical units, org structure
  • Absolute performance (measured latency)

If this is the core of the problem, FSL is at most an aid.
(SLA works only as a relative, discrete step notion.)

So the value isn't fake

FSL checks the spec, not the product

Even past the gates, if the next pieces are missing, a green CI lies to you. "Should we use it" must include whether you can realize the value.

Correct rules are a separate matter. If the rules themselves are wrong, the verdict rests on wrong rules. → Always assign a human owner of the rules.

A hollow spec is false comfort. Zero counterexamples may mean "correct" — or "constrains nothing." → Use fslc mutate kill-rate to confirm the spec actually bites.

If you connect an implementation, the Adapter is everything. If observe() doesn't reflect real state, tests pass while the impl is broken. → Treat the Adapter as a human-review object.

Liveness does not propagate through refinement. Safety (invariants) flows down; response (leadsTo) does not. → Re-verify per layer, or use preserve progress.

If you can't meet these, the right call — even when gates ① and ② pass — is to keep it to lightweight use (an aid for pre-implementation spec review). Don't force it all the way to implementation conformance.
(For the full "not omnipotent" picture, see the concept page.)

Two "should we use it" questions

Per-feature and per-project are different calls

"Write it for this feature now" and "adopt it on the project" are different questions, with different criteria.

Per feature: write it now?

Answering the three core questions is enough.

  • ① Is there a state machine?
  • ② Can order/flags reach a bad state?
  • ③ Is it finite & discrete?

Three yeses → write it. AI drafts the spec, so authoring cost is near zero — drop the old "worth it?" threshold far lower than before.

Per project: adopt it?

Here you ask about the setup.

  • Is there a human owner of the rules?
  • Is there a reviewer for AI-written FSL?
  • (If connecting) can you build an Adapter / logs?
  • Can CI stop on verification failure?

Multiplier: the more you let AI write the state logic, the more value an external standard adds. Start with a 2-week, one-flow PoC.

Which layer

Once you decide to use it, pick a layer

When unsure, start at the requirements layer. Use business only when the As-Is/To-Be control mapping is clear, and design only when an agreed requirements contract exists.

LayerUse it whenMain vocabularyGuide
business Business policy, controls, KPIs, As-Is/To-Be consistency / control preservation actor / stage / policy / goal Business Layer
requirements Acceptance criteria, forbidden flows, NFRs, requirement-ID traceability entity / acceptance / forbidden / REQ-ID Requirements Layer
design Internal state, queues, compose, refinement to requirements, typestate state / action / mapping Design Layer

In one line

Remember this sentence

It's a state machine, the interaction of order/flags/permissions can reach a dangerous state, it's expressible as finite & discrete, and there's a human owner of the rules — that's when FSL pays off.
Size (the number of states) is not a deciding factor.
