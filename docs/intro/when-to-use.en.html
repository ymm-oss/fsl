When to use it

Start from a plain fact: you'd document this anyway.

Any business rule, requirement, or design worth writing down in prose is worth writing in FSL instead — it costs about the same, and now a machine can check it. From there, two payoffs decide how much further to lean in: a single spec's way of breaking — can order or flags interact into a state that must never happen? — and FSL's real edge, that writing business, requirements, and design in it catches the drift between layers too.

Because writing it costs what documenting it would have cost anyway, the default is to write broadly across the layers, not to ration FSL for only the highest-risk spot. It still isn't a tool for everything — you'll also take home exactly what FSL genuinely can't express. (10 min, no jargon)

Payoff ①: within a single spec

Decide by how it breaks, not how many states

First, the within-one-spec question. The number of states is irrelevant. Ask —
can some order of operations or combination of flags slip into a state that can't legitimately exist?

❌ Not "many states → use it"

A hundred screens on a single linear browse flow rarely hide loopholes. Sheer count is not a reason to reach for FSL.

✅ "Breakable by interaction → use it"

Even 3 states and 2 flags hide loopholes once back, cancel, retry, or permission branching get involved. That's where it pays off.

So the same test fits a tiny confirmation modal and a huge payment flow alike.
The difference isn't scale — only "can order or flags break it?"

Payoff ②: cross-layer alignment — FSL's edge

It doesn't stop at verifying one island

Classic formal methods take one hard spot and verify it in isolation (the island model).
FSL is different: it stitches business, requirements, design, and implementation with refinement and checks whether the layers still mean the same thing.

🏝 Classic formal methods = islands

Model one hard spot and ask "is this spec correct?" Keeping the layers aligned stays a manual job.

🔗 FSL = across the layers

Connect business ⊒ requirements ⊒ design ⊒ implementation and verify that a requirement is honored by the design / a control still bites at the lowest layer / an As-Is→To-Be change preserves a control. The question becomes "do these layers still mean the same thing?"

So the more of the contract layers (business, requirements, design) you write in FSL, the more the machine keeps catching cross-layer drift — spec-vs-implementation mismatch. That is FSL managing your app's alignment. Verifying the connection is not an advanced topic — it is one of FSL's primary uses, and combined with the documentation baseline above, it's another reason the default is breadth: writing only the highest-risk island forfeits this payoff entirely.

Decision flow (sizing up payoff ①)

Pass three gates, top to bottom

First decide whether it pays off as a single spec. Work down in order; wherever you get stuck — that's your answer.
(Cross-layer alignment — payoff ② — is the "second lens" just below.)

① Is there a state machine?

Can you draw boxes (states) and arrows (operations)?
No → static display, pure decoration, free-text content. Out of scope — there's nothing here to verify or to write a spec about.

↓ Yes

② Can interaction reach a "bad" state?

Do order, flags, permissions, async, or retry combine to slip into a state that must never happen?
No → verification payoff is low, but write it anyway as a thin spec if it's something you'd document — that's the default. Skip FSL only where you'd write no doc and run no check at all.

↓ Yes

③ Is it expressible as finite & discrete?

Are real-time values, probability, continuous quantities, or free text not the core?
No → the core of the model won't fit. Out of scope (table below).

↓ Yes

✅ A candidate for FSL

What remains is whether you can realize the value — covered next.

The second lens: is there connectivity value? (sizing up payoff ②)

Even if the three gates don't all pass — even if payoff ① alone is thin — it is high-value when cross-layer alignment is the point: a requirement provably honored by the design, a regulatory control that still bites at the lowest layer, an As-Is→To-Be change that preserves a control. When that's the goal, write the layers and verify the seams with chain. This is not an advanced topic — it is a primary use of FSL.

The same lens also says when to stop: the abstraction tax. If there is really only one hard altitude, don't manufacture three layers (you'd just write the same thing three times at different verbosity). Island-shaped hard spots stay single-spec, as before. This guards against fake layering, not against breadth — writing one honest layer per real altitude is still the default.

The wider you write, the more alignment management pays off — but this is a judgment aid, not a mandate to make every task three-layered: write broadly because it's the documentation you'd produce anyway, not because every feature needs all three layers (FSL is the contract spine, not the entire product process).

Where it fits

High-yield, thin-but-worth-it, and out of scope

The crucial split: "thin verification payoff, but still your documentation" versus "not expressible at all (out of scope)." Don't conflate them — thin is a reason to keep the spec small, never a reason to skip writing it if you'd document the thing anyway.

✅ High-yield

  • Payments, refunds, billing
  • Applications, approvals, send-backs
  • Reservations, inventory, allocation
  • Permissions, authorization, audit logs
  • Queues, jobs, async processing
  • SLA, timeout, retry
  • Contract states, plan changes, cancellation
  • Screen transitions / wizards / double-submit guards / unsaved-changes / disable-while-loading

△ Thin verification payoff — write it anyway

  • Static-display-only screens
  • Simple CRUD
  • UI decoration with few constraints
  • Processing with almost no transitions

These rarely hide a loophole worth verifying — but if you'd write a design or requirements note for them anyway, write a thin FSL spec instead: same cost, and now it's checked. Reach for tests alone only where you'd neither document nor verify this at all.
And if it's part of alignment with an upper layer (payoff ②), it's worth writing even when thin on its own.

⛔ Out of scope (not expressible)

  • Real-time values, continuous amounts of money
  • Probability, expected value, statistical properties
  • Semantic correctness of free text / prompts
  • Continuous quantities, physical units, org structure
  • Absolute performance (measured latency)

This bucket is reserved for what FSL cannot express, not for what merely has low verification payoff — that case belongs in the "thin" bucket above, and is still worth writing. If continuous or probabilistic behavior is the core of the problem, FSL is at most an aid.
(SLA works only as a relative, discrete step notion.)

So the value isn't fake

FSL checks the spec, not the product

Even past the gates, if the next pieces are missing, a green CI lies to you. "Should we use it" must include whether you can realize the value.

Correct rules are a separate matter. If the rules themselves are wrong, the verdict rests on wrong rules. → Always assign a human owner of the rules.

A hollow spec is false comfort. Zero counterexamples may mean "correct" — or "constrains nothing." → Use fslc mutate kill-rate to confirm the spec actually bites.

If you connect an implementation, the Adapter is everything. If observe() doesn't reflect real state, tests pass while the impl is broken. → Treat the Adapter as a human-review object.

Liveness does not propagate through refinement. Safety (invariants) flows down; response (leadsTo) does not. → Re-verify per layer, or use preserve progress.

If you can't meet these, the right call — even when gates ① and ② pass — is to keep it to lightweight use (an aid for pre-implementation spec review). Don't force it all the way to implementation conformance.

Two "should we use it" questions

Per-feature and per-project are different calls

"Write it for this feature now" and "adopt it on the project" are different questions, with different criteria.

Per feature: write it now?

Answering the three core questions is enough.

  • ① Is there a state machine?
  • ② Can order/flags reach a bad state?
  • ③ Is it finite & discrete?

Three yeses → write it. Even on two — no on ② — write it anyway if you'd document the thing regardless. AI drafts the spec, so authoring cost is near zero — drop the old "worth it?" threshold far lower than before.

Per project: adopt it?

Here you ask about the setup.

  • Is there a human owner of the rules?
  • Is there a reviewer for AI-written FSL?
  • (If connecting) can you build an Adapter / logs?
  • Can CI stop on verification failure?

Multiplier: the more you let AI write the state logic, the more value an external standard adds. Start with a 2-week, one-flow PoC.

Once it's in CI, the corpus is never "proved once and done": every change re-verifies it, refine/chain turns a change in one layer into a mechanical check of the layers below, and drift or regressions get caught on the next commit — not rediscovered in production. The same corpus doubles as onboarding context, for new engineers and for AI agents alike.

Which layer

Once you decide to use it, pick a layer

When unsure, start at the requirements layer. Use business only when the As-Is/To-Be control mapping is clear, and design only when an agreed requirements contract exists.

In one line

Remember this sentence

If you'd write documentation for it anyway, write it in FSL. On top of that baseline, watch for two payoffs: a single spec's "way of breaking," and/or the "cross-layer drift" from business to design. The latter is the edge classic formal methods lack: the more of the contract layers you write, the more it manages your app's alignment.
Size (the number of states) is not a deciding factor.
