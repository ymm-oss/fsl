Writing the Requirements Layer

Make requirements
readable by counterexample.

The requirements layer carries requirement IDs, inputs, guards, acceptance flows, forbidden flows, and NFRs. It makes business intent concrete enough for design and tests.

1. Boundary

Requirements split business intent into inputs, guards, and examples

The business layer says what the organization allows. The requirements layer says which inputs and conditions move the lifecycle, and which concrete flows must be accepted or rejected.

Business layer

A claim is eventually decided; only approved claims are paid; refund handling can reach a settled state.

Requirements layer

Amount input, thresholds, auto approval, manager approval, acceptance flows, forbidden flows, and NFRs.

Priority: preserve requirement IDs and original text. Counterexamples should tell reviewers which request broke.

2. Basic Shape

Add inputs, guards, updates, and traceability to process

For a single-entity lifecycle, start with process Entity with field: Type.

requirements ExpenseRequirements {
  number Amount
  const AUTO_LIMIT = 1

  process Claim with amount: Amount {
    stages Draft, Submitted, Approved, Rejected, Paid
    initial Draft
    transition submit Draft -> Submitted by Employee
      with a: Amount
      when a > 0
      set amount = a
      covers REQ-1 "The applicant submits an expense claim by entering an amount"

    transition auto_approve Submitted -> Approved by System
      when amount <= AUTO_LIMIT
      covers REQ-2 "Claims at or below AUTO_LIMIT are auto-approved"

    transition mgr_approve Submitted -> Approved by Manager
      when amount > AUTO_LIMIT
      covers REQ-3 "Claims above AUTO_LIMIT are approved by a manager"

    transition reject Submitted -> Rejected by Manager
      when amount > AUTO_LIMIT
      covers REQ-3 "Claims above AUTO_LIMIT may be rejected by a manager"

    transition pay Approved -> Paid by Finance
      covers REQ-4 "Only approved claims are paid"
  }
}

verify {
  instances Claim = 3
  values Amount = 0..3
}
ConstructMeaningGuidance
number AmountA numeric kind.Its finite range is set in verify { values ... }.
process Claim with amount: AmountLifecycle plus carried data.Keep only data needed for requirement decisions.
with a: AmountOperation input.A value passed when the transition runs.
when ...Enabled condition.Thresholds, permissions, state, deadlines.
set amount = aCarried-data update.Stores values needed later.
covers REQ-1 "..."Requirement ID and source text.Flows into diagnostics, counterexamples, and scenarios.

3. Positive and Negative Examples

Use acceptance for must-allow and forbidden for must-reject

A safety invariant can miss under-constraint. Requirements should include both flows that must work and flows that must be rejected.

Acceptance flow

acceptance AC-1 "low-amount claims can be paid" {
  submit(0, 1)
  auto_approve(0)
  pay(0)
  expect Claim 0 in Paid
}

A flow that must be accepted. It is replay-checked and flows into scenarios and test generation.

Forbidden flow

forbidden FB-1 "draft claims cannot be paid" {
  pay(0)
  expect rejected
}

A flow whose final step must be rejected. If it is accepted, a guard is probably too weak.

4. Non-Functional Requirements

Permission, audit, capacity, and deadlines can be modeled here

Many NFRs become ordinary guards or invariants. Deadlines use time and deadline.

NFRHow to write itNote
Permissionwhen or requires role checks.Make who can run the operation explicit.
AuditAudit state plus safety rules.Implementation logs connect later through replay.
CapacityBounded numeric types, Seq capacity, count bounds.Prefer type bounds where possible.
SLA / timeouttime + deadline.Discrete ticks, not wall-clock milliseconds.
Probability, percentiles, usabilityKeep in prose documents.FSL does not directly guarantee them.

SLA pattern

time {
  urgent respond_due
  age wait_age[t: Ticket] while tickets[t] == Accepted
}

requirement NFR-1 "respond within 3 ticks after acceptance" {
  fair action respond_due(t: Ticket) {
    requires tickets[t] == Accepted
    requires wait_age[t] >= SLA_TICKS
    tickets[t] = Responded
  }

  deadline wait_age <= SLA_TICKS
}

SLA trap: if an always-enabled action is urgent, time can freeze and the deadline may pass vacuously. Make only the deadline-reached handler urgent.

5. Business Connection

implements checks that requirements preserve business intent

When requirements implement a business spec, add implements Abs from "file" { }. If names match, FSL can synthesize the mapping.

Name-aligned connection

requirements ExpenseRequirements {
  implements ExpenseToBe from "business.fsl" { }
  ...
}

Keep entity, stage, and transition names aligned when the meaning is the same.

Explicit mapping

implements ExpenseToBe from "business.fsl" {
  maps auto
  map status = req_status
}

Use explicit maps only when names or structure differ.

Verification

$ fslc verify requirements.fsl --depth 8

The JSON includes both requirements verification and the business refinement result.

6. Checklist

Review before design

  1. Does every important transition carry a requirement ID?

    Without covers REQ-..., counterexamples are harder to trace back to source intent.

  2. Are boundary conditions explicit?

    Spell out <= AUTO_LIMIT versus > AUTO_LIMIT. Boundary mistakes are common.

  3. Do you have both acceptance and forbidden flows?

    Positive examples alone can miss weak guards.

  4. Did you separate FSL-checkable NFRs from prose-only NFRs?

    Permission, audit, capacity, and SLA can be modeled. Probability, percentiles, real milliseconds, and usability stay in prose.

  5. Did the layer weaken business intent?

    Treat an implements failure as a possible loss of business meaning.

Next: hand off to design

Once requirements are stable, use spec and refinement mappings for implementation-near state and actions.
