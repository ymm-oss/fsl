Functional DDD / async effects

Describe domain transitions with domain

fsl-domain makes aggregate ownership, command intent, accepted events, pure decide/evolve logic, saga/process-manager coordination, and async effect lifecycles explicit enough to verify, replay, and generate implementation scaffolds.

Implementation details live in DESIGN-domain.md and DESIGN-effect.md.

Model

Keep the domain core pure

Aggregate

Owns state, commands, events, errors, invariants, and the consistency boundary.

Decide

Maps state plus command to accepted events or rejected intent. No I/O belongs here.

Evolve

Applies accepted events to state as pure transition logic.

Effect

Models external async work with correlation, retry, timeout, and idempotency boundaries.

Saga

Coordinates eventual steps, awaits outcomes, and declares compensation across boundaries.

Shape

Write commands, events, and effect outcomes together

domain OrderAsyncEffect {
  implementation_profile functional_ddd

  type OrderStatus = Pending | Approved | Cancelled
  type PaymentStatus = NotRequested | PaymentPending | Captured | Failed | TimedOut

  aggregate Order {
    id OrderId
    state {
      status: OrderStatus = Pending;
      payment_status: PaymentStatus = NotRequested;
    }

    command RequestPaymentCapture { input payment_request_id: PaymentRequestId }
    event PaymentCaptureRequested { payment_request_id: PaymentRequestId }
    event PaymentCaptured { payment_request_id: PaymentRequestId }
    event PaymentFailed { payment_request_id: PaymentRequestId }
    event PaymentCaptureTimedOut { payment_request_id: PaymentRequestId }
    error CannotCaptureUnapprovedOrder

    decide RequestPaymentCapture {
      rejects CannotCaptureUnapprovedOrder when status != Approved
      emits PaymentCaptureRequested
    }

    evolve PaymentCaptureRequested { payment_status = PaymentPending }
    evolve PaymentCaptured {
      requires status != Cancelled
      payment_status = Captured
    }
  }

  effect CapturePayment {
    async
    irreversible
    idempotency_key Order.id
    correlation_id PaymentCaptureRequested.payment_request_id
    handles PaymentCaptureRequested
    emits one_of [PaymentCaptured, PaymentFailed, PaymentCaptureTimedOut]
    retry { max_attempts 3 }
    timeout after 10m emits PaymentCaptureTimedOut
  }
}

Run it

Use kernel verification plus domain-specific findings

fslc check examples/domain/order_async_effect.fsl
fslc verify examples/domain/order_async_effect.fsl --depth 6
fslc domain check examples/domain/order_async_effect.fsl
fslc domain analyze examples/domain/order_async_effect.fsl
fslc domain expand examples/domain/order_async_effect.fsl
fslc domain generate examples/domain/order_functional_ddd.fsl --target typescript -o src/domain
fslc domain generate examples/domain/order_functional_ddd.fsl --target python

ResultMeaning
verified_under_assumptionsThe generated kernel model verified, with finite-model and scaffold assumptions stated in JSON.
violatedA hard domain finding blocked formal verification, or the generated kernel found a real property violation.
warning findingA design risk such as missing timeout/fallback or late completion policy. It does not claim a formal violation by itself.

Boundary

What v0 proves

The verifier checks the modeled transition system: aggregate invariants, command/evolve behavior, saga step guards, completion requiring a pending request, retry bounds, and sticky success. Runtime replay checks finite logs as observation evidence. Neither proves real gateway behavior, queue delivery, wall-clock timeouts, or production exactly-once semantics.
