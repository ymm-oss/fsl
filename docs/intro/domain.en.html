Functional DDD / async effects

Check domain transitions and async effects as a spec with domain

fsl-domain puts aggregate ownership, command intent, accepted events, pure decide/evolve logic, async effect lifecycles, and saga/process-manager coordination into one specification. The spec lowers to the existing kernel for formal verification, structural risks come back as stable JSON findings, and real runtime behavior is checked as replay observation evidence.

Implementation details live in DESIGN-domain.md and DESIGN-effect.md.

What it prevents

Find domain-boundary and async breakage before release

In real implementations, I/O leaks into command acceptance logic, a completion event arrives with no way to tell which request it completes, and an irreversible operation such as a payment capture runs twice after a retry. fsl-domain lowers the domain to a finite transition system and reports these risks mechanically, as verification results and findings.

Pure-core completeness

Every command must have a decide, every event must have an evolve, and no evolve may write state outside its aggregate.

Correlation

An async effect completion requires a prior pending request tied by correlation_id. A missing declaration is uncorrelated_async_completion.

Idempotency

An irreversible effect must declare an idempotency_key. If it is missing, an error finding stops formal verification itself.

Stuck processes

Effects left pending with no exit, saga steps that never observe completion, and cyclic waits are reported as warning or error findings.

Model

domain lowers to the existing kernel

domain is not a new verifier kernel. Declarations are parsed into a typed IR and lowered to the existing state-machine kernel. You can inspect the generated kernel spec directly with fslc domain expand.

domain elementLowered shapeReal example (from fslc domain expand)
Aggregate state fieldA snake_case kernel state variable.Order.status → order_status: OrderStatus
Command + decide + evolveOne kernel action. requires becomes a guard; rejects E when C becomes requires not (C).RequestPaymentCapture → order_request_payment_capture(payment_request_id)
Event occurrenceA per-step Bool flag recording which event just happened.event_PaymentCaptured: Bool
Enum memberA namespaced member, so separate enums can reuse the same word without collision.Pending → OrderStatus_Pending
Aggregate invariant / can(Command)A kernel invariant. can(C) expands to C's decide preconditions: all requires clauses plus the negation of every rejects ... when condition.noDuplicateCapture → Order_noDuplicateCapture
Effect lifecycleFinite per-correlation maps plus completion and retry kernel actions.capture_payment_status: Map<PaymentRequestId, CapturePaymentEffectStatus>, capture_payment_attempts (0..3)
Saga step / compensationKernel actions guarded by event flags, plus observe actions for completions.saga_order_fulfillment_capture_payment, saga_order_fulfillment_observe_payment_captured

The effect lifecycle is a finite state machine keyed by correlation_id: NotStarted -> Pending -> Succeeded | Failed | TimedOut | Cancelled | Compensated. Completion actions require that correlation to be Pending, retry actions require a Failed / TimedOut status with attempts < max_attempts, and Succeeded is sticky — enforced by the generated transition check CapturePayment_SuccessSticky.

Shape

Write the aggregate and its effects in one spec

decide is a pure function from state and command to either accepted events or a named rejection; evolve is a pure function applying an accepted event to state. I/O lives only on the effect side.

domain OrderAsyncEffect {
  implementation_profile functional_ddd

  type OrderStatus = Pending | Approved | Cancelled
  type PaymentStatus = NotRequested | PaymentPending | Captured | Failed | TimedOut

  aggregate Order {
    id OrderId
    state {
      status: OrderStatus = Pending;
      payment_status: PaymentStatus = NotRequested;
    }

    command ApproveOrder {}
    command CancelOrder {}
    command RequestPaymentCapture { input payment_request_id: PaymentRequestId }

    event OrderApproved {}
    event OrderCancelled {}
    event PaymentCaptureRequested { payment_request_id: PaymentRequestId }
    event PaymentCaptured { payment_request_id: PaymentRequestId }
    event PaymentFailed { payment_request_id: PaymentRequestId }
    event PaymentCaptureTimedOut { payment_request_id: PaymentRequestId }

    error CannotCancelCapturedOrder
    error CannotCaptureUnapprovedOrder
    error DuplicatePaymentCapture

    decide ApproveOrder {
      requires status == Pending
      emits OrderApproved
    }

    decide CancelOrder {
      rejects CannotCancelCapturedOrder when payment_status == Captured
      requires status in [Pending, Approved]
      emits OrderCancelled
    }

    decide RequestPaymentCapture {
      rejects CannotCaptureUnapprovedOrder when status != Approved
      rejects DuplicatePaymentCapture when payment_status in [PaymentPending, Captured]
      emits PaymentCaptureRequested
    }

    evolve OrderApproved { status = Approved }
    evolve OrderCancelled { status = Cancelled }
    evolve PaymentCaptureRequested { payment_status = PaymentPending }
    evolve PaymentCaptured {
      requires status != Cancelled
      payment_status = Captured
    }
    evolve PaymentFailed { payment_status = Failed }
    evolve PaymentCaptureTimedOut { payment_status = TimedOut }

    invariant noDuplicateCapture {
      payment_status == Captured -> not can(RequestPaymentCapture)
    }
  }

  effect CapturePayment {
    async
    irreversible
    idempotency_key Order.id
    correlation_id PaymentCaptureRequested.payment_request_id
    handles PaymentCaptureRequested
    emits one_of [PaymentCaptured, PaymentFailed, PaymentCaptureTimedOut]
    retry {
      max_attempts 3
      backoff exponential
    }
    timeout after 10m emits PaymentCaptureTimedOut
    compensation {
      emits PaymentFailed
    }
  }
}
Effect declarationMeaning
asyncAn effect whose completion arrives in a later step. It makes correlation_id mandatory.
irreversibleA side effect that cannot be undone, such as a payment. It requires an idempotency_key, and missing compensation is a warning.
correlation_id Event.fieldThe key tying a request to its completion; it indexes the lifecycle maps.
idempotency_key Order.idA declared stable key that guards against double commits.
handles / emits one_of [...]The request event it consumes and the set of completion events it may produce.
retry { max_attempts 3 }The retry bound after failure or timeout. In the kernel it becomes an attempts < 3 guard.
timeout after 10m emits ...The exit from pending. 10m is a modeled event, not wall-clock time.
reliable + outboxAn effect claiming delivery guarantees must be paired with an outbox boundary; otherwise reliable_effect_without_outbox_boundary is reported.

Saga

Coordinate cross-boundary processes with a saga

Multi-step eventual processes are written as a saga. It starts on a starts_on event; each step sends a request with emits, waits with awaits one_of [...], and exits with timeout after. A compensation block declares "if X happens after Y, emit Z".

saga OrderFulfillment {
  starts_on OrderApproved
  outbox OrderOutbox
  inbox FulfillmentInbox

  step ReserveInventory {
    async
    emits InventoryReservationRequested
    awaits one_of [InventoryReserved, InventoryReservationFailed]
    timeout after 5m emits InventoryReservationFailed
  }

  step CapturePayment {
    async
    requires InventoryReserved
    emits PaymentCaptureRequested
    awaits one_of [PaymentCaptured, PaymentFailed, PaymentCaptureTimedOut]
    timeout after 10m emits PaymentCaptureTimedOut
  }

  step ShipOrder {
    requires PaymentCaptured
    emits ShipmentRequested
  }

  compensation {
    when PaymentFailed after InventoryReserved {
      emits InventoryReleaseRequested
    }
  }
}

Saga awaits / after clauses enter the kernel as per-step event observation actions (generated actions such as saga_order_fulfillment_observe_payment_captured), and the result JSON carries the DOMAIN-ASSUME-SAGA-OBSERVED-HISTORY assumption. Durable process history is not treated as an unbounded kernel proof — it is the boundary checked through finite-log observation with fslc domain replay.

Run it

Use kernel verification plus the domain subcommands

Because domain files lower to the kernel, plain fslc check / fslc verify accept them as-is. The dialect boundary — findings, scaffolds, replay — is exposed by the fslc domain subcommands.

fslc check  examples/domain/order_async_effect.fsl
fslc verify examples/domain/order_async_effect.fsl --depth 8
fslc domain check   examples/domain/order_async_effect.fsl
fslc domain analyze examples/domain/order_fulfillment_saga.fsl
fslc domain expand  examples/domain/order_async_effect.fsl
fslc domain generate examples/domain/order_functional_ddd.fsl --target typescript -o src/domain
fslc domain testgen  examples/domain/order_functional_ddd.fsl --target vitest -o order.domain.test.ts
fslc domain replay   examples/domain/order_async_effect.fsl --logs examples/domain/order_async_effect_replay.jsonl
ResultHow to read it
verified_under_assumptionsA successful domain check: formal_result: "verified", with the kernel details (invariants_checked, action_coverage, …) nested under kernel, alongside generated_actions and the assumption list.
violated + formal_result: "not_run"A severity-error finding stopped formal verification before it ran. Exit code 1.
analyzeddomain analyze: a JSON summary of aggregate / effect / saga ownership (state, commands, events, steps, compensations, outbox/inbox).
generateddomain generate / domain testgen: returns the generated files list and the output location.
conformance_checked / nonconformantdomain replay: a finite-log observation result with guarantee_kind: "runtime_observed". Not a formal proof.

Findings

Errors block verification; warnings flag design risk

Findings are stable JSON under the fsl-domain-finding.v0 schema. Severity error marks a structural defect that would make the kernel run overstate its guarantee, so the check returns violated without running the kernel. Severity warning is a design-review risk and does not block the kernel run by itself.

FindingSeverityMeaning
missing_decide_for_commanderrorA command has no corresponding decide.
missing_evolve_for_eventerrorAn event has no corresponding evolve.
aggregate_boundary_violationerrorAn evolve writes state outside its aggregate.
uncorrelated_async_completionerrorAn async effect has no correlation_id. In replay, a completion with no prior request reports the same kind.
irreversible_effect_without_idempotency_keyerrorAn irreversible effect declares no idempotency_key.
process_wait_cycleerrorThe saga wait graph is cyclic, so no progress is possible without external completion.
pending_effect_without_timeout_or_fallbackwarningNo exit from pending: neither timeout nor retry is declared.
late_completion_without_stale_policywarningNo stale policy for a success completion arriving late, e.g. after cancellation.
missing_compensation_for_irreversible_effectwarningAn irreversible effect has no compensation, either on the effect or in a saga.
reliable_effect_without_outbox_boundarywarningA reliable claim without an outbox boundary overstates runtime delivery evidence.
saga_dead_endwarningA saga with no steps, or an async step with no completion observation (awaits / timeout).

Real example: running fslc domain check on examples/domain/unsafe_irreversible_effect_without_idempotency.fsl returns one error (irreversible_effect_without_idempotency_key) and two warnings (pending_effect_without_timeout_or_fallback, missing_compensation_for_irreversible_effect). Because of the error, the result is violated with formal_result: "not_run". Every finding carries repair_candidates (e.g. add idempotency_key <stable key> to irreversible effect CapturePayment) that feed straight into an LLM write→verify→repair loop.

Scaffold and replay

Connect to implementation scaffolds and runtime observation

domain generate

--target typescript is the richest target: types.ts, <aggregate>/decide.ts, <aggregate>/evolve.ts, <aggregate>/adapter.ts, plus effects.ts when effects are declared and process-manager.ts when sagas are declared. python | kotlin | swift | rust are simple pure-domain scaffolds (python emits a single domain_scaffold.py).

domain testgen

Generates a vitest conformance test scaffold. It replays a trace baked at generation time by the Monitor (the concrete evaluator) under a fixed seed; every test is skipped until you wire makeAdapter() to your implementation. This is the same adapter boundary as the existing testgen conformance tests.

domain replay

Feeds a JSONL / JSON runtime log through the model. The four event kinds are command / domain_event / effect_request / effect_completion.

Boundary

Separate what is proved from what is not

What the kernel proves

Bounded aggregate invariants, rejected commands being no-ops by construction (rejection conditions lower to guards, so no state changes), completion-requires-request, retry bounds, and sticky success status.

Finite-model assumption

DOMAIN-ASSUME-FINITE-DOMAIN-MODEL: IDs and undeclared scalar input types are modeled as finite 0..1 ranges. This is not a proof over unbounded id spaces.

Saga history

DOMAIN-ASSUME-SAGA-OBSERVED-HISTORY: awaits / after are per-step observation flags. Durable process history is checked through replay evidence.

The outside world

Real payment gateways, queue delivery, wall-clock time (10m is a modeled event), and production idempotency across unbounded keys are not proved.

Generated code

DOMAIN-ASSUME-GENERATED-SCAFFOLD: the output is a scaffold, not production architecture proof. Runtime conformance still needs the adapter / replay evidence boundary.

Replay is observation

conformance_checked means this finite log matched the model — observation evidence, not formal proof. Log absence is not proof of non-occurrence.
