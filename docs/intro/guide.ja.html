概念のつぎへ

仕組みと、使い方。

「機械が総当たりで確かめる」——その内側では何が起きているのか。
そして、自分のプロジェクトでどう回すのか。

概念の入門ページを先に読むと、ここからがスムーズです。

▽ スクロール

検証エンジン

検証は2段構え：まず「総当たり」、次に「証明」

FSL の検証器 fslc には2つのモードがあります。順に強くなります。

① 総当たり検査（BMC）

fslc verify --depth N

初期状態から N 手以内の全経路を総当たり。反例が出れば確実にバグ。出なければ「深さ N までは反例なし（verified）」。速くて、反例が具体的。

② k帰納法（無限の深さ）

fslc verify --engine induction

「ルールを満たすどんな状態から1手進めても、ルールは保たれる」ことを示す。成功すれば深さ無限で proved。総当たり検査より厳密。

まず総当たり検査（BMC）：

$ fslc verify order_fixed.fsl --depth 8
{ "result": "verified", "spec": "OrderWorkflow" }   // 深さ8まで反例なし

深さ8まで反例なし。でも「どんなに深くても安全」と言い切れる？ そこで帰納法の出番です。

帰納法を動かす

帰納法は「もし、この状態から始まったら？」を突く

帰納法は、到達できるかに関係なくルールを満たすあらゆる状態を出発点に試します。だから時に、実際には起きない状態を持ち出して「ここから1手で破れる」と指摘します。これが反例候補（CTI）です。下のボタンで実際に動かしてみてください。

🔎 fslc verify --engine induction 
// この1行を足すと proved になる（補助不変条件）
invariant ApprovedImpliesFlag { status == Approved => approved }

※ これも実際の fslc 出力に基づきます（unknown_cti → 補助不変条件を追加 → proved）。

CTI（反例候補）の読み方

CTI は「必ずバグ」ではありません。意味は2通り：
(a) 不変条件が弱い＝実際には起きない状態を、ルールが排除できていない → 補助不変条件で「承認済 ⇒ approved」を教えると proved。
(b) 本物の到達可能なバグ＝その状態に実際に行けてしまう → 仕様か実装を直す。
どちらかを見極めるのが人の仕事です。

縦のつながり

3つの書き方を詳細化チェックでつなぐ

同じ対象領域を粒度ちがいで書き、詳細化チェックで接続します。FSL ではこの確認を refinement と呼びます。上の層の意図が下の層で保たれているかを fslc refine が機械的に照合し、要件IDが全層の診断に伝わります。

業務層 business

規則・統制・KPI・ゴール。業務としての矛盾、禁止状態への到達、ゴール到達可能性を確認。

↓ 詳細化チェック

要件層 requirements

要件ID・受け入れ条件・禁止フロー・非機能要件（権限/監査/SLA等）。反例から元の要望へ遡れる。

↓ 詳細化チェック

設計層 design

状態機械・データ・非同期・合成。安全性・到達性・応答性・写像で要件充足を照合。

↓ シナリオ生成・テスト生成・ログ再生

実装・品質確認

シナリオ・pytest雛形を生成。実装ログの適合も確認。

fslc refine impl.fsl abs.fsl mapping.fsl で1段の詳細化を、fslc chain fsl-project.toml で全段を一括検証できます。

これは実装前に一度だけ回すものではなく、CIの上で回してください。コミットのたびにコーパス全体が再検証され、ある層への変更は下位層への機械的なチェックに変換され、仕様と実装のズレはインシデントとして表面化する前にその場で捕まります。同じコーパスは、新しく入るエンジニアや、コードを引き継ぐAIエージェントが、システムの振る舞いを理解するために最初に読むものにもなります。

道具箱

fslc コマンド早見

出力はすべて機械可読なJSON（AIが読んで直すため）。終了コードは 0＝verified/proved/refines…、1＝violated/反例、2＝仕様エラー。

書く・検証する

check — 構文・型のみ（高速ループ）
verify --depth N — 総当たり検査・最短反例
verify --engine induction — 無限の深さで証明

仕様の質を測る

--vacuity error — 空虚な性質を検出
--strict-tags — タグ漏れ・未参照の検査
mutate — 制約の強さを測る
explain — 検証範囲・反実仮想

実装につなぐ

scenarios — 統合テスト雛形JSON
testgen -o test.py — pytest用のテスト雛形
replay --trace log.json — ログ適合チェック
typestate --ts — 状態の遷移を型に変換

階層をつなぐ

refine impl abs map — 詳細化の照合
chain project.toml — 業務→実装を一括
合成した仕様（compose）の verify も可能

実際に回す

書く → 検証する → 直す、を回す

FSL は「人が文法を覚えて書く」前提ではありません。AIが下書きし、fslc が反例を返し、AIが直す。あなたは意図と最終判断を担います。

01

意図を渡す

「未承認は出荷しない」など守りたいことを言葉で。

02

AIが下書き

スキルが文法と修復手順を供給。.fsl を生成。

03

fslc が反例

矛盾・抜け穴・到達不能を JSON で返す。

04

AIが修復

反例を読み、破れなくなるまで反復。

AIが読むのは、こういう JSON（反例の手順データ）：

{
  "result": "violated",
  "invariant": "ShipOnlyAfterApproval",
  "violated_at_step": 2,
  "last_action": { "name": "ship" },
  "trace": [ /* Draft → submit → Submitted → ship → Shipped */ ]
}

導入は3コマンド（詳細は README）：

# 1. 入れる（fslc とスキルが入る）
$ bash install.sh

# 2. AIに頼む（スキル経由）
「$fsl-requirements で『承認前は出荷不可・返金は二重に走らない』要件を書いて、
 検証して、反例が出たら直して、無くなるまで回して」

# 3. 必要ならテスト雛形と実装連携も生成させる
$ fslc testgen spec.fsl -o tests/test_conformance.py

適用

どこから始める？

リスクの高い所から始めるのは、あくまで着手順の話であってゴールではありません。仕様コーパスはプロジェクトのドキュメントそのものであり、失敗が痛い所だけに配分する希少資源ではないので、目指す最終状態は業務・要件・設計を通じた広いカバレッジです。

🎯 リスクの高い基幹

従来の形式手法に近い使い方。

決済・返金 / 申請・承認 / 予約・在庫引当 / 権限・監査 / キュー・非同期 / SLA・タイムアウト。まずはここから——間違うといちばん痛い所。

🧩 小さな状態管理

AIが下書きするから割に合う使い方。

画面遷移・戻る/取消・確認モーダル / 未保存・保存中 / ウィザード / 二重送信防止 / retry・pending・failed / 権限による操作可否。状態の抜け穴の事前点検に。

判断基準は「状態の数」ではなく、「操作順やフラグの組み合わせで、意図しない状態に入れてしまうか」。3状態・2フラグでも、戻る/取消/再試行/権限分岐があるなら FSL 化する価値があります。まずは2週間で終わる小さな試作検証から始め、そこからカバレッジを広げ続けてください——コーパスは一度きりの演習ではなく、プロジェクトが生きているドキュメントであり続けます。

触ってみる

小さな仕様をひとつ、AIに書かせて fslc verify にかけてみてください。
反例が返ってくる体験が、いちばんの理解になります。
