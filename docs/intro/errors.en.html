Errors & Exit Codes

"Read the counterexample and fix it" is FSL's core loop. This page indexes the result/kind/violation_kind vocabulary, the exit-code contract, and how to read each failure shape.

The contract

The JSON envelope and exit codes

Every command prints one JSON object to stdout, and its result field maps to a process exit code. That mapping (exit_code()) is reproduced verbatim on the CLI Reference page — here is the short version.

Exit codeMeaningTypical result
0Successverified / proved / refines / conformant / generated / typestate / mutated / explained / ok, etc.
1The check ran to completion but didn't holdviolated / reachable_failed / unknown_cti / nonconformant / refinement_failed, etc.
2Spec errorresult: "error" with kind in parse/name/type/semantics/io (also vacuity errors under --vacuity error)
3Internal errorresult: "error" with kind: "internal"

What to read even when it succeeds

What to check even when result is a success

warnings

verified/proved don't guarantee an empty warnings array — reachable deadlocks and similar non-bug-but-worth-knowing facts land here.

completeness

"bounded" means "no counterexample up to this depth"; "unbounded" means induction proved it at infinite depth. See Workflow for the difference.

unknown_cti

k-induction returned a counterexample-to-induction. Reachability is undetermined; adding an auxiliary invariant is the standard move — see Mechanisms.

action_coverage

Flags actions that were never enabled during the search — a way to catch an action you thought was reachable but actually isn't.
