Get Started

Five minutes to reading a counterexample.

Install, verify a broken spec, read the counterexample JSON, fix it, and go all the way to a proof. Every command here is copy-pasteable.

New to FSL? Read the concept introduction first for the intuition.

0. Install

Two dependencies: lark and z3-solver

No C++ compiler or separate Z3 install needed. Mac/Windows/Linux all work with plain pip install (Python 3.9+).

git clone https://github.com/ymm-oss/fsl && cd fsl
python3 -m venv .venv && source .venv/bin/activate
pip install -e ".[dev]"   # lark, z3-solver, pytest + an editable install of fslc

1. check

The fast loop first: syntax and types only

fslc check never calls Z3 — it only checks syntax, names, and types. Run it on every edit.

$ fslc check docs/intro/specs/order_buggy.fsl
{
  "fsl": "1.0",
  "result": "ok",
  "spec": "OrderWorkflow",
  "warnings": []
}

Syntax and types are fine. But this spec has a rule violation baked in — check doesn't look that far.

2. verify — violated

Now bounded model checking: let the machine find the counterexample

fslc verify tries every possible sequence of actions up to depth 8 with Z3. If a rule breaks, it returns the shortest counterexample.

$ fslc verify docs/intro/specs/order_buggy.fsl --depth 8
{
  "fsl": "1.0",
  "result": "violated",
  "spec": "OrderWorkflow",
  "violation_kind": "invariant",
  "invariant": "ShipOnlyAfterApproval",
  "violated_at_step": 2,
  "last_action": { "name": "ship", "params": {} },
  "trace": [
    { "step": 0, "state": { "status": "Draft", "approved": false } },
    { "step": 1, "state": { "status": "Submitted", "approved": false },
      "action": { "name": "submit" },
      "changes": { "status": { "from": "Draft", "to": "Submitted" } } },
    { "step": 2, "state": { "status": "Shipped", "approved": false },
      "action": { "name": "ship" },
      "changes": { "status": { "from": "Submitted", "to": "Shipped" } } }
  ]
}

Reading this counterexample: since result is "violated", invariant and trace are all you need. Two steps — submit, then ship — reach Shipped with approved still false, breaking ShipOnlyAfterApproval: the workflow can ship without ever being approved.

3. Fix it

A one-line guard

The ship action's requires was guarding against the wrong state.

  action ship() {
-   requires status == Submitted   // BUG: should require Approved
+   requires status == Approved    // FIX: ship only after approval
    status = Shipped
  }

4. verify — verified

Verify the fixed spec at depth 8 again

$ fslc verify docs/intro/specs/order_fixed.fsl --depth 8
{
  "fsl": "1.0",
  "result": "verified",
  "spec": "OrderWorkflow",
  "checked_to_depth": 8,
  "completeness": "bounded",
  "invariants_checked": ["_bounds_status", "ShipOnlyAfterApproval"],
  "action_coverage": { "submit": true, "approve": true, "ship": true, "cancel": true },
  "warnings": [
    { "message": "deadlock reachable at step 3 (state: status=Shipped, approved=true)",
      "hint": "add an enabled action, declare intended stops in a terminal { } block, or use --deadlock=ignore if intentional" }
  ]
}

Read the warnings too: result: "verified" does not mean warnings is empty. The deadlock warning here just means the verifier doesn't yet know that Shipped is meant to be a terminal state (say so with a terminal { } block). It's not a bug — but don't get into the habit of skipping past warnings unread.

verified is a bounded result: no counterexample up to depth 8. To check to infinite depth, the next step is induction.

5. verify --engine induction

Finally, a proof: to infinite depth

order_proved.fsl adds one auxiliary invariant, which lets k-induction prove the property at infinite depth.

$ fslc verify docs/intro/specs/order_proved.fsl --engine induction
{
  "fsl": "1.0",
  "result": "proved",
  "spec": "OrderWorkflow",
  "engine": "induction",
  "completeness": "unbounded",
  "invariants_checked": ["_bounds_status", "ShipOnlyAfterApproval", "ApprovedImpliesFlag"],
  "warnings": []
}

completeness: "unbounded" means ShipOnlyAfterApproval is now proved to hold no matter how long a run gets. See Workflow for how to get from BMC to proved in your own specs.
