AI hard-contracts / statistical evidence

One discipline for AI tool boundaries, agent structure, and statistical quality

fsl-ai covers AI systems in three layers. ai_component is the hard-contract layer: tool authority, human approval, and forbidden tools checked through kernel expansion. agent is the structural layer: recursive agent grants, visibility, and orchestration. And dataset / evaluator / statistical_property / ai_migration / observed_property form the evidence layer, which treats statistical quality, migration regression, and operational drift as external evidence with formal_result: "not_run".

Implementation details live in DESIGN-ai-hard.md and DESIGN-stochastic.md.

What it prevents

Catch four kinds of "it looks fine" AI failures

fsl-ai does not try to prove that natural-language answers are true. Instead, it separates boundaries that can be checked deterministically by pre-execution guards from claims that must be handled honestly as statistics or observation, and reports each failure with a dedicated finding.

Tool-boundary breaches

Forbidden tool execution and irreversible execution without human approval are detected both by kernel invariants and by runtime replay.

Agent authority leaks

Structural analysis detects child grants exceeding the parent boundary, visibility leaks across siblings, low-trust paths reaching high-authority tools, and review-gate bypass.

Point-estimate release decisions

A bare "accuracy was 0.92" claim is rejected. Nothing becomes statistically_supported unless the Wilson interval's lower bound supports the threshold.

Silent regression and drift

Metric regressions from prompt/model/retriever migrations are caught by fslc ai regress; production telemetry threshold breaches and drift by fslc ai drift — both as threshold-bearing evidence.

Guarantee boundary

Fix upfront what may be called proved — and what may not

The core of fsl-ai is a guarantee boundary that fixes, per contract class, how the claim is handled and which vocabulary reports it. Every result and finding carries this classification.

Contract classExamplesHandlingResult vocabulary
syntactic / structural hardTool schema, authority, forbidden tool, human approval token, symbolic precondition evidence.Static check, kernel expansion, runtime guard/replay finding.verified_under_assumptions, violated, finding kind ai_hard_contract_violation.
agent structuralNested agent grants, visibility, delegation, review gates, tool reachability.Structural evidence from graph analysis; not proof.agent_analyzed, finding kind agent_structural_violation.
evaluator-backedGroundedness, source support, prompt-injection judgment.External evaluator evidence; not kernel proof.evaluator_supported; never displayed as proved.
statisticalAccuracy, recall, hallucination rate, per-slice metrics.External stochastic evidence layer (fslc ai eval / regress).statistically_supported / statistically_unsupported.
observedUndeclared tool execution, schema mismatch in logs, production telemetry drift.fslc ai replay / fslc ai drift observation evidence only.replay_conformant / replay_nonconformant, observed_supported / observed_mismatch, finding kind observed_contract_violation.

Prompt injection and RAG groundedness may be called "hard" only when the checked predicate is structural and guard-backed — for example "no tool call executes when the event marks schema_valid=false" or "a citation ID is in a declared finite allow-list." A semantic claim such as "the answer is supported by the source" requires an evaluator and cannot be displayed as formal proof.

Hard contract

Declare tools and authority explicitly with ai_component

The hard-contract layer models an AI component only at its observable tool boundary: model/prompt/input/output metadata, tool declarations with schema names, symbolic preconditions, authority, fallback, and rule selection via check hard.

ai_component RefundAgentToolSafety {
  model refund_model_v1;
  prompt refund_prompt_v1;
  input RefundRequestV1;
  output RefundDecisionV1;

  tool SearchOrder {
    schema SearchOrderV1;
    precondition order_exists;
    effect read_order;
  }

  tool RefundPayment irreversible {
    schema RefundPaymentV1;
    precondition order_paid;
    precondition amount_refundable;
    effect payment_refunded;
  }

  tool DeleteCustomerData irreversible {
    schema DeleteCustomerDataV1;
    effect customer_data_deleted;
  }

  authority {
    may_execute SearchOrder;
    requires_human_approval RefundPayment;
    forbidden DeleteCustomerData;
  }

  fallback {
    when low_confidence require human_review;
    when missing_order_evidence require refuse_with_reason;
  }

  check hard {
    rule tool_authority;
    rule human_approval_required;
    rule forbidden_tool_blocked;
    rule tool_schema_declared;
    rule tool_precondition_declared;
  }
}
RuleWhat it checks
tool_authorityA forbidden tool cannot also be executable.
human_approval_requiredAn irreversible executable tool must be in requires_human_approval.
forbidden_tool_blockedThe generated kernel state machine has no transition that can make a forbidden tool's tool_executed entry true.
tool_schema_declaredExecutable tools must declare a schema name.
tool_precondition_declaredDeclared symbolic preconditions are checked during replay as business precondition evidence, distinct from tool-schema conformance.

Omitting check hard checks all five rules — the safe default. Also, unlike the business/requirements dialects, no field on ai_component accepts a "description text" tag: everything here is a bare identifier or number.

Lowering

ai_component lowers to the existing kernel

The AI dialect adds no probability, percentile, or evaluator semantics to the verifier. The hard contract becomes finite state and ordinary invariants, checked by fslc ai check (BMC or k-induction underneath).

Finite tool state

A Tool enum, Map<Tool, Bool> state for human_approved, tool_executed, and tool_suggested, plus fallback_required: Bool become ordinary kernel state.

Generated actions

suggest_*, approve_*, execute_*, and fallback_* actions model approval tokens and guarded execution.

Generated invariants

Forbidden tools never execute, and irreversible or approval-required tools execute only after approval.

Fallback is structural only

when <reason> require <target> lowers to a fallback_<reason> action and ghost state, but target is a free label, so no invariant is generated — this is not proof that the fallback route was actually taken. Each reason must be unique per component.

Recursive agents

Nested agents are scoped agents, not sub_agents

agent Parent { agent Child { ... } } creates a lexical namespace such as Parent.Child. Parent authority and context are never implicitly inherited; children receive them through explicit grant declarations, which must stay inside the immediate parent's boundary. Runtime collaboration is the separate orchestration graph.

agent SupportOrchestrator {
  context [CustomerTicket, ApprovedSupportDocs];
  tools [SearchDocs, CheckPolicy, SendEmail];
  authority {
    may_execute [SearchDocs, CheckPolicy];
    requires_human_approval [SendEmail];
  }
  review_gate PolicyCheckAgent;

  agent RetrievalAgent {
    trust medium;
    grant authority [SearchDocs];
    grant context [ApprovedSupportDocs];
    tools [SearchDocs];
    authority { may_execute [SearchDocs]; }
    output RetrievedSources visibility [parent, PolicyCheckAgent];
  }

  agent PolicyCheckAgent {
    trust high;
    grant authority [CheckPolicy];
    grant context [CustomerTicket, ApprovedSupportDocs];
    tools [CheckPolicy];
    authority { may_execute [CheckPolicy]; }
    contract { hard { rule PolicyDecisionCited; } }
    output PolicyDecision visibility parent;
  }

  agent SendAgent {
    trust high;
    grant authority [SendEmail];
    grant context [CustomerTicket];
    tool SendEmail irreversible {
      schema SendEmailV1;
    }
    authority { requires_human_approval [SendEmail]; }
    output SendResult visibility parent;
  }

  orchestration {
    RetrievalAgent -> PolicyCheckAgent;
    PolicyCheckAgent -> SendAgent;
  }

  failure_policy {
    when RetrievalAgent.failed -> retry up_to 2;
    when RetrievalAgent.failed_after_retry -> HumanReviewPending;
  }
}

fslc ai check analyzes this as agent_analyzed (formal_result: "not_run") and returns six separated graphs in graph_summary — scope_tree, delegation_graph, authority_graph, information_flow_graph, tool_reachability_graph, and failure_policy. Structural violations are reported as finding kind agent_structural_violation.

ViolationMeaning
child_authority_exceeds_parent_authorityA child's grants/tools/authority exceed the immediate parent boundary.
child_context_exceeds_parent_contextA child uses context beyond its granted context.
visibility_leak_across_sibling_agentsOutput visibility to a sibling with no delegation path.
low_trust_agent_path_to_high_authority_toolA path from a trust low agent to a high-authority tool. Changing RetrievalAgent in the snippet above to trust low really does return this finding.
irreversible_operation_without_human_approval_pathA path to an irreversible tool without requires_human_approval.
policy_review_bypass_in_orchestrationAn orchestration path reaching high-authority tooling that skips all declared review_gate children.

Mind the exact scope. trust is a free identifier; only the literal low has a dedicated check (low_trust_agent_path_to_high_authority_tool) — medium and high parse but have no dedicated check yet. contract { hard { rule <Name>; } } is parsed and echoed under each agent's agent_ir.contracts (e.g. {"hard_rules": ["PolicyDecisionCited"]}) but is not yet cross-checked against anything — it is forward-declared metadata. As with ai_component, no field here accepts a "description text" tag.

Statistical evidence

Bind quality to evidence with dataset / evaluator / statistical_property

fslc ai eval is a deterministic checker over precomputed eval JSONL. It runs no AI model and no evaluator; it supports only Bernoulli/proportion metrics (accuracy, recall, hallucination_rate, wrong_tool_call_rate, and other 0/1 metrics), judged with Wilson intervals. Results always carry formal_result: "not_run".

dataset SupportEvalV3 {
  source "examples/ai/support_eval_v3.jsonl"

  population {
    language in ["ja", "en"]
    ticket_type in ["refund", "bug", "urgent"]
  }

  slice JapaneseRefundTickets {
    language == "ja"
    ticket_type == "refund"
  }
}

evaluator SupportAnswerJudge {
  input question: Question
  input answer: Answer
  output grounded: Bool
  output confidence: Float

  calibration {
    dataset HumanLabeledGroundednessV2
    require agreement_with_human >= 0.90
  }
}

statistical_property LooseQuality {
  target SupportAnswerAgent
  dataset SupportEvalV3
  evaluator SupportAnswerJudge
  confidence 0.95

  require ci_lower(metric.accuracy, 0.95) >= 0.45

  slice JapaneseRefundTickets {
    require min_samples >= 5
    require ci_lower(metric.accuracy, 0.95) >= 0.35
  }
}

One eval record is one Bernoulli observation (schema fsl-ai-eval-record.v0; case_id, slice, metric, boolean outcome, and evaluator.id are required, and duplicate (case_id, slice, metric) records are dataset_invalid):

{"schema_version":"fsl-ai-eval-record.v0","case_id":"support-001","component":"SupportAnswerAgent","dataset":"SupportEvalV3","slice":"all","metric":"accuracy","outcome":true,"evaluator":{"id":"SupportAnswerJudge","calibration_status":"trusted"}}

With k successes out of n observations, estimate = k / n, and the Wilson interval at confidence c yields ci_lower / ci_upper. Requirements must take the form ci_lower(metric.X, 0.95) >= T or ci_upper(metric.X, 0.95) <= T. Actually running the examples gives:

PropertySlicenEstimateci_lower (95%)ThresholdResult
LooseQualityall100.80≈ 0.490≥ 0.45statistically_supported
LooseQualityJapaneseRefundTickets51.00≈ 0.566≥ 0.35 (+ min_samples >= 5)statistically_supported
StrictQualityJapaneseRefundTickets51.00≈ 0.566≥ 0.80statistically_unsupported

The third row is the point. Even a perfect 5/5 on the slice (estimate 1.00) gives a Wilson lower bound of only about 0.566 at n=5, so a 0.80 claim is not supported. Rejecting "it all passed, so we're fine" — sample size included — is this layer's job.

PriorityStatusMeaning
1dataset_invalidRecord schema violation, duplicate case for a slice/metric, or a required slice field is missing.
2evaluator_untrustedEvaluator calibration evidence is missing or below the declared trust threshold.
3insufficient_samplesSome required slice has n < min_samples.
4inconclusiveThe declared metric or interval method cannot be computed. Point-estimate-only requirements are rejected here too.
5statistically_unsupportedEnough samples, but the Wilson bound does not support the threshold.
6statistically_supportedEvery required gate and bound check passes.

A point-estimate-only requirement such as require accuracy >= 0.92 is rejected at eval time as inconclusive (exit 1) — it does not pass with a warning. Each slice is an independent gate: the overall result is supported only when every slice passes its own min_samples and bound check. No family-wise multiple-testing correction is claimed by this layer.

Migration / drift / compat

Migration regression, production drift, and environment compatibility under the same discipline

ai_migration declares, across a prompt/model/retriever migration, that the hard contract is preserved and metrics do not regress; observed_property declares production telemetry thresholds and drift. Both are external evidence with formal_result: "not_run".

ai_migration PromptV7ToV8 {
  from SupportAnswerAgent {
    model gpt_5_5
    prompt support_answer_prompt_v7
    retriever support_docs_index_v12
  }

  to SupportAnswerAgent {
    model gpt_5_5
    prompt support_answer_prompt_v8
    retriever support_docs_index_v14
  }

  preserve {
    hard_contract SupportAnswerContract.hard

    no_regression {
      dataset SupportEvalV3
      metric accuracy drop <= 0.05
      metric hallucination_rate increase <= 0.02
    }
  }
}

observed_property SupportAgentOperationalQuality {
  target SupportAnswerAgent
  source production_logs
  window last_7_days

  require observed(metric.hallucination_rate) <= 0.30
  require drift(metric.refusal_rate) <= 0.10 compared_to previous_7_days
}

  requires model.gpt_5_5, prompt.support_answer_prompt_v8,

Fold this fragment into a dbsystem environment and check it with fslc compat check --include-ai — AI capabilities join server/mobile/DB artifacts in the same finite snapshot compatibility model. A provider gap is reported as required_capability_missing.

Finding contract

Every finding has the same shape: fsl-ai-finding.v0

Hard-contract violations, agent structural violations, unsupported statistics, and drift all come back through one finding schema. The required fields are fsl, result, kind, severity, component, contract, tool, failed_rule, violation, guarantee_kind, evidence, witness, minimal_conflict_set, repair_candidates, and assumptions. Each repair candidate carries a weakens_spec flag, so "make it pass by weakening the spec" repairs are explicitly marked.

guarantee_kindEmitted byMeaning
syntactic_hardHard-contract violations from ai check / ai replayDeterministic facts over declared boundaries and guard assumptions.
agent_structuralRecursive agent analysisStructural evidence over graphs; not proof.
runtime_observedLog mismatches in replay, ai driftObservation evidence; absence from logs is not proof of unused behavior.
evaluator_supportedFuture evaluator findingsEvaluator evidence; never displayed as proved.
statistically_supported / statistically_unsupportedai eval / ai regressWilson-bound statistical evidence; never displayed as proved.

Boundary

Do not confuse what is checked with what is merely recognized

Unchecked block boundaries

ai_action, a standalone retriever block, trust_boundary, and a top-level authority { target ... } (the example's ai_contract behaves the same way) are recognized only as block boundaries. The parser never descends into their bodies, so even garbage inside passes check. They are echoed as bare {kind, name} entries under raw_blocks and constrain nothing.

The checked surface, exactly

Kernel-backed ai_component / agent, plus external-evidence dataset / evaluator / statistical_property / ai_migration / observed_property. Writing a contract anywhere else has no effect.

Forward-declared metadata

An agent's contract { hard { rule ...; } } and failure_mode blocks are parsed and listed under agent_ir.contracts / failure_modes, but not yet cross-checked against evidence. Tracked metadata, not verified claims.

No evaluator execution

fslc ai eval only reads precomputed JSONL. It does not run LLM evaluators, sample providers, or mutate prompts. Only boolean outcomes are supported; continuous scores are out of scope. Evaluator trust is a calibration-evidence question (missing evidence means evaluator_untrusted).

formal_result: "not_run", consistently

Replay, agent analysis, eval, regress, compare, drift, and compat results all carry formal_result: "not_run". None of them may be displayed as proved or verified. Only the ai_component hard-contract expansion carries a kernel formal result.

Time, probability, LLM truth

A "last 7 days" window is a log-selection label, not a wall-clock proof, and drift is the difference between two observation windows. The truth or groundedness of LLM output itself is proved by no layer of this dialect.
