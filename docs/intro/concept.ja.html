形式手法を知らない人のためのFSL

仕様書は“読める”。
でも“正しい”かは、
確かめられない。

AIが、仕様もコードもテストも高速に書く時代。
では、それが「本当に正しい」保証は、誰がするのでしょう。

FSL は、仕様を 機械が総当たりで確かめる ための言語です。
これまで書いていたプレーンテキストの仕様書の代わりに書きます。人が読む説明と機械が検証する対象が同じ1つのテキストになるので、両者がズレることはありません。
このページを最後まで読むと、その意味が体でわかります。（10分・専門用語なし）

権限のすり抜け

権限のないユーザーが、特定の順で操作すると状態を変えられた。

分解すると、たった3つ

ソフトウェアは「状態・操作・ルール」でできている

状態

いまどうなっているか

下書き / 申請済 / 承認済 / 出荷済 …。アプリが取りうる「状況」。

操作

何ができるか

申請する / 承認する / 出荷する / 取り消す。状態を動かす「行動」。

ルール

何を守るべきか

「承認していない注文は、出荷されない」。絶対に破れてはいけない約束。

バグの多くは、このルールが、ある操作の順番で破れてしまうこと。
では、その「破れ方」をどうやって見つければいいでしょう？

例で考える

「注文」を図にしてみる

箱が状態、矢印が操作。守りたいルールはひとつ——
「承認済を経ていない注文は、出荷済にならない」。

下書き →(申請)→ 申請済 →(承認)→ 承認済 →(出荷)→ 出荷済。途中で取り消すこともできる。一見、何の問題もなさそうに見えます。

ここが核心

テストは「1本の道」、機械は「全部の道」

🧪 テスト

人が思いついた1つの手順を確かめる。

「申請→承認→出荷」はOK。でも、確かめたのは選んだこの1本だけ。

🔎 形式手法（FSL）

考えうるすべての操作順を機械が探索する。

全部の道を試すと…承認を飛ばして出荷に行ける抜け穴が見つかった。

テストは「選んだ道」しか見ません。
FSL は、ルールがどんな順番でも破れないかを、機械に総当たりで調べさせます。

魔法の瞬間

破れるなら、機械が「その手順」を見せてくれる

ルールを書いて検証ボタンを押すと、FSL はこう答えます——
「破れます。これが、その最短の手順です」。

いま検証しているルール（FSLの抜粋）：

// 出荷は承認のあとだけ、のはず…
action ship() {
  requires status == Submitted   // ← バグ：本当は Approved であるべき
  status = Shipped
}
invariant ShipOnlyAfterApproval { status == Shipped => approved }
🔎 fslc verify の実行

※ このアニメは、実際の fslc verify が返した反例の手順データ（JSON）をそのまま再生しています。脚色なし。

これは “バグの再現手順を、実装する前に” 受け取れる、ということ。
疲れ知らずの品質確認担当が、ありとあらゆる順番を試して、破れる道を1本突きつけてくれます。

体験してみる

あなたは、ルールを破れますか？

ボタンで操作してみてください。押せるボタンだけが有効になります。
「承認なしで出荷済」に到達できたら、あなたの勝ち。

正しい版では、手でいくら試しても破れません。
バグ版に切り替えると破れます——でも、人が全パターンを試すのは大変。
機械なら、この盤面の全手順を一瞬で試し尽くします。

なぜ今、これが効くのか

AIが書き、機械が確かめ、AIが直す

昔の形式手法は「専門家が何週間もかけて書くもの」でした。FSL は順番を変えます。

01

人が意図を渡す

「未承認は出荷しない」など、守りたいことを言葉で伝える。

02

AIが仕様を下書き

FSL の文法はAIが知っている。人は覚えなくていい。

03

機械が反例を返す

fslc が矛盾・抜け穴・反例を JSON で返す。

04

AIが読んで直す

反例を読み、仕様を修正。これを破れなくなるまで繰り返す。

人がやるのは意図を決めることと業務上の最終判断だけ。
だから今や、確認モーダルのような小さな状態管理にも気軽に使えます。

もうひとつの特徴

業務から実装まで、1本の線でつながる

同じ「注文」を、粒度を変えて縦に貫けます。上の意図が下までちゃんと反映されているかを、機械が追跡します。

業務（ビジネスルール）

返品は出荷後30日以内、など。業務としての矛盾がないかを確かめる。

↓ シナリオ生成・テスト生成・ログ再生

実装・品質確認

仕様からテスト雛形やシナリオを生成。実装のログが仕様に従うかも確認。

これは実装前に一度だけ済ませる証明ではありません。この鎖はCIの上で生き続けます。どの層への変更も、その都度すべて再検証され、refine は上位層の変更を下位層への機械的なチェックに変換し、ズレは次のコミットで見つかります——本番で事故として発覚するのではなく。同じ仕様の鎖は、新しく入るエンジニアにとっても、コードを引き継ぐAIエージェントにとっても、そのままオンボーディング資料になります。

正直なところ

FSL は万能ではありません

確かめるのは「書いたルールの範囲」だけ。 ルール自体が間違っていれば、検証結果も間違ったルールに基づきます。

テストの代わりにはなりません。 実装・UI・外部連携の確認はテストの役割。両者は補い合う関係です。

ただし、仕様書の代わりにはなります。 手で書いていたプレーンテキストの設計書・仕様書、それを機械も検証できる形にしたのが FSL です。fslc html や explain --readable は、同じソースを人が読むための説明としてそのまま描画するので、別に保守すべき文書は増えません。

最後は人が判断する。 「AIが書いた仕様が、本当に元の業務ルールを表しているか」は、人が確認します。FSL はその判断を助ける外部の基準です。

種明かし

いま体験したのが「形式手法」です

状態と操作とルールを書き、機械があらゆる順番を調べ、破れるなら反例を返す——
難しそうな名前のわりに、やっていることはシンプルでした。
