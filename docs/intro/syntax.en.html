1. Minimal Spec

Start with one ordinary spec

A design-layer spec declares constants, types, state, initial state, actions, and properties.

spec Counter {
  const LIMIT = 3
  type Count = 0..LIMIT

  state { n: Count }
  init { n = 0 }

  action inc() {
    requires n < LIMIT
    n = n + 1
    ensures n == old(n) + 1
  }

  invariant NeverTooHigh { n <= LIMIT }
  reachable CanHitLimit { n == LIMIT }
  terminal { n == LIMIT }
}
PartRoleNote
constInteger constant.Useful for limits and capacities.
typeBounded integer domain.Out-of-range values are checked automatically.
stateLogical state being verified.Only supported state types are allowed.
initInitial state.Assign each state variable exactly once.
actionOne atomic transition.Right-hand sides read the old state.
PropertiesRules to prove or witnesses to find.Each named property appears in diagnostics.

2. Top-Level Forms

FSL files can contain several top-level forms

The core spec is not the only entry point. FSL also has role-specific layers and connection files.

spec

spec Name {
  type ...
  state { ... }
  init { ... }
  action ...
  invariant ...
}

Design-layer state machine and properties.

business + verify

business Name {
  actor Customer, Manager
  entity Return
  process Return { ... }
  policy POL-1 "..." every Return in Requested must eventually be Approved
}
verify { instances Return = 3 }

Business processes, controls, goals, and KPIs.

requirements + verify

requirements Name {
  number Amount
  process Claim with amount: Amount { ... }
  acceptance AC-1 "..." { ... }
  forbidden FB-1 "..." { ... }
}
verify { instances Claim = 3 values Amount = 0..3 }

Requirement IDs, inputs, acceptance flows, forbidden flows, and NFRs.

refinement

refinement ImplRefinesAbs {
  impl Impl
  abs Abs
  maps auto
  map status = impl_status
  action do_it(x: Id) -> do_it(x)
}

Mapping file used by fslc refine.

compose

compose System {
  use Cart as cart from "cart.fsl"
  use Payment as pay from "payment.fsl"
  action checkout(u: cart.UserId, p: pay.PayId) =
    cart.checkout(u) || pay.capture(p) { }
}

Synchronize actions from multiple specs.

governance

governance Controls {
  authority Finance owns CTRL-1
  control CTRL-1 "..." owner Finance applies_to Claim
  delegates Biz from "biz.fsl" {
    require CTRL-1
    CTRL-1 is satisfied_by policy POL-1
  }
  preservation Reform {
    before AsIs from "asis.fsl"
    after  ToBe from "tobe.fsl"
    preserve CTRL-1
    checked_by refinement "tobe_refines_asis.fsl"
  }
}

Cross-business control catalog, checked by fslc check.

3. Types

Types are restricted so verification stays finite

FSL accepts enough structure for real workflows, but avoids arbitrary nested data that makes model checking explode.

TypeSyntaxUse
Int / Booln: Int, ok: BoolIntegers and booleans. Be careful with unbounded Int.
Bounded domaintype Qty = 0..5Automatic boundary checking.
Enumenum St { Open, Closed }Status names and finite categories.
Structstruct Order { st: St, buyer: Option<UserId> }Group scalar fields.
OptionOption<ItemId>None or one scalar value. Prefer over sentinel values.
MapMap<ItemId, Qty>Bounded key to scalar, Option, or struct value.
SetSet<UserId>A set of bounded scalar values.
SeqSeq<JobId, CAP>Capacity-bounded queue or log.
Relationrelation User -> RoleBounded binary link graph.
Dialect kindsentity Claim, number AmountBusiness/requirements kinds bounded by verify.

Allowed state-variable shapes

Use scalar, Option<scalar>, struct, map, set, sequence, or relation. Avoid nested maps and containers inside containers.

state {
  status: St,
  cart: Option<ItemId>,
  orders: Map<OrderId, Order>,
  locked: Set<UserId>,
  queue: Seq<JobId, CAP>,
  delegates: relation User -> User
}

4. State and Init

state declares logical state, init sets the start

init must deterministically assign each state variable exactly once. Maps and sets can be initialized with literals or forall.

Bulk map initialization

type ItemId = 0..2
type Qty = 0..5
state { stock: Map<ItemId, Qty> }
init {
  forall i: ItemId {
    stock[i] = 1
  }
}

Option, Set, Seq

state {
  selected: Option<ItemId>,
  shipped: Set<OrderId>,
  queue: Seq<JobId, CAP>
}
init {
  selected = none
  shipped = Set {}
  queue = Seq {}
}

5. Actions

action combines guards, local bindings, assignments, and postconditions

One step is one enabled action instance. Parameterized actions are checked across their finite parameter values.

fair action checkout(u: UserId) "REQ-4: only stocked item can be checked out" {
  requires cart[u] is some(i)
  requires stock[i] > 0
  let old_stock = stock[i]

  stock[i] = stock[i] - 1
  cart[u] = none

  ensures stock[i] == old(stock[i]) - 1
  ensures old_stock == old(stock[i])
}
SyntaxMeaningNote
fair actionWeak-fair progress action.Use only for actions that drive liveness.
requires exprEnabled condition.Multiple clauses are conjoined.
let x = exprLocal binding inside an action.Not a state update.
ensures exprPostcondition.Use old(expr) to read pre-state.
"ID: text"Traceability tag.Flows into diagnostic JSON.

Simultaneous assignment: every right-hand side reads the old state. Assigning to the same variable or field twice on the same path is a semantics error.

6. Expressions

Expressions combine arithmetic, logic, quantifiers, collections, and Option

CategorySyntaxExample
Arithmetic+ - * / %, min, max, absabs(balance) <= LIMIT

Option extraction

requires cart[u] is some(i)
requires stock[i] > 0

Use x == some(e) to compare a complete Option value. Use is some(v) when the following expression needs a bound payload.

Index guard for Seq

forall i in 0..CAP-1 {
  i < q.size() => jobs[q.at(i)] == Queued
}

Always guard at(i) with a range condition.

Conditional expression

invariant EffectiveLimit {
  limit == if premium then PREMIUM_LIMIT else STANDARD_LIMIT
}

if c then a else b is allowed anywhere an expression is accepted. Both branches have one type; only the selected branch is evaluated.

7. Properties

Choose the property form that matches the requirement

SyntaxMeaningExample
invariant Name { P }P in every reachable state.status == Shipped => approved
trans Name { P }P over each transition. Use old().stock[0] <= old(stock[0])
reachable Name { P }P can be reached.stock[0] == 0
leadsTo Name { P ~> Q }When P happens, Q eventually happens.pending ~> done
leadsTo ... within KQ within K steps.pending ~> within 3 done
helpful action(args)Per-binding progress action for ranked induction.helpful step(c)
leadsTo ... decreases MRanking for induction proof of progress.decreases remaining
unless Name { P unless Q }Safety: P holds until Q.Safety of a held state.
until Name { P until Q }Combines unless safety with leadsTo liveness.Hold the state, yet eventually complete.
terminal { P }Intended stopping state.status == Done or status == Failed

The full form of leadsTo (responsiveness)

leadsTo EventuallyPaid "REQ-9: approved claims are eventually paid" {
  forall c: Claim {
    claims[c].st == Approved ~> claims[c].st == Paid
  }
  helpful pay(c)
  decreases unpaid_count
}

The outer forall checks responsiveness per target. Add fair to the actions that drive progress, within K to bound the response in steps, helpful action(args) to identify the per-binding progress action, and decreases M to supply a ranking for the induction proof. helpful does not create fairness; the action still needs fair action. The ~> arrow is only valid inside leadsTo.

NFR / SLA syntax

requirements SupportSLA {
  entity Ticket
  enum TicketStage { New, Accepted, Responded }
  state { tickets: Map<Ticket, TicketStage> }
  init {
    forall t: Ticket { tickets[t] = New }
  }

  time {
    urgent respond_due
    age wait_age[t: Ticket] while tickets[t] == Accepted
  }

  requirement NFR-1 "respond within 3 ticks of acceptance" {
    action accept(t: Ticket) {
      requires tickets[t] == New
      tickets[t] = Accepted
    }

    fair action respond_due(t: Ticket) {
      requires tickets[t] == Accepted
      requires wait_age[t] >= 3
      tickets[t] = Responded
    }
    deadline wait_age <= 3
  }
}
verify {
  instances Ticket = 2
}

A single time block sits directly under requirements; each deadline lives inside an individual requirement. The usual shape marks the action that only becomes enabled at the deadline as urgent.

9. Refinement and Compose

Connect layers and components explicitly

Refinement mapping

refinement DesignRefinesReq {
  impl Design
  abs Req
  maps auto

  map status = if impl_done then Done else Pending
  action enqueue(c: Claim) -> stutter
  action pay(c: Claim) -> pay(c)
}

Composition

compose OrderSystem {
  use ShoppingCart as cart from "cart_v1.fsl"
  use Payment as pay from "payment.fsl"

  fair action checkout_and_pay(u: cart.UserId, p: pay.PayId) =
      cart.checkout(u) || pay.capture(p) { }

  internal cart.checkout
  internal pay.capture
}

10. Idioms

Preferred FSL forms

Use Option, not sentinels

cart: Option<ItemId>
cart = none
cart = some(i)

Use types, not boundary invariants

type Qty = 0..5
state { stock: Map<ItemId, Qty> }

Do not hand-write stock[i] >= 0.

Guard partial operations

action take() {
  requires q.size() > 0
  x = q.head()
  q = q.pop()
}

Flatten two-dimensional maps

const SLOTS = 4
type Cell = 0..ROOMS*SLOTS-1
// room = c / SLOTS, slot = c % SLOTS

Record history with ghost state

state { ever_locked: Map<UserId, Bool> }
// in lock action: ever_locked[u] = true

Use terminal for intended stops

terminal { status == Done or status == Failed }

Sweep bounded scope

fslc sweep spec.fsl \
  --instances Case=1..3 \
  --depth 1..8 \
  --property AtMostOne

sweep.results records each run; sweep.minimal_counterexample points at the first failing scope.

11. Practice

Practice reading real specs

Map each plain-language requirement to the FSL syntax that expresses it. The reading is given under each card.

  1. "Once submitted, it is eventually approved or rejected"

    Write a leadsTo, or the business-layer every ... must eventually be .... Check whether the action that drives progress needs fair.

  2. "It must not be paid before approval"

    Use an invariant such as paid => approved, or a forbidden flow in the requirements layer that rejects an unapproved payment.

  3. "Stock never goes negative"

    Use type Qty = 0..N. Instead of a hand-written safety rule, guard checkout with requires stock[i] > 0.

  4. "The detailed design has a reserve action, but it is hidden from the requirements layer"

    In the refinement mapping, write action reserve(...) -> stutter. The check confirms the abstract state does not change.

  5. "Never pop from an empty queue"

    Guard head/pop with requires q.size() > 0, or if q.size() > 0 { ... }.

Read an actual spec

After reading the syntax once, work through specs/cart_v1.fsl, then specs/mutex_queue.fsl, then examples/e2e/ to connect the basic shape to the three-layer structure.
