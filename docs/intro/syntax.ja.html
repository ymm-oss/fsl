文法を通して学ぶ

FSL の文法を、
読む順番で学ぶ。

最小の spec から、型、式、文、検証する性質、業務層・要件層、詳細化、合成、非機能要件まで。

このページは「AIが書いたFSLを読める」「反例を見て直す場所がわかる」ことをゴールにしています。

これは30分の読み方入門です。網羅的な正典は言語リファレンス（docs/LANGUAGE.mdからの生成物）にあります——特定の構文を引きたいときはそちらへ。

目次

文法の全体像

1. 最小 specまず読める形をつかむ 2. トップレベル設計・要件・業務・合成・詳細化 3. 型値の種類、対応表、集合、列、構造体 4. 状態と初期化state と init 5. 操作action と実行条件 6. 式演算子・量化・集約・値の有無 7. 検証する性質安全性・到達性・応答性 8. 要件層工程・受け入れ手順・禁止手順・非機能要件 9. 業務層担当者・方針・ゴール・KPI 10. 詳細化・合成層と部品を接続する構文 11. 定番パターンよく使う書き方と禁止形 12. 練習構文を自分で読む

読み方の前提: spec、requirements、business などはFSLの予約語です。コードの中では英語のまま使いますが、本文では「設計層」「要件層」「業務層」のように日本語の役割名を併記します。

用語ミニ辞典: 型＝値の種類と範囲。式＝条件や計算。性質＝守るべきルール。要件層＝企画・受け入れ条件を書く層。業務層＝業務ルールを書く層。詳細化＝粗い仕様と細かい仕様の対応確認です。

1. 最小の仕様

まず、1ファイルの基本形を読む

設計層の基本は spec Name { ... } です。型を宣言し、状態を置き、初期化し、操作を書き、性質を書きます。

spec Counter {
  const LIMIT = 3
  type Count = 0..LIMIT

  state { n: Count }
  init { n = 0 }

  action inc() {
    requires n < LIMIT
    n = n + 1
    ensures n == old(n) + 1
  }

  invariant NeverTooHigh { n <= LIMIT }
  reachable CanHitLimit { n == LIMIT }
  terminal { n == LIMIT }
}
部分役割注意
const整数定数。範囲や列の容量に使えます。
type範囲つき整数型。範囲外は自動で検出されます。
state検証対象の論理状態。許可される型に制限があります。
init初期状態。各 state 変数を決定的に1回初期化します。
action1ステップの遷移。本文の右辺はすべて旧状態を読みます。
検証する性質守る/到達する/応答する性質。結果JSONの診断単位になります。

2. トップレベル

FSL ファイルの先頭に置ける形式

FSL のトップレベルは、設計の spec だけではありません。業務、要件、合成、詳細化の対応表も同じ言語セットに含まれます。

spec

spec Name {
  const ...
  type ...
  state { ... }
  init { ... }
  action ...
  invariant ...
}

設計層の核。状態機械、データ構造、検証する性質を直接書きます。

要件層 requirements + 検証範囲 verify

requirements Name {
  number Amount
  process Claim with amount: Amount { ... }
  acceptance AC-1 "..." { ... }
}
verify { instances Claim = 3 }

要件層。工程、要件ID、受け入れ手順、禁止手順、非機能要件を扱います。

業務層 business + 検証範囲 verify

business Name {
  actor Customer, Manager
  entity Return
  process Return { ... }
  policy POL-1 "..." every Return in Requested must eventually be Approved
}
verify { instances Return = 3 }

業務層。業務ステージ、統制、KPI、ゴールを書きます。

詳細化 refinement

refinement ImplRefinesAbs {
  impl Impl
  abs Abs
  maps auto
  map status = impl_status
  action do_it(x: Id) -> do_it(x)
}

詳細仕様が抽象仕様を逸脱しないかを見る対応表ファイルです。

合成 compose

compose System {
  use Cart as cart from "cart.fsl"
  use Payment as pay from "payment.fsl"
  action checkout(u: cart.UserId, p: pay.PayId) =
    cart.checkout(u) || pay.capture(p) { }
}

複数の設計仕様 spec を名前空間つきで合成し、同時に実行する操作 action を書きます。

統制カタログ governance

governance Controls {
  authority Finance owns CTRL-1
  control CTRL-1 "..." owner Finance applies_to Claim
  delegates Biz from "biz.fsl" {
    require CTRL-1
    CTRL-1 is satisfied_by policy POL-1
  }
  preservation Reform {
    before AsIs from "asis.fsl"
    after  ToBe from "tobe.fsl"
    preserve CTRL-1
    checked_by refinement "tobe_refines_asis.fsl"
  }
}

業務をまたぐ統制カタログ。fslc check で検証します。

verify { ... } の配置: 業務層・要件層の有限範囲は、同じファイルのトップレベルに置く別ブロックです。spec の中には置きません。

範囲を掃引する: fslc sweep spec.fsl --instances Case=1..3 --depth 1..8 --property AtMostOne は、各検証結果を sweep.results に、最初に失敗した範囲を sweep.minimal_counterexample に出します。

3. 型

型は「検証できる形」に制限されている

FSL の型は、有限探索しやすく、AIが誤用しにくい範囲に絞られています。state 変数に置ける型には制限があります。

型構文使いどころ
整数・真偽値n: Int, ok: Bool整数と真偽値。Int は非有界なので状態には慎重に使います。
範囲つき整数型type Qty = 0..5境界チェックが自動で入ります。
対称な範囲つき整数型symmetric type TaskId = 0..2特別扱いしない主体ID。応答性探索で同じ形の状態をまとめるために使います。
列挙型enum St { Open, Closed }状態名や分類。
対称な列挙型symmetric enum Worker { A, B }特別扱いしない有限主体。
構造体struct Order { st: St, buyer: Option<UserId> }複数フィールドをまとめる。フィールドは単純値または値がない/ある型。
値がない/ある型Option<ItemId>値がない/ある。-1 のような特別値の代わり。
対応表Map<ItemId, Qty>範囲つきキーから値へ。値は単純値、値がない/ある型、構造体。
集合Set<UserId>範囲つき値の集合。
列Seq<JobId, CAP>容量つき列。FIFOやログ。
関係relation User -> Role有限範囲の2項リンク。
業務対象・数値種別entity Claim, number Amount業務層・要件層の型種別。範囲は verify ブロックで指定します。

状態 state 変数に置ける型

許可されるのは、単純値、値がない/ある型、構造体、対応表、集合、列、関係です。対応表・集合・関係の端点には、基本的に有限範囲の値を使います。

// OK
state {
  status: St,
  cart: Option<ItemId>,
  orders: Map<OrderId, Order>,
  locked: Set<UserId>,
  queue: Seq<JobId, CAP>,
  delegates: relation User -> User
}

// NG の代表: 対応表の入れ子 / 構造体そのものの Option / 構造体内の集合 など

4. 状態と初期化

state は論理状態、init は探索の出発点

init は毎回同じ初期状態を作る必要があります。対応表や集合は、直接書く値（リテラル）や forall で初期化できます。

対応表 Map をまとめて初期化

type ItemId = 0..2
type Qty = 0..5
state { stock: Map<ItemId, Qty> }
init {
  forall i: ItemId {
    stock[i] = 1
  }
}

値がない/ある型・集合・列の初期化

state {
  selected: Option<ItemId>,
  shipped: Set<OrderId>,
  queue: Seq<JobId, CAP>
}
init {
  selected = none
  shipped = Set {}
  queue = Seq {}
}

初期化の原則: state の各変数は init でちょうど1回値を決めます。未初期化や矛盾した初期化は仕様エラーになります。

5. 操作

action は実行条件、一時変数、代入、事後条件でできている

1ステップは1つの操作候補です。パラメータつき action は、有限範囲の全パラメータ値について検査されます。

fair action checkout(u: UserId) "REQ-4: only stocked item can be checked out" {
  requires cart[u] is some(i)
  requires stock[i] > 0
  let old_stock = stock[i]

  stock[i] = stock[i] - 1
  cart[u] = none

  ensures stock[i] == old(stock[i]) - 1
  ensures old_stock == old(stock[i])
}
構文意味注意
fair action応答性で弱公平性を仮定する操作。進捗を担う action にだけ付けます。
param: Type型つきパラメータ。合成した仕様では alias.Type のように部品名つきの型名も書けます。
param in lo..hi範囲パラメータ。探索範囲を直接指定します。
requires expr実行できる条件。複数書くと「すべて成り立つ必要がある」という意味。
let x = expraction 内の一時的な名前づけ。状態更新ではありません。
ensures expr事後条件。old(expr) で旧状態を参照できます。
"ID: text"要件タグ。診断JSONへ要件情報として流れます。

代入先

x = expr
m[k] = expr
m[k].field = expr
o.field = expr
s = s.add(x)
q = q.pop().push(y)
r = r.add(a, b)

同時代入の意味

action 本文の右辺はすべて旧状態を読みます。書かない変数は自動で保持されます。同じパスで同じ変数・同じフィールドを2回書くと意味エラーです。

// NG: 同じ path で二重代入
action bad() {
  x = 1
  x = 2
}

6. 式

式の構文をまとめて押さえる

FSL の式は、算術、比較、論理、量化、集約、集合や列の操作、値がない/ある型の取り出しを組み合わせます。

分類構文例
算術+ - * / %, 単項マイナス -, min, max, absabs(balance) <= LIMIT
比較== != < <= > >=stock[i] > 0
論理and or not =>paid => approved
量化forall, existsforall i: ItemId { stock[i] >= 0 }
集約count, sumsum(i: Idx of log.at(i) where i < log.size())
一意性unique, exactlyOneexactlyOne(u: UserId where owner[u])
値がない/ある型none, some(e), 構造的 ==/!=, x is some(v)cart[u] == some(i) は比較、is some(i) は束縛
集合Set {}, .add, .remove, .contains, .size()active.contains(u)
列Seq {}, .push, .pop, .head, .at, .contains, .sizeq.size() > 0 => q.head() == j
関係relation A -> B, .add, .remove, .contains, reachable, acyclicacyclic(delegates)
構造体S { f: e }, x.fieldorders[o].st == Paid
特殊old(expr), stage(x)total == old(total) + 1

束縛の3形式

forall x: T { expr }
forall i in 0..CAP-1 { expr }
forall x in active where ready[x] { expr }

型で回す、範囲で回す、集合や列の中身で回す、の3種類です。exists、unique、exactlyOne でも同じ考え方です。

値がある場合だけ中身を取り出す

// OK
requires cart[u] is some(i)
requires stock[i] > 0

// NG
// requires cart[u] == some(i)

Option値全体の比較には x == some(e)、後続式で中身へ名前をつける場合は x is some(v) を使います。

詳細化の対応表だけで使える if式

map holder[s: SeatId] =
  if slots[s].st == Sold then slots[s].buyer else none

if c then a else b は詳細化の map や action 引数の中だけで使えます。通常の spec の式では使いません。

7. 検証する性質

どの性質を、どの構文で書くか

FSL で検証する性質は、検証結果の単位です。1つの要求を1つの宣言に分けると、反例が読みやすくなります。

構文意味例
invariant Name { P }全到達状態で P。status == Shipped => approved
trans Name { P }各操作の前後で P。旧状態は old()。stock[0] <= old(stock[0])
reachable Name { P }P に到達できること。stock[0] == 0
leadsTo Name { P ~> Q }P になったらいつか Q。pending ~> done
leadsTo ... within KK手以内の応答。pending ~> within 3 done
helpful action(args)対象ごとの進捗操作を指定。helpful step(c)
leadsTo ... decreases M帰納法で使う減少量。decreases remaining
unless Name { P unless Q }Q まで P を保つ安全性。保持状態の安全性。
until Name { P until Q }安全性 unless と応答性 leadsTo の組み合わせ。保ちながらいつか完了。
terminal { P }意図した停止状態。status == Done or status == Failed

応答性 leadsTo の完全形

leadsTo EventuallyPaid "REQ-9: approved claims are eventually paid" {
  forall c: Claim {
    claims[c].st == Approved ~> claims[c].st == Paid
  }
  helpful pay(c)
  decreases unpaid_count
}

外側の forall は、対象ごとに応答性を分けて検査します。helpful action(args) はその対象の進捗を担う操作を指定し、decreases の帰納証明で使います。ただし helpful は公平性を作りません。実際の進捗操作には下位層の fair action が必要です。~> は leadsTo の中だけで使えます。

8. 要件層

要件層 requirements は、要件IDと受け入れ条件を持つ

企画担当が扱う要件は、工程、受け入れ手順、禁止手順で書くのが基本です。複雑な状態が必要なときだけ、設計層と同じ細かい書き方を使います。

requirements ExpenseRequirements {
  implements ExpenseToBe from "1_business.fsl" { }

  number Amount
  const AUTO_LIMIT = 1

  process Claim with amount: Amount {
    stages Draft, Submitted, Approved, Rejected, Paid
    initial Draft
    transition submit Draft -> Submitted by Employee
      with a: Amount when a > 0 set amount = a
      covers REQ-1 "申請者が金額を入力して経費を申請する"
    transition auto_approve Submitted -> Approved by System
      when amount <= AUTO_LIMIT
      covers REQ-2 "少額申請は自動承認される"
    transition pay Approved -> Paid by Finance
      covers REQ-4 "承認済み申請だけが支払われる"
  }

  acceptance AC-1 "少額申請は支払いまで進める" {
    submit(0, 1) auto_approve(0) pay(0)
    expect Claim 0 in Paid
  }

  forbidden FB-1 "下書き申請は支払えない" {
    pay(0)
    expect rejected
  }
}
verify {
  instances Claim = 3
  values Amount = 0..3
}
構文意味
implements Abs from "file" { ... }上位の業務層などを実装する。空なら名前一致の対応表を自動生成。
process E with f: T業務対象のステージ機械と持ち回るデータを宣言。
transition name A -> B by Actorステージ遷移。担当者、入力、条件、更新、要件IDを持てます。
acceptance ID "..." { steps expect ... }許可されるべき具体手順。検査時に実行器で再生されます。
forbidden ID "..." { steps expect rejected }最後の手順が拒否されるべき禁止手順。
branches { when ... { ... } maps ... }複雑な操作の分岐を要求ごとに分割。

NFR / SLA の構文

requirements SupportSLA {
  entity Ticket
  enum TicketStage { New, Accepted, Responded }
  state { tickets: Map<Ticket, TicketStage> }
  init {
    forall t: Ticket { tickets[t] = New }
  }

  time {
    urgent respond_due
    age wait_age[t: Ticket] while tickets[t] == Accepted
  }

  requirement NFR-1 "受理から3 tick以内に応答する" {
    action accept(t: Ticket) {
      requires tickets[t] == New
      tickets[t] = Accepted
    }

    fair action respond_due(t: Ticket) {
      requires tickets[t] == Accepted
      requires wait_age[t] >= 3
      tickets[t] = Responded
    }
    deadline wait_age <= 3
  }
}
verify {
  instances Ticket = 2
}

time は要件層 requirements の直下に1つ。deadline は個別要件 requirement の中に置きます。期限到達時にだけ有効になる操作 action を urgent にするのが基本形です。

9. 業務層

業務層 business は、業務プロセスと統制を読む人向けに書く

業務層では、実装状態名ではなく、担当者、業務対象、工程、方針、ゴール、KPI を使います。

business ReturnHandling {
  actor Customer, Manager
  entity Return

  process Return {
    stages Requested, Approved, Rejected, Refunded
    initial Requested
    transition approve Requested -> Approved by Manager
    transition reject Requested -> Rejected by Manager
    transition refund Approved -> Refunded by Manager
  }

  kpi refunded = count Return in Refunded

  policy PAY-2 "すべての申請はいずれ判断される"
    every Return in Requested must eventually be Approved or Rejected or Refunded

  goal AllSettled "すべての申請は完了状態にできる"
    all Return can be Refunded or Rejected
}
verify {
  instances Return = 3
}
構文意味
actor A, B遷移の実行者一覧。
entity E業務対象。個数は verify で指定します。
process E { stages ... }業務ステージの状態機械。
policy ID "..." invariant { P }業務上の安全性。
policy ... responds { P ~> Q }明示的な応答性。
every E in S must eventually be A or B「Sのものは、いずれAまたはBになる」という読みやすい短縮形。
goal some E can reach S到達可能性の業務ゴール。
goal all E can be A or B全ケースを完了状態にできること。

10. 詳細化と合成

層をつなぐ構文、部品を合成する構文

大きな仕様は、上位意図と下位設計を詳細化チェックでつなぎ、複数コンポーネントを合成で同期させます。

詳細化の対応表

refinement DesignRefinesReq {
  impl Design
  abs Req
  maps auto

  map status = if impl_done then Done else Pending
  map owner[c: Claim] = impl_owner[c]

  action enqueue(c: Claim) -> stutter
  action pay(c: Claim) -> pay(c)

  preserve progress {
    respond EveryRequestHandled by pay, reject
  }
}

合成

compose OrderSystem {
  use ShoppingCart as cart from "cart_v1.fsl"
  use Payment as pay from "payment.fsl"

  state { linked: Int }
  init { linked = 0 }

  fair action checkout_and_pay(u: cart.UserId, p: pay.PayId) =
      cart.checkout(u) || pay.capture(p) {
    requires pay.payments[p].st == Authorized
    linked = linked + 1
  }

  internal cart.checkout
  internal pay.capture
}
構文意味
maps auto同名・互換の状態/操作を自動対応。
map x = expr抽象状態変数 x を実装状態から計算。
map x[k: T] = expr対応表 Map の要素ごとの写像。
action impl(...) -> abs(...)詳細操作を抽象操作に対応。
action impl(...) -> stutter抽象状態を変えない内部ステップ。
preserve progress上位の応答性が下位で失われていないか追加チェック。
use Spec as alias from "file"合成 compose の部品取り込み。
a.act(...) || b.act(...)複数部品の操作を同時に実行。
internal alias.actionその操作の単独実行を禁止。

11. 定番パターン

FSL らしい書き方と、避ける書き方

文法を覚えるだけでは、良い仕様にはなりません。よくある表現は、次の形に寄せます。

特別値ではなく Option

// OK
cart: Option<ItemId>
cart = none
cart = some(i)

-1 などの特別値を使わない。

境界ルール invariant ではなく型

type Qty = 0..5
state { stock: Map<ItemId, Qty> }

stock[i] >= 0 は手書きしない。

未定義になり得る操作は条件で守る

action take() {
  requires q.size() > 0
  x = q.head()
  q = q.pop()
}

空 pop/head と範囲外 at は自動検出されます。

列 Seq は添字条件で守る

invariant GoodQueue {
  forall i in 0..CAP-1 {
    i < q.size() => jobs[q.at(i)] == Queued
  }
}

at(i) を読むなら、実際に要素が入っている範囲かを条件で守ります。

2次元の対応表は1つのキーに畳む

const SLOTS = 4
type Cell = 0..ROOMS*SLOTS-1
// room = c / SLOTS, slot = c % SLOTS

対応表の入れ子は使わず、2つの軸を1つの範囲つき型にまとめます。

過去の事実は補助変数に記録する

state { ever_locked: Map<UserId, Bool> }
// lock action で ever_locked[u] = true

「過去にXした」は、検証用の補助状態に記録します。

12. 練習

構文を読めるようになる練習

次の問いを、FSL 構文に対応づけて読んでください。答えは各カードの下にあります。

  1. 「申請済になったら、いつか承認または却下される」

    leadsTo または業務層の every ... must eventually be ... で書きます。進捗を担う action に fair が必要かを確認します。

  2. 「承認前に支払われてはいけない」

    invariant で paid => approved、または要件層 requirements の forbidden で未承認支払い手順を拒否させます。

  3. 「在庫はマイナスにならない」

    type Qty = 0..N を使います。手書きの安全ルールではなく、checkout などに requires stock[i] > 0 を付けます。

  4. 「詳細設計では予約操作があるが、要件層には見せない」

    詳細化の対応表で action reserve(...) -> stutter と書きます。抽象状態が変わらないかが検査されます。

  5. 「空キューから取り出さない」

    requires q.size() > 0 または if q.size() > 0 { ... } で head/pop を条件で守ります。
