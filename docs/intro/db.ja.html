DB / multi-environment compatibility

dbsystem でDB変更と複数環境を仕様として確認する

fsl-db は、DB単体ではなく、スキーマ変更、マイグレーション、アプリ成果物の読み書き能力、複数環境のロールアウト範囲を1つの仕様にまとめ、互換性違反を反例つきで返す dialect です。

設計の詳細は DESIGN-db.md にあります。

何を防ぐか

DBとアプリのずれをリリース前に見つける

本番では、古いアプリ、新しいアプリ、worker、batch、外部クライアント、途中のスキーマが短時間だけ同居します。fsl-db はその同居期間を有限のスナップショットとして扱い、読み書き先のカラムが存在するか、NOT NULL 化の順序が安全かを検査します。

読み取り互換性

active な成果物が読むカラムが、その環境・スキーマ範囲で存在することを確認します。

書き込み互換性

成果物が書くカラムが先に削除されていないか、移行中の二重書き込みが成り立つかを確認します。

NOT NULL の順序

追加、backfill、set_not_null の順序を確認し、未 backfill のカラムを必須化する失敗を検出します。

環境ごとの同居期間

environment の schema lo..hi と when schema で、prod/staging などの実際の同居範囲を明示します。

複数環境

DB schema version だけではなく、環境にいる artifact を書く

提案の中心は、バージョン番号ではなく capability です。「server_v3 だから安全」ではなく、「この環境にいる artifact が何を読み、何を書き、どの schema 範囲に存在しうるか」を書きます。

artifact server_v1 {
  reads users.id, users.legacy_name;
  writes users.legacy_name;
}

artifact server_v2 {
  reads users.id, users.legacy_name, users.display_name;
  writes users.legacy_name, users.display_name;
}

artifact worker_v1 {
  reads users.legacy_name;
}

environment prod {
  schema 0..4;
  active server_v1 when schema 0..0;
  supported server_v1 when schema 0..1;
  active server_v2 when schema 1..3;
  may_exist worker_v1 when schema 0..4;
}
宣言意味主な使いどころ
activeその環境に確実に存在し、互換性を満たすべき artifact。現在稼働中の server、worker、batch。
supported互換性を保証する対象として残す artifact。rollback 対象、サポート対象の旧 server。
may_exist現実に残っている可能性がある artifact。古い worker、外部クライアント、mobile/offline 系の近似。
when schema lo..hiその artifact が同居しうる schema window。割合 rollout、kill switch、段階的切替の有限近似。

複数環境は environment を複数書くことで表します。各環境の schema lo..hi は、宣言順の migration plan で到達可能な有限 snapshot 群です。任意の schema と artifact の直積ではなく、artifact ごとの when schema window が同居関係を決めます。

モデル

dbsystem は既存カーネルへ展開される

dbsystem は新しい verifier kernel ではありません。DB向けの宣言を typed IR に読み込み、既存の状態機械へ展開します。

dbsystem の要素意味展開後の考え方
database初期スキーマ、テーブル、カラムのライフサイクル。column_exists、column_backfilled、column_not_null などの状態。
migrationスキーマバージョンを1つ進める操作。schema_version == from を guard に持つ1つの action。
artifactアプリ/worker/API などの読み書き能力。互換性 rule から生成される不変条件と metadata。
environmentどの schema と artifact が同居するか。有限の到達可能スナップショット。

基本形

仕様には schema、migration、artifact、environment をまとめて書く

dbsystem UserDb {
  database app {
    schema 0
    table users {
      column id: Int present backfilled not_null;
      column display_name: Text absent;
    }
  }

  migration add_display_name from 0 to 1 {
    add users.display_name nullable;
  }
  migration backfill_display_name from 1 to 2 {
    backfill users.display_name;
  }
  migration require_display_name from 2 to 3 {
    set_not_null users.display_name;
  }

  artifact server_v2 {
    reads users.id, users.display_name;
    writes users.display_name;
    accepts api.CreateUserV1, api.CreateUserV2;
    responds response.email;
  }

  artifact ios_v1 {
    calls api.CreateUserV1;
    emits_offline api.CreateUserV1 ttl 2;
    expects response.email;
  }

  environment prod {
    schema 0..3;
    flag email_v2 { off, on } default off;
    active server_v2 when schema 1..3 when flag email_v2=on;
    supported server_v2 when schema 1..3;
    may_exist ios_v1 when schema 0..3;
  }

  environment staging {
    schema 0..3;
    active server_v2 when schema 1..3;
  }

  check compatibility {
    rule all_active_reads_exist;
    rule all_active_writes_exist;
    rule removed_only_after_unused;
    rule not_null_after_backfill;
    rule api_calls_accepted;
    rule api_responses_expected;
    rule offline_payloads_accepted;
  }
}

結果読み方
verified_under_assumptions有限のロールアウト範囲と完全な capability 宣言を前提に、互換性 rule が通った状態。
column_removed_while_still_read同居期間中の成果物が、すでに削除されたカラムを読んでいる。
column_removed_while_still_written同居期間中の成果物が、すでに削除されたカラムへ書こうとしている。
not_null_before_backfillbackfill 前に NOT NULL 化する順序違反。
api_response_field_missing旧クライアントが期待する応答フィールドを、同居中の提供側が返していない。
offline_payload_not_acceptedTTL window 内に残る offline payload を、同居中の提供側が受け付けない。
observed_mismatchruntime log が仕様宣言と食い違った。形式証明ではなく観測証拠。

境界

実装済みの有限モデルと、残る境界を分ける

データ保存性・rollback

data_preserved と rollback_equivalent は有限の抽象行モデルで確認します。全本番データへの証明ではありません。

API / offline / feature flag

API operation、response field、offline payload TTL、feature flag は有限の capability/tick/variant として記述できます。割合や確率の証明ではなく、有限 snapshot です。

runtime observation

fslc db observe は「まだ古い成果物がいる」「未宣言の読み書きがある」を observed_mismatch として返します。ログ不在は未使用の証明ではありません。

SQL/ORM import

fslc db import は最小 SQL DDL と Prisma schema の scalar field を dbsystem に変換します。未対応構文は明示 warning になります。

外部 evidence

本番データの diff/audit evidence と DB engine dry-run evidence は、formal_result: "not_run" の別JSON schemaで扱います。

時間・確率

割合 rollout、日数 TTL、offline client は確率や実時間としては証明しません。有限の同居 window と tick として近似します。
