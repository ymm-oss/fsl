要件層 requirements は、要件ID、入力、条件、受け入れ手順、禁止手順、非機能要件を書く層です。業務意図を、テストと設計へ渡せる粒度まで具体化します。

読み方の前提: requirements、process、acceptance などはFSLの予約語です。本文では「要件層」「工程」「受け入れ手順」のように日本語名を併記します。

1. 役割

要件層は、業務意図を入力・条件・例に分解する

業務層が「何を業務として守るか」を書くのに対し、要件層は「どの入力で、どの条件なら、どう進むか」を書きます。

業務層

申請は判断される、承認後だけ支払う、返金は完了状態にできる。業務担当者が確認する抽象度。

要件層

金額入力、しきい値、自動承認、管理者承認、受け入れ手順、拒否されるべき手順。企画・受け入れ確認の抽象度。

重要: 要件層では、要件IDと原文を落とさないことが最優先です。反例が出たとき、どの要求が破れたのかをJSONから辿れるようにします。

2. 基本形

process に入力・条件・更新・要件IDを足す

単一の業務対象がステージを進む要件は、process Entity with field: Type で書くのが基本です。

requirements ExpenseRequirements {
  number Amount
  const AUTO_LIMIT = 1

  process Claim with amount: Amount {
    stages Draft, Submitted, Approved, Rejected, Paid
    initial Draft
    transition submit Draft -> Submitted by Employee
      with a: Amount
      when a > 0
      set amount = a
      covers REQ-1 "申請者が金額を入力して経費を申請する"

    transition auto_approve Submitted -> Approved by System
      when amount <= AUTO_LIMIT
      covers REQ-2 "少額申請は自動承認される"

    transition mgr_approve Submitted -> Approved by Manager
      when amount > AUTO_LIMIT
      covers REQ-3 "高額申請は管理者が承認する"

    transition reject Submitted -> Rejected by Manager
      when amount > AUTO_LIMIT
      covers REQ-3 "高額申請は管理者が却下できる"

    transition pay Approved -> Paid by Finance
      covers REQ-4 "承認済み申請だけが支払われる"
  }
}

verify {
  instances Claim = 3
  values Amount = 0..3
}
構文意味書くときの目安
number Amount数値の種類。範囲は verify { values ... } で指定します。
process Claim with amount: Amountステージと持ち回るデータ。申請金額、期限、区分など、要件判断に必要な値。
with a: Amount操作の入力。ユーザーやシステムが操作時に渡す値。
when ...実行できる条件。しきい値、状態、権限、期限など。
set amount = a持ち回るデータの更新。あとで判断に使う値だけ保存します。
covers REQ-1 "..."要件IDと原文。診断JSON、反例、シナリオに伝わります。

3. 受け入れ・禁止

許す手順は acceptance、拒否すべき手順は forbidden

安全ルールだけでは「本来拒否されるべき操作が受け入れられた」ことを見逃す場合があります。要件層では、正例と負例を両方書きます。

受け入れ手順

acceptance AC-1 "少額申請は支払いまで進める" {
  submit(0, 1)
  auto_approve(0)
  pay(0)
  expect Claim 0 in Paid
}

許可されるべき手順です。検査時に実行され、シナリオやテスト生成にも流れます。

禁止手順

forbidden FB-1 "下書き申請は支払えない" {
  pay(0)
  expect rejected
}

最後の操作が拒否されるべき手順です。受け入れられた場合、条件が弱すぎることがわかります。

使い分け: acceptance は「できるべきこと」を守ります。forbidden は「できてはいけないこと」を守ります。重要な要件には、できる例と拒否される例をペアで置くと読みやすくなります。

4. 非機能要件

権限・監査・容量・期限は、要件層で扱う

非機能要件の多くは、通常の条件や安全ルールで書けます。期限のように時間を扱うものだけ、time と deadline を使います。

非機能要件書き方注意
権限操作条件 when または requires。誰が実行できるかを曖昧にしない。
監査監査状態と安全ルール。イベントログが必要なら設計層・実装接続へ送る。
容量範囲つき数値、列 Seq の容量、件数の上限。境界は手書きルールではなく型で表せるなら型に寄せる。
SLA・期限time + deadline。実時間ミリ秒ではなく、離散的な tick として検査する。
確率・百分位・使いやすさFSL外の文書に書く。FSLで直接保証しません。

SLA の基本形

requirements SupportSLA {
  entity Ticket
  const SLA_TICKS = 3
  enum TicketStage { New, Accepted, Responded }

  state { tickets: Map<Ticket, TicketStage> }
  init {
    forall t: Ticket { tickets[t] = New }
  }

  time {
    urgent respond_due
    age wait_age[t: Ticket] while tickets[t] == Accepted
  }

  requirement NFR-1 "受理から3 tick以内に応答する" {
    action accept(t: Ticket) {
      requires tickets[t] == New
      tickets[t] = Accepted
    }

    fair action respond_due(t: Ticket) {
      requires tickets[t] == Accepted
      requires wait_age[t] >= SLA_TICKS
      tickets[t] = Responded
    }

    deadline wait_age <= SLA_TICKS
  }
}

verify {
  instances Ticket = 2
}

SLAの落とし穴: いつでも有効な操作を urgent にすると、時間が進まず、期限が空虚に通ることがあります。期限到達時にだけ有効になる操作を urgent にします。

5. 業務層との接続

implements で、要件が業務意図を落としていないか見る

要件層が業務層を実装する場合、implements Abs from "file" { } を置きます。名前がそろっていれば、FSL が対応表を自動生成できます。

名前一致でつなぐ

requirements ExpenseRequirements {
  implements ExpenseToBe from "business.fsl" { }
  ...
}

業務層と同じ対象名・ステージ名・遷移名を使うと、自動対応しやすくなります。

ずれがある場合

implements ExpenseToBe from "business.fsl" {
  maps auto
  map status = req_status
}

名前や構造が違う場合は、明示的な対応を書きます。

検証時の見え方

$ fslc verify requirements.fsl --depth 8

要件自身の検証に加えて、業務層への詳細化チェック結果がJSONに含まれます。

6. チェックリスト

要件層レビューで見ること

  1. 各遷移に要件IDがあるか

    covers REQ-... がないと、反例から原要求へ戻りにくくなります。

  2. 境界値が明示されているか

    <= AUTO_LIMIT と > AUTO_LIMIT のように、境界を曖昧にしません。

  3. 受け入れ手順と禁止手順が両方あるか

    正例だけだと条件の弱さを見逃します。重要な拒否条件には forbidden を置きます。

  4. 非機能要件をFSLで書けるものと書けないものに分けたか

    権限・監査・容量・SLAは書けます。確率、百分位、実時間ミリ秒、使いやすさは文書に残します。

  5. 業務層を弱めていないか

    implements の失敗は、要件が業務意図を落とした可能性として読みます。
