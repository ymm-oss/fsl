1. Boundary

Write business vocabulary, not implementation vocabulary

The business layer is for lifecycle and controls that domain stakeholders can validate. UI screens, API names, queues, database flags, and retry mechanics belong lower.

Write here

Actors, entities, stages, transitions, policies, goals, and KPIs. Example: every return request is eventually approved, rejected, or refunded.

Do not write here

Database columns, screen routing, API endpoints, async-job internals, retry strategy, real-time milliseconds, probabilities, or prose rationale.

2. Basic Shape

Use actor, entity, and process

Start with the business object and its allowed stage transitions. FSL expands this readable form into a verifiable state machine.

business ReturnHandling {
  actor Customer, Manager
  entity Return

  process Return {
    stages Requested, Approved, Rejected, Refunded
    initial Requested
    transition approve Requested -> Approved by Manager
    transition reject  Requested -> Rejected by Manager
    transition refund  Approved  -> Refunded by Manager
  }
}

verify {
  instances Return = 3
}
ConstructMeaningGuidance
actorThe roster of people or systems that perform business transitions.Names used in by must be declared here.
entityThe business object with a lifecycle.Return, claim, order, contract, ticket.
processThe stage machine for the entity.Write only transitions allowed by the business.
transitionOne business operation.Who moves which stage to which stage.

3. Controls and Goals

Use policy for controls and goal for desired reachability

Keep "must never happen" and "must be possible" separate. Use kpi for count projections, not for rules.

Readable stage syntax

kpi refunded = count Return in Refunded

policy EveryRequestDecided "every request is eventually decided"
  every Return in Requested must eventually be Approved or Rejected or Refunded

goal AllSettled "all cases can be completed"
  all Return can be Refunded or Rejected

Expression form

policy ApprovedEventuallyRefunded "approved returns are eventually refunded" responds {
  forall r: Return {
    stage(r) == Approved ~> stage(r) == Refunded
  }
}

goal SomeRefunded "at least one return can be refunded" {
  exists r: Return { stage(r) == Refunded }
}
PurposeConstructFailure tells you
Safety / prohibitionpolicy ... invariant { ... }A trace that violates a business rule.
No unattended casespolicy ... every ... must eventually ...A case can remain pending or loop forever.
Reachabilitygoal ... can reach ...A supposedly possible business outcome is unreachable.
Projectionkpi name = count Entity in StageMetadata for explanation and scenarios, not a constraint.

Trace controls with control, scale them with governance

A control is named governance metadata (owner, severity, applies_to). It generates no property by itself — attach it to a checkable rule with satisfies so a violation reports which control broke. Controls shared across business specs lift into a standalone governance catalog.

Declare and satisfy a control

control CTRL-DECISION
  "Every return must be adjudicated"
  owner Manager
  severity high
  applies_to Return

policy EveryRequestDecided "every request is eventually decided"
  satisfies CTRL-DECISION
  every Return in Requested must eventually be Approved or Rejected or Refunded

Lift shared controls into a catalog

governance EnterpriseReturnControls {
  authority Operations owns CTRL-DECISION
  control CTRL-DECISION "Every return must be adjudicated"

  delegates ReturnHandling from "return_policy.fsl" {
    require CTRL-DECISION
    CTRL-DECISION is satisfied_by policy EveryRequestDecided
  }

  preservation ReturnReform {
    before AsIsReturn from "asis_return.fsl"
    after  ToBeReturn from "tobe_return.fsl"
    preserve CTRL-DECISION
    checked_by refinement "tobe_refines_asis.fsl"
  }
}
ConstructMeaningNote
control ID "..." owner ... severity ... applies_to ...Governance metadata for a control.Catalog entry only; generates no property.
policy/goal ... satisfies CTRLTies a checkable rule to a control.A violation reports requirement.controls in JSON.
governance { authority ... owns ... }A standalone, cross-business control catalog.Records enterprise ownership above the business specs.
delegates B from "..." { require / is satisfied_by }Maps each control to a business policy/goal.fslc check verifies every required control is satisfied.
preservation { before / after / preserve / checked_by refinement }Keeps controls across an As-Is → To-Be reform.Runs the refinement (depth 8); reports refines.

fslc check governance.fsl validates the referenced controls, business files, and satisfying policies/goals, and runs each preservation refinement — results land under governance.preservations. Unknown control references are errors; declared-but-unused controls are warnings.

4. Verification Bounds

verify decides how many entities to model

Business entity declarations are finite for verification. More instances expose interaction bugs but cost more to search.

Single-case flow

verify {
  instances Return = 1
}

Useful for validating basic lifecycle transitions.

Multi-case behavior

verify {
  instances Return = 3
}

Use this when counts, parallel cases, or unattended cases matter.

Numeric bounds belong lower

number Amount
verify { values Amount = 0..3 }

Amounts and deadline values are usually modeled in the requirements layer.

5. Handoff

Stable names make requirements refinement easier

The requirements layer connects with implements. If entity, stage, and transition names stay aligned, FSL can often synthesize identity mappings.

Business layer

business ExpenseToBe {
  actor Employee, Manager, Finance, System
  entity Claim
  process Claim {
    stages Draft, Submitted, Approved, Rejected, Paid
    initial Draft
    transition submit Draft -> Submitted by Employee
    transition approve Submitted -> Approved by Manager
    transition reject Submitted -> Rejected by Manager
    transition pay Approved -> Paid by Finance
  }
}

Requirements layer

requirements ExpenseRequirements {
  implements ExpenseToBe from "business.fsl" { }

  number Amount
  process Claim with amount: Amount {
    stages Draft, Submitted, Approved, Rejected, Paid
    initial Draft
    transition submit Draft -> Submitted by Employee with a: Amount
      when a > 0 set amount = a
      covers REQ-1 "The applicant submits an expense claim by entering an amount"
  }
}

Do not weaken business intent to make refinement pass. If the lower layer fails to implement the business layer, first inspect whether the requirement dropped or changed the business rule.

6. Checklist

Review before moving to requirements

  1. Is the entity a lifecycle object?

    Use entity for objects such as claims, returns, orders, and contracts, not for plain input values.

  2. Are stage names business-readable?

    Implementation states such as Queued or Retrying should wait for the design layer unless they are truly business vocabulary.

  3. Are unattended states covered?

    Use every ... must eventually ... for cases that must not remain pending.

  4. Are KPIs separate from rules?

    kpi is a projection. Constraints such as "at most N" should be policies or lower-layer requirements.
