読み方の前提: state や action のような英単語は、FSLの「予約語」です。日本語に置き換えて書くことはできないため、この教材では「予約語」と「意味」を必ず並べて説明します。

用語ミニ辞典: 状態空間＝あり得る状態の集まり。反例＝ルールが破れる具体的な手順。帰納法＝無限に深い手順でも破れないことを示す考え方。応答性＝「いつか完了する」性質。詳細化＝粗い仕様を細かい仕様で実現できているかの確認です。

1. モデル化

FSL はアプリを「状態と操作の図」として見る

FSL の中心にあるのは、状態、操作、ルールです。専門用語ではこの図を遷移系と呼びます。コード全体をそのまま検証するのではなく、守りたい振る舞いに関係する状態だけを取り出します。

状態 state論理状態。例: 注文ステータス、在庫、キュー、承認済フラグ。
初期状態 init最初の状態。全ての状態変数を決定的に初期化する。
操作 action1ステップの操作。requires が実行できる条件、本文が次状態。
性質invariant / trans / reachable / leadsTo で守ること。
反例手順破れるなら、最短の操作列と状態差分が返る。

FSL の小さな状態図

spec OrderFlow {
  enum St { Draft, Submitted, Approved, Shipped }
  state { status: St, approved: Bool }
  init { status = Draft approved = false }

  action submit() {
    requires status == Draft
    status = Submitted
  }

  action approve() {
    requires status == Submitted
    status = Approved
    approved = true
  }

  action ship() {
    requires status == Approved
    status = Shipped
  }

  invariant ShipOnlyAfterApproval {
    status == Shipped => approved
  }
}

読み替え

FSL意味
state検証器が追跡する変数集合。
init探索の出発点。
action有効なら1手進められる遷移。
requiresその操作を押せる条件。
invariant全到達状態で破れてはいけない性質。

2. 状態空間

総当たりできるように、世界を有限に切る

FSL はアプリ全体の無限の現実を扱うのではなく、検証したい性質に必要な範囲を有限のモデルにします。有限化の主役は、範囲つき整数型、列挙型、列の容量、検証範囲の指定です。

範囲つき整数型

type Qty = 0..5
type UserId = 0..2

在庫やユーザーIDを有限範囲にする。範囲外は自動で type_bound として検出されます。

列挙型 enum

enum St { Open, Closed }

状態名を有限集合にする。状態機械やステージ表現に向きます。

列 Seq の容量

const CAP = 3
state { q: Seq<JobId, CAP> }

キューは容量つき。満杯への push や空への pop は自動診断されます。

検証範囲 verify

verify {
  instances Claim = 3
  values Amount = 0..10
}

業務層・要件層で使う対象数や数値範囲を指定します。

有限化は、仕様を弱めることではありません。 検証したい相互作用が現れる最小モデルを選びます。複数主体の相互作用なら、2個では対称性で隠れることがあるため、3個以上を試すのが有効な場合があります。

3. 総当たり検査

総当たり検査は「深さ K までの全経路」を調べる

総当たり検査（BMC）は「有界モデル検査」の略です。init から始まり、0手、1手、2手…K手までのすべての操作 action の組み合わせを探索します。反例があれば最短の手順が返ります。

実行

$ fslc verify specs/cart_v1.fsl --depth 8
{
  "result": "verified",
  "completeness": "bounded",
  "checked_to_depth": 8
}

意味

結果読み方
verified深さ K まで反例なし。無限の証明ではない。
violated反例あり。返ってくる手順は最短の再現手順。
reachable_failed到達してほしい状態に K 手以内で届かない。

  1. 状態を1つ置く

    まず init の状態を作ります。全ての invariant はここでも検査対象です。

  2. 有効な操作 action を列挙する

    各 action とパラメータ値の組について、requires が成り立つかを調べます。

  3. 次状態を作る

    action 本文を同時代入として評価します。書いていない変数は自動的に保持されます。

  4. 性質を検査する

    安全ルール、前後関係、境界、未定義操作、停止、応答性などを確認します。

4. 帰納法

k帰納法は「到達した状態」だけを見ない

総当たり検査は init から到達できる範囲を調べます。k帰納法は、より強く「性質を満たすすべての状態から1手進めても性質が保たれるか」を調べます。

土台の確認

深さ K まで、実際に到達できる経路に反例がないことを確認します。総当たり検査と同じ役割です。

一歩進める確認

不変条件を満たす任意の状態列から、次の1手で不変条件が破れないことを確認します。

unknown_cti は「証明に足りない情報」

帰納法が失敗しても、必ずしも到達可能なバグとは限りません。CTI は「反例候補」のことで、「この状態があり得るなら次の1手で破れる」という候補です。

{
  "result": "unknown_cti",
  "cti": {
    "states": [
      { "status": "Approved", "approved": false },
      { "status": "Shipped", "approved": false }
    ]
  }
}

この例では、実際には Approved なら approved == true のはずです。その事実を補助不変条件として足すと証明できます。

invariant ApprovedImpliesFlag {
  status == Approved => approved
}

判断の分岐: CTI の開始状態が本当に到達可能なら、仕様または設計のバグです。到達不能だがドメイン上の真実で排除できるなら、補助不変条件を追加します。

5. 応答性

leadsTo は「いつか応答する」を調べる

invariant は「悪い状態に入らない」を見る安全性です。leadsTo は「ある状態になったら、いずれ別の状態になる」を見る応答性です。

応答性

fair action finish(j: JobId) {
  requires jobs[j] == Running
  jobs[j] = Done
}

leadsTo EventuallyDone {
  forall j: JobId {
    jobs[j] == Running ~> jobs[j] == Done
  }
}

検証器が探す失敗

形意味
輪っか状の反例P が保留されたまま同じ論理状態のループに入る。
停止による未完了P が保留されたまま、全操作ができない状態で止まる。
減少量の証明失敗decreases が非負・単調減少を示せない。

fair

弱公平性。ある操作候補がループ中ずっと有効なら、いつか実行されると仮定します。進捗を担う action にだけ付けます。

within K

P ~> within K Q は K 手以内の応答。総当たり検査の範囲でチェックされます。

decreases

減少量の指定。帰納法で「進むたびに数値が下がる」ことを示せると、応答性も無限深で証明できます。

6. 自動チェック

書かなくても検査されるルールがある

FSL は、よくある落とし穴を「人間が手でルールを書く注意力」に任せず、自動チェックに寄せています。

自動チェック検出するもの代表結果
type_bound範囲つき型、対応表 Map の値、構造体フィールド、列 Seq の実データ部分の範囲外。violated / type_bound
partial_op空の列 Seq への pop/head、範囲外 at、0除算。violated / partial_op
操作の有効性一度も実行可能にならない action と、ブロックしている requires。action_coverage
意図しない停止すべての action が無効になる停止状態。警告または violated
空虚な検証前提が一度も成り立たない含意、到達しない応答性のきっかけ、常に真の requires。warnings または error

重要: type Qty = 0..5 と書いたなら、qty >= 0 のような境界ルールは書きません。FSL が自動で境界を検査するため、手書きすると仕様の意図がぼやけます。

7. 階層化

FSL は層ごとの意図を詳細化チェックでつなぐ

業務ルール、要件、設計、実装は同じ粒度では書けません。FSL は、各層を別の言葉で書き、下位が上位の意図から逸脱していないかを詳細化チェックで確認します。

業務層 business

担当者、業務対象、業務プロセス、統制、ゴール、KPI。業務ステージと統制を読む人向けに書く。

↓ 実装関係・詳細化チェック

要件層 requirements

要件ID、受け入れ手順、禁止手順、非機能要件、SLA。企画担当が確認できる粒度で書く。

↓ 詳細化の対応表

設計層 spec

状態、操作、安全ルール、前後関係、到達性、応答性。実装に近い状態機械として書く。

↓ テスト生成・ログ再生

実装

アダプタやログ投影で、実装の挙動が仕様に合うかを見る。

詳細化チェックが見ること

詳細側の1ステップを抽象側の1ステップまたは「抽象側では何も起きない内部ステップ」に写像し、抽象側の実行条件と状態が壊れないかを調べます。

refinement CartImplRefinesCart {
  impl CartImpl
  abs  ShoppingCart

  maps auto
  map stock[i: ItemId] = impl_stock[i] - reserved[i]
  action reserve(i: ItemId) -> stutter
  action impl_checkout(u: UserId) -> checkout(u)
}

8. 実装接続

仕様は、実装テストの判定基準になる

FSL の検証は「仕様自体の整合性」を見るものです。実装が仕様通りかは、監視器、シナリオ生成、テスト生成、ログ再生でつなぎます。

監視器 Monitor

FSL の具体的な実行器。外部の証明器 Z3 なしで action を1手進め、状態差分を返します。

シナリオ生成 scenarios

到達性、操作の有効性、応答性から統合テスト用のシナリオ JSON を生成します。

テスト生成 testgen

アダプタを埋めるだけの pytest 雛形を生成し、実装と仕様をランダムな操作列で照合します。

ログ再生 replay

実システムのイベントログを仕様に流し、有限ログとして適合するか確認します。

アダプタの契約

class Adapter:
  def reset(self):
    ...

  def step(self, action, params):
    ...

  def observe(self):
    return {
      "status": "Submitted",
      "cart": None,
      "queue": [0, 1]
    }

observe は FSL の論理状態と同じ形に投影します。列挙型は名前、値がない/ある型は None または値、列はリスト、対応表は辞書です。

9. 診断結果

JSON は、AIが修復するための受け渡し形式

FSL の出力は、人が眺めるログではなく、AIが読んで次の修復行動を決めるための構造化データです。

結果意味次の行動
violated到達可能な反例がある。返ってきた手順の最後の操作と状態差分を読み、実行条件・代入・性質を直す。
reachable_failed到達してほしい状態が届かない。深さ不足なら探索深さを上げる。制約が強すぎるなら制約を見直す。
unknown_cti帰納法に必要な補助不変条件が不足、または本物のバグ。CTI の開始状態が現実にあり得るか判断する。
error構文・型・意味・受け入れ手順・禁止手順のエラー。位置、期待された構文、修正ヒントを見て直す。
proved不変条件が無限深で証明された。到達性、空虚な検証、変異検査、実装接続で意図とのズレを確認する。

10. 限界

FSL が保証するもの、しないもの

保証する

書かれた仕様の内部整合性。到達可能な反例、境界違反、未定義操作、到達不能、応答性の未完了、詳細化の逸脱など。

保証しない

仕様が元の業務意図を正しく表していること。確率、百分位、実時間ミリ秒、UIの使いやすさ、外部サービスの実挙動。

実務の要点: 仕様を書く前に、状態・操作・境界条件・例外・優先順位を人が確認します。検証器は「書いたこと」を強く確かめますが、「書くべきだったのに抜けたこと」は、受け入れ手順、禁止手順、空虚な検証の検出、変異検査、人間のレビューで補います。
