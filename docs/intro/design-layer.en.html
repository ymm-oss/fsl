Writing the Design Layer

Turn a requirements contract into
implementable, provable state.

The design layer is an ordinary spec: internal state, actions, queues, and outboxes. A separate refinement mapping proves the design still obeys the requirements above, and the same spec becomes the oracle for implementation conformance below — this file is the design document, not a stand-in for one, so there's nothing else to keep in sync.

1. Boundary

Write implementation-near structure, not policy

The design layer takes an agreed requirements contract and adds the detail an implementer needs: internal substages, queues, two-phase operations, retries, and derived state. It does not re-decide policy — that stays in the layers above.

Write here

Internal substages, queues and outboxes, two-phase operations (submit then confirm), retries, decomposition, derived or cached state, and the refinement mapping up to requirements.

Do not do here

Do not weaken or edit the requirements or business spec to make the design pass. If the upper contract is too strict or ambiguous, report the counterexample and ask for a contract decision.

The design layer is just spec. Unlike business and requirements, there is no dedicated design keyword: you write the kernel spec at an implementation-near altitude, then connect it upward with a mapping file. business ⊒ requirements ⊒ design ⊒ implementation.

2. Basic Shape

An ordinary spec with internal state and actions

Start from the requirement's lifecycle, then add the states and operations the implementation actually has. The internal substages, the ledger, and the outbox below are design-only — they do not appear in requirements.

spec ExpenseDesign {
  type Claim = 0..2
  type Amount = 0..3
  const AUTO_LIMIT = 1
  const OUTBOX_CAP = 3
  enum DesignSt { DesignDraft, DesignAutoApproved, DesignPaymentSubmitted, DesignPaid }
  struct DesignClaim { st: DesignSt, amount: Amount }

  state { design: Map<Claim, DesignClaim>, paid_count: Int, outbox: Seq<Claim, OUTBOX_CAP> }
  init {
    forall c: Claim { design[c] = DesignClaim { st: DesignDraft, amount: 0 } }
    paid_count = 0
    outbox = Seq {}
  }

  fair action pay_submit(c: Claim) {       // visible: the claim becomes paid
    requires design[c].st == DesignAutoApproved
    requires outbox.size() < OUTBOX_CAP
    design[c].st = DesignPaymentSubmitted
    paid_count = paid_count + 1
    outbox = outbox.push(c)
  }
  fair action pay_confirm(c: Claim) {      // internal: settle the submitted payment
    requires design[c].st == DesignPaymentSubmitted
    design[c].st = DesignPaid
  }
  fair action outbox_send() {             // internal: drain the notification outbox
    requires outbox.size() > 0
    outbox = outbox.pop()
  }
}
ConstructMeaningDesign-layer guidance
enum DesignStImplementation-near substages.DesignPaymentSubmitted has no requirement-layer counterpart; it collapses upward later.
state { ... }Internal logical state.Ledgers, queues, and caches that requirements never mention belong here.
Seq<T, N>A bounded queue / outbox.Capacity is finite so the verifier can search it.
fair actionAn operation the verifier may not starve.fair is what later powers preserve progress.

Prove the design on its own first: fslc verify design.fsl --engine induction checks its invariants and partial-operation safety before you connect it upward.

3. Refine Up

Prove the design still obeys requirements with a refinement mapping

The mapping lives in a third file that pollutes neither spec. It maps every internal action either to one requirement action or to stutter ("nothing happens upstairs"), and maps every abstract state variable back out of the internal state.

refinement DesignRefinesRequirements {
  impl ExpenseDesign        // the detailed/design spec
  abs  ExpenseRequirements  // the requirements spec

  map claim_stage[c: Claim] =
    if design[c].st == DesignDraft then Draft
    else if design[c].st == DesignAutoApproved then Approved
    else if design[c].st == DesignPaymentSubmitted then Paid
    else Paid
  map claim_amount[c: Claim] = design[c].amount

  action pay_submit(c)  -> pay(c)     // visible step → requirement action
  action pay_confirm(c) -> stutter     // internal: invisible to requirements
  action outbox_send()  -> stutter
}

fslc refine design.fsl requirements.fsl mapping.fsl --depth 8 runs a bounded forward simulation and reports result: "refines" or a typed failure.

What refine checksMeaningFailure tells you
Init correspondenceThe mapped initial state satisfies the requirement's init.abs_state_mismatch — a map or init bug.
Transition correspondenceEvery design step maps to a requirement action that is enabled, and the update lines up.abs_requires_failed — the design bypassed an upper guard.
StutterAn action mapped to stutter truly leaves the abstract view unchanged.stutter_changed_abs — a false "internal-only" claim.
Mapping rangeMapped values stay inside the abstract type.map_out_of_bounds — the projection escaped the abstract domain.

✅ Safety is included

If refine passes, the design cannot break the requirement invariants and guards (the controls). Internal detail is allowed as long as it does not violate the contract.

⚠ Liveness is not — by default

Because refinement permits stutter, a design can drop a response the requirements guaranteed and still pass. To keep it, opt in with preserve progress (next section).

Two mapping mechanisms hide internal detail. action ... -> stutter hides an internal-only operation, and a map expression absorbs internal fields (for example map stock[i] = impl_stock[i] - reserved[i] hides a reserved field). maps auto synthesizes identity entries for same-named state and actions; explicit entries override it. if … then … else … is allowed only inside mapping expressions, never in an ordinary spec.

4. Preserve Progress

Pull an upper leadsTo through the mapping

When a response the requirements promised still matters, add a preserve progress block. It re-checks the named abstract leadsTo over the design's executions, using the design's own fair actions for fairness.

refinement DesignKeepsLivenessProgress {
  impl DesignKeepsLiveness
  abs  ClaimPolicy

  map s = if d == DSubmitted then Submitted
          else if d == DApproved then Approved
          else if d == DRejected then Rejected
          else Paid

  action route()   -> stutter
  action approve() -> approve()
  action reject()  -> reject()
  action pay()     -> pay()

  preserve progress {
    respond EveryClaimDecided by approve, reject   // by-actions name impl actions; fairness comes from their `fair`
  }
}
If the design…ResultHow to fix
spins in a cycle while the response is pendingprogress_lost (a lasso counterexample)Add or restore fair on the lower progress action.
deadlocks before respondingprogress_lost (a stall)Enable a path to the response, or write a lower leadsTo … decreases … and verify with --engine induction.

Liveness comes only from where you ask for it. preserve progress does not change safety refinement, and fairness is never inherited automatically — it comes from the design's fair action declarations.

5. Compose

Assemble verified components with compose

Large designs are built from parts. compose brings verified specs together under namespaces, runs them in lockstep with synchronized actions, and hides steps that should only fire as part of a larger operation.

compose OrderSystem {
  use ShoppingCart as cart from "cart_v1.fsl"
  use Payment      as pay  from "payment.fsl"

  state { orders_linked: Int }
  init  { orders_linked = 0 }

  fair action checkout_and_pay(u: cart.UserId, p: pay.PayId) =
      cart.checkout(u) || pay.capture(p) {     // both components advance in one step
    requires pay.payments[p].st == Authorized
    orders_linked = orders_linked + 1
  }

  internal cart.checkout                       // fires only via synchronization
  internal pay.capture
}
ConstructMeaningNote
use A as ns from "..."Import a component under a namespace.State and actions are referenced as ns.x.
a.act(...) || b.act(...)A synchronized action: both run together.The merged step shares one requires / body / ensures.
internal ns.actionThe action no longer fires on its own.It only runs as part of a synchronized action.

Fairness is not inherited through synchronization. A synchronized action is fair only if you declare it so; otherwise FSL warns fair_not_inherited. A compose can itself be the impl side of fslc refine, so a whole assembly can be proven against one requirements contract.

6. Handoff

The design becomes the oracle for implementation conformance

A proven design does not prove the code. You connect them: generate tests from the spec, replay real logs against it, or embed a runtime monitor. Each needs an Adapter that projects implementation state into the spec's logical shape.

Monitor

Embed fslc.runtime.Monitor in the implementation to reject any step that would violate the spec at run time.

scenarios

Generate concrete traces (including from covers / acceptance intent) that an implementation must reproduce.

testgen

fslc testgen emits a pytest scaffold that random-walks the implementation with the spec as oracle.

replay

fslc replay checks a finite event log against the spec. Note: leadsTo cannot be judged from a finite log.

The adapter contract

The generated test harness calls Adapter.observe(), which reads the real implementation and returns the state in the spec's logical shape — the same variables, the same bounded types. Conformance is only as good as this projection: if observe() lies, the oracle lies.

class ExpenseAdapter(Adapter):
    def observe(self) -> dict:
        # project the real system into the spec's state variables
        return {
            "design": {c.id: {"st": c.stage, "amount": c.amount} for c in self.repo.all()},
            "paid_count": self.ledger.count(),
            "outbox": [m.claim_id for m in self.queue.pending()],
        }

✅ What the design layer guarantees

The design's own invariants and partial-op safety (with induction, unboundedly); safety inclusion up to requirements via refine; opt-in liveness where you wrote preserve progress + fair; and implementation conformance only where testgen or replay is anchored to real code or logs.

⚠ What it does not

That the requirements themselves are correct (refine assumes abs was verified separately); automatic liveness propagation; real code correctness until a harness is anchored; and anything outside the kernel — probability, percentiles, real-time milliseconds, or external-service behavior.

7. Checklist

Review before implementation

  1. Is every internal action mapped?

    Each design action maps either to one requirement action or to a justified stutter. A missing entry is a type error.

  2. Is every abstract variable mapped?

    The mapping must cover all requirement-layer state, or refine reports a missing map.

  3. Did you prove the design on its own?

    Re-check its invariants with verify and --engine induction before connecting upward.

  4. Does any response still matter?

    If so, cover it with preserve progress and a lower fair action — safety refinement alone will not keep it.

  5. Are proof categories reported separately?

    Design invariants proved, refinement-to-requirements proved, and implementation conformance not proved unless testgen or replay is anchored to real code or logs.
