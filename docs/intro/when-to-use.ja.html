いつ使うべきか

「使うべき？」の答えは、
状態の数では決まらない。

決め手は「壊れ方」。ある操作の順番やフラグの組み合わせで、
あってはならない状態に滑り込めるか——基準はそれだけです。

FSL は全部に使う道具ではありません。
このページは、どこに効いて、どこは要らないかの線引きを持ち帰るためのものです。（10分・専門用語なし）

▽ スクロール

たったひとつの判断軸

「状態の多さ」ではなく「壊れ方」で決める

状態がいくつあるかは関係ありません。問うのはひとつ——
ある操作の順番やフラグの組み合わせで、本来あり得ない状態に滑り込めるか。

❌ 「状態が多い＝使う」ではない

100画面あっても、一本道の閲覧フローなら抜け穴は生まれにくい。数の多さは出番の理由になりません。

✅ 「相互作用で壊れ得る＝使う」

3状態2フラグでも、戻る・取消・再試行・権限分岐が絡めば、人が一目で追えない抜け穴が生まれる。ここが出番です。

だから、小さな確認モーダルにも、巨大な決済フローにも、同じ基準が効きます。
違うのは規模ではなく、「順番やフラグで壊れ得るか」だけ。

判断フロー

3つのゲートを、上から順に通す

上から順に通します。どこかで詰まったら、そこが答えです。

① 状態遷移があるか

箱（状態）と矢印（操作）で描けるか。
No → 静的表示・単純CRUD・装飾。対象外。

↓ Yes

② 相互作用で「危険な状態」に入り得るか

順序・フラグ・権限・非同期・再試行が絡んで、あってはならない状態へ滑り込めるか。
No → 一本道。多くはテストで足りる（低優先）。

↓ Yes

③ 有限・離散で表現できるか

実時間の値・確率・連続量・自由文が「核心」でないか。
No → モデルの核心が乗らない。対象外（下の表）。

↓ Yes

✅ FSL候補

残るは「価値を実現できる前提」を満たすか——それは次の節で。

どこに効くか

効きやすい・効きにくい・そもそも対象外

いちばん大事なのは、「やればできるが薄い（低優先）」と「そもそも表現できない（対象外）」を混同しないこと。

✅ 効きやすい（高利得）

  • 決済・返金・請求
  • 申請・承認・差戻し
  • 予約・在庫・引当
  • 権限・認可・監査ログ
  • キュー・ジョブ・非同期
  • SLA・timeout・再試行
  • 契約状態・プラン変更・解約
  • 画面遷移／ウィザード／二重送信防止／未保存確認／loading中の操作禁止

△ 低優先（できるが薄い）

  • 静的表示だけの画面
  • 単純CRUD
  • 制約の少ないUI装飾
  • 状態遷移がほぼない処理

一本道なら、既存テストで足りることが多い。

⛔ 対象外（表現できない）

  • 実時間の値・連続的なお金の量
  • 確率・期待値・統計的性質
  • 自由文／プロンプトの意味的正しさ
  • 連続量・物理量・組織構造
  • 性能の絶対値（レイテンシ実測）

ここが問題の核心なら、FSLは補助止まり。
（SLAは「離散ステップの相対」としてのみ可）

価値が“偽物”にならない前提

FSLが確かめるのは「仕様」であって、製品ではない

ゲートを通っても、次が欠けると緑のCIが嘘をつきます。「使うべきか」には、価値を実現できる体制まで含めて判断します。

ルールの正しさは別物。 書いたルールが間違っていれば、検証結果もその間違いに基づきます。→ ルールの最終責任者（人間）を必ず置く。

空仕様は“偽の安心”。 反例ゼロは「正しい」のではなく「何も制約していない」のかもしれない。→ fslc mutate のkill-rateで、仕様が実際に効いているかを確認する。

接続するなら Adapter が命。 observe() が実状態を映さなければ、壊れていてもテストが通る。→ Adapterは人間レビューの対象。

liveness は詳細化で伝播しない。 安全性（invariant）は下層へ伝わるが、応答性（leadsTo）は伝わらない。→ 各層で再検証 or preserve progress。

これらを満たせないなら、ゲート①②が○でも「軽量用途（実装前の仕様レビューの補助）に留める」のが正解です。実装接続まで無理に広げない。
（“万能ではない”の全体像は概念ページへ）

二つの「使うべきか」

案件単位の判断と、導入の判断は別もの

「いま、この機能に書くか」と「プロジェクトに導入するか」は、別の問いです。基準も違います。

案件単位：いま、これを書く？

核心の3問に即答できれば足ります。

  • ① 状態遷移があるか
  • ② 順序やフラグで不正状態に入り得るか
  • ③ 有限・離散で書けるか

3つYesなら書く。AIが初稿を書くので作図コストはほぼゼロ——従来の「割に合う線」より大幅に下げてよい。

プロジェクト単位：導入する？

問うのは体制です。

  • ルールの最終責任者がいるか
  • 生成FSLをレビューする人がいるか
  • （接続時）Adapter・ログを用意できるか
  • CIで検証失敗を止められるか

乗数：AIにstate logicを書かせている度合いが高いほど、外部基準としての価値が上がる。進め方は2週間・1フローのPoCから。

どの層で書くか

「使う」と決めたら、層を選ぶ

迷ったら要件層から。業務層は「As-Is/To-Beの統制」が明確なとき、設計層は「合意済みの要件契約」があるときだけ。

層使うとき主な語彙教材
business 業務ポリシー・統制・KPI・As-Is/To-Beの無矛盾／統制保存 actor / stage / policy / goal 業務層
requirements 受入基準・禁止フロー・NFR・要件IDトレース entity / acceptance / forbidden / REQ-ID 要件層
design 内部状態・キュー・compose・要件への詳細化・typestate state / action / mapping 設計層

まとめ

覚えるのは、この一文

状態機械であり、順序・フラグ・権限の相互作用で危険状態に入り得て、有限・離散で書けて、ルールの責任者がいる——このときFSLが効く。
サイズ（状態の多さ）は判断材料ではない。
