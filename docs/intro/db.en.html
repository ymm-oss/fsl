DB / multi-environment compatibility

Use dbsystem to check DB changes and environments as a spec

fsl-db is not just a DB-schema dialect. It models schema changes, migrations, application artifact read/write capabilities, and multi-environment rollout ranges in one machine-checkable specification.

Implementation details live in DESIGN-db.md.

What it prevents

Find DB/application drift before release

During rollout, old artifacts, new artifacts, workers, batch jobs, external clients, and intermediate schemas may coexist briefly. fsl-db models that coexistence as finite snapshots, then checks that every read/write target exists and that NOT NULL transitions happen after backfill.

Read compatibility

Every active artifact must read only columns that exist in its environment and schema window.

Write compatibility

Artifacts must not write columns that have already been removed during a coexistence window.

NOT NULL ordering

Column creation, backfill, and set_not_null are checked so a required column is not introduced before data exists.

Environment windows

environment schema ranges and when schema clauses make prod/staging coexistence explicit.

Multiple environments

Write the artifacts present in each environment, not just the DB schema version

The central idea is capability, not version numbers. Instead of saying "server_v3 is safe," say what each artifact in an environment reads, writes, and which schema window it may coexist with.

artifact server_v1 {
  reads users.id, users.legacy_name;
  writes users.legacy_name;
}

artifact server_v2 {
  reads users.id, users.legacy_name, users.display_name;
  writes users.legacy_name, users.display_name;
}

artifact worker_v1 {
  reads users.legacy_name;
}

environment prod {
  schema 0..4;
  active server_v1 when schema 0..0;
  supported server_v1 when schema 0..1;
  active server_v2 when schema 1..3;
  may_exist worker_v1 when schema 0..4;
}
DeclarationMeaningMain use
activeAn artifact definitely present in that environment and required to be compatible.Running servers, workers, and batch jobs.
supportedAn artifact kept inside the supported compatibility contract.Rollback targets and supported older servers.
may_existAn artifact that may still exist in reality.Old workers, external clients, and approximated mobile/offline clients.
when schema lo..hiThe schema window where that artifact may coexist.Finite modeling of percentage rollout, kill switches, and staged cutovers.

In the MVP, multiple environments are represented by multiple environment blocks. Each schema lo..hi range is the finite set of snapshots reachable through the declared migration plan. It is not an arbitrary Cartesian product of schemas and artifacts; each artifact's when schema window defines coexistence.

Model

The MVP lowers to the existing kernel

dbsystem is not a new verifier kernel. It parses DB declarations into a typed IR and lowers them to the existing state-machine kernel.

dbsystem elementMeaningLowered shape
databaseInitial schema, tables, and column lifecycle.State such as column_exists, column_backfilled, and column_not_null.
migrationOne schema-version transition.One action guarded by schema_version == from.
artifactApplication, worker, API, or job read/write capabilities.Generated invariants and metadata for compatibility rules.
environmentWhich schemas and artifacts coexist.Finite reachable rollout snapshots.

Shape

Write schema, migrations, artifacts, and environments together

dbsystem UserDb {
  database app {
    schema 0
    table users {
      column id: Int present backfilled not_null;
      column display_name: Text absent;
    }
  }

  migration add_display_name from 0 to 1 {
    add users.display_name nullable;
  }
  migration backfill_display_name from 1 to 2 {
    backfill users.display_name;
  }
  migration require_display_name from 2 to 3 {
    set_not_null users.display_name;
  }

  artifact server_v2 {
    reads users.id, users.display_name;
    writes users.display_name;
  }

  environment prod {
    schema 0..3;
    active server_v2 when schema 1..3;
    supported server_v2 when schema 1..3;
  }

  environment staging {
    schema 0..3;
    active server_v2 when schema 1..3;
  }

  check compatibility {
    rule all_active_reads_exist;
    rule all_active_writes_exist;
    rule removed_only_after_unused;
    rule not_null_after_backfill;
  }
}

Run it

Use ordinary check/verify and DB-specific findings

fslc check examples/db/safe_add_nullable_column.fsl
fslc verify examples/db/safe_add_nullable_column.fsl --depth 8
fslc db check examples/db/safe_add_nullable_column.fsl
fslc db check examples/db/safe_dual_write_backfill_switch_read_drop_old.fsl --engine induction
ResultHow to read it
verified_under_assumptionsCompatibility rules hold under finite rollout windows and complete capability declarations.
column_removed_while_still_readAn artifact still reads a column that has been removed in the coexistence window.
column_removed_while_still_writtenAn artifact still writes a column that has been removed in the coexistence window.
not_null_before_backfillA column is made NOT NULL before the backfill step that justifies it.

Boundary

Separate MVP multi-env checks from later phases

Data preservation and rollback

Row-level preservation and up; down equivalence are outside the MVP. The intended future model is refinement/simulation.

API / offline / feature flags

The proposal treats these as important compatibility dimensions, but the MVP formal rules primarily cover DB column read/write compatibility and NOT NULL ordering. API contracts, offline TTL, and feature-flag variants are later phases.

Runtime observation

Logs can provide evidence that old artifacts or undeclared accesses still exist, but log absence is not a formal proof of unused behavior.

SQL/ORM import

Prisma, Rails, Drizzle, SQL DDL, and other importers are outside the MVP. Today, write dbsystem directly.

Time and probability

Percentage rollout, TTL days, and offline clients are not proved as probability or wall-clock time. Model them as finite coexistence windows and ticks.
