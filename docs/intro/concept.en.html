FSL for people who don't know formal methods

You can read a spec.
Whether it's correct,
you can't check.

AI now writes specs, code, and tests at high speed.
So who guarantees they're actually correct?

FSL is a language for letting a machine check your spec by brute force.
You write it in place of the plain-language spec doc you'd write anyway — the same text people read is the text the machine verifies, so it can never drift out of sync.
By the end of this page, you'll feel what that means. (10 min, no jargon)

First, recall

That bug? Probably a "state loophole"

Ever had one of these? They share one shape: in some order of operations, you slip into a state that must never happen.

📦

Shipped before approval

"Ship" became clickable before "Approve", so it shipped while unapproved.

↩️

Lost on Back

Hit Back mid-edit and landed in a limbo screen — neither saved nor cancelled.

💳

Double submit, double charge

Mashed the submit button and the payment ran twice.

🔓

Permission slip-through

A user without permission could change state via a certain order.

Break it down: just three

Software is made of states, operations, and rules

state

What it looks like now

Draft / Submitted / Approved / Shipped … the situations the app can be in.

operation

What you can do

submit / approve / ship / cancel — actions that move the state.

rule

What must hold

"An unapproved order is never shipped." A promise that must never break.

Most bugs are this rule breaking under some order of operations.
So how do you find that break?

Think with an example

Let's draw "an order"

Boxes are states, arrows are operations. One rule to keep:
"An order that hasn't passed Approved never becomes Shipped."

Draft →(submit)→ Submitted →(approve)→ Approved →(ship)→ Shipped. You can cancel partway, too. Looks perfectly fine, doesn't it?

Here's the core

A test is one path; the machine is every path

🧪 Test

Checks one procedure a person thought of.

submit→approve→ship is OK. But you only checked this one path you chose.

🔎 Formal methods (FSL)

The machine explores every possible order.

Try all paths and… a loophole that skips approval to ship appears.

A test only sees the path you chose.
FSL has the machine brute-force whether the rule breaks under any order.

The magic moment

If it can break, the machine shows you the exact steps

Write a rule, hit verify, and FSL answers —
"It breaks. Here are the shortest steps."

The rule being checked (FSL excerpt):

// ship only after approval, supposedly…
action ship() {
  requires status == Submitted   // ← bug: should be Approved
  status = Shipped
}
invariant ShipOnlyAfterApproval { status == Shipped => approved }

That means you get the bug's reproduction steps before you implement.
A tireless QA tries every order and hands you one path that breaks.

Why this works now

AI writes, the machine checks, AI fixes

Old-school formal methods meant experts writing for weeks. FSL flips the order.

01

A human gives intent

State what must hold — like "never ship unapproved" — in words.

02

AI drafts the spec

AI knows FSL's grammar. You don't memorize it.

03

The machine returns counterexamples

fslc returns contradictions, loopholes, counterexamples as JSON.

04

AI reads and fixes

Reads the counterexample, fixes the spec. Repeat until nothing breaks.

Humans only decide intent and make the final business call.
So today it's cheap enough even for a tiny thing like a confirmation modal.

Another trait

From business to implementation, on one line

Handle the same "order" at different granularities. The machine tracks that the upper intent is actually reflected below.

Business (rules)

Returns within 30 days of shipment, etc. Check for business-level contradictions.

↓ refinement

Requirements (PM / acceptance)

Written with requirement IDs. Trace from a counterexample back to the original request.

↓ refinement

Design (state machines / data)

Implementation-near transitions. Check they satisfy the requirements.

↓ scenarios / testgen / replay

Implementation / QA

Generate test scaffolds and scenarios from the spec. Check that impl logs conform.

This isn't a one-time proof before you ship. The whole chain lives in CI: every change to any layer re-verifies it, refine turns an upper-layer edit into a mechanical check of what's below, and drift gets caught on the next commit — not after it reaches production. The same chain of specs also works as onboarding material, for new engineers and for AI agents alike.

Honestly

FSL is not omnipotent

It checks only the range of the rules you wrote. If the rules themselves are wrong, the verdict rests on wrong rules.

It's not a replacement for tests. Implementation, UI, and integrations are tests' job. The two complement each other.

It is a replacement for the spec document, though. The prose design/spec doc you'd otherwise write by hand — FSL is that doc, in a form the machine can also check. fslc html and explain --readable render that same source as reading material for people, so there's no separate document to keep in sync.

A human decides in the end. Whether an AI-written spec truly captures the original rules is for a human to confirm. FSL is an external standard that aids that judgment.

The reveal

What you just experienced is "formal methods"

Write states, operations, and rules; the machine checks every order; if it breaks, it returns a counterexample —
simpler than the scary name suggests.
