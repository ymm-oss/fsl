AI hard-contracts

fsl-ai でAIのtool境界とagent構造を守る

fsl-ai は、決定的に検査できるAI境界を扱います。ai_component はtool宣言、authority、human approval、forbidden tool、fallback、runtime event replayを、agent はrecursive scope、grant、visibility、orchestration、failure policyを扱います。

設計の詳細は DESIGN-ai-hard.md にあります。

境界

hard contract と evaluator / statistical evidence を分ける

fsl-aiは、自然言語回答の真偽を証明しようとはしません。実行前guardの構造事実を検査し、統計品質・migration regression・driftは証明ではなく外部証拠として扱います。

契約例fsl-aiでの扱い
syntactic_hardtool schema、approval token、forbidden tool、precondition evidence。kernel展開と ai_hard_contract_violation。
agent_structuralnested agent grant、visibility、delegation、review gate、tool reachability。agent_structural_violation。構造証拠であり証明ではない。
runtime_observed未宣言toolやschema mismatchがログに出た。observed_contract_violation。証明ではなく観測。
evaluator_supportedgroundedness、source support、prompt injection判定。後続phase。proved とは表示しない。
statistically_supportedaccuracy、recall、hallucination rate、slice指標。fslc ai eval。信頼区間つき証拠であり証明ではない。

構文

tool と authority を明示する

ai_component RefundAgentToolSafety {
  model refund_model_v1;
  prompt refund_prompt_v1;
  input RefundRequestV1;
  output RefundDecisionV1;

  tool SearchOrder {
    schema SearchOrderV1;
    precondition order_exists;
  }

  tool RefundPayment irreversible {
    schema RefundPaymentV1;
    precondition order_paid;
    precondition amount_refundable;
  }

  tool DeleteCustomerData irreversible {
    schema DeleteCustomerDataV1;
  }

  authority {
    may_execute SearchOrder;
    requires_human_approval RefundPayment;
    forbidden DeleteCustomerData;
  }

  fallback {
    when low_confidence require human_review;
  }
}

Recursive agents

入れ子agentは sub_agent ではなく、scope付きagent

agent Parent { agent Child { ... } } は Parent.Child のようなlexical namespaceを作ります。親のauthority/contextは暗黙継承されず、子には明示grantが必要です。実行時の協調は別に orchestration で書きます。

agent SupportOrchestrator {
  context [CustomerTicket, ApprovedSupportDocs];
  tools [SearchDocs, CheckPolicy];
  authority { may_execute [SearchDocs, CheckPolicy]; }

  agent RetrievalAgent {
    grant authority [SearchDocs];
    grant context [ApprovedSupportDocs];
    tools [SearchDocs];
    authority { may_execute [SearchDocs]; }
    output RetrievedSources visibility [parent, PolicyCheckAgent];
  }

  agent PolicyCheckAgent {
    grant authority [CheckPolicy];
    grant context [CustomerTicket, ApprovedSupportDocs];
    tools [CheckPolicy];
    authority { may_execute [CheckPolicy]; }
  }

  orchestration {
    RetrievalAgent -> PolicyCheckAgent;
  }

  failure_policy {
    when RetrievalAgent.failed -> retry up_to 2;
  }
}

展開

ai_component は既存kernelへ展開される

AI dialect は verifier に確率や evaluator semantics を追加しません。hard contract は有限状態と通常の不変条件になります。

有限のtool状態

Tool、human_approved、tool_executed、tool_suggested は通常のkernel stateです。

生成action

approve_* と execute_* が、approval token とguardされた実行を表します。

生成invariant

forbidden tool は実行されず、irreversibleまたはapproval-requiredなtoolは承認後だけ実行できます。

実行

guardとreplayの失敗はAI専用findingで読む

fslc ai check examples/ai/refund_agent_tool_safety.fsl
fslc ai check examples/ai/recursive_support_agent.fsl
fslc ai replay examples/ai/refund_agent_tool_safety.fsl --logs examples/ai/runtime_conformant.jsonl
fslc ai replay examples/ai/refund_agent_tool_safety.fsl --logs examples/ai/runtime_human_approval_bypass.jsonl
fslc ai replay examples/ai/refund_agent_tool_safety.fsl --logs examples/ai/runtime_observed_mismatch.jsonl
結果読み方
verified_under_assumptions有限のhard-contract展開が、完全なcapability宣言とruntime guardを前提に検証された状態。
agent_analyzedrecursive agent graph が構造的に整合している状態。formal_result は not_run のまま。
replay_conformant観測event logが宣言されたhard境界に合っていた。証明ではなく証拠。
ai_hard_contract_violationforbidden tool、approval bypass、schema invalidity、business precondition mismatch が観測された。
observed_contract_violationruntime が宣言外のcapabilityを使った。
