AI hard-contracts / statistical evidence

fsl-ai でAIのtool境界・agent構造・統計品質を1つの規律で扱う

fsl-ai は3つの層でAIシステムを扱います。ai_component はtool authority・human approval・forbidden toolをkernel展開で検査するhard contract層、agent はrecursive agentのgrant・visibility・orchestrationを解析する構造層、そして dataset / evaluator / statistical_property / ai_migration / observed_property は統計品質・migration regression・運用driftを formal_result: "not_run" の外部証拠として扱うevidence層です。

設計の詳細は DESIGN-ai-hard.md と DESIGN-stochastic.md にあります。

何を防ぐか

「動いているように見えるAI」の4種類の事故を検出する

fsl-aiは、自然言語回答の真偽を証明しようとはしません。代わりに、実行前guardで決定的に検査できる境界と、統計・観測として正直に扱うべき証拠を分離し、それぞれの事故を専用findingで返します。

tool境界の突破

forbidden toolの実行、human approvalなしのirreversible実行を、kernel invariantとruntime replayの両方で検出します。

agent権限の漏れ

子agentのgrant超過、sibling間のvisibility leak、low-trust agentからhigh-authority toolへの到達、review-gate bypassを構造解析で検出します。

点推定だけのリリース判断

「accuracyが0.92だった」だけの主張を拒否します。Wilson intervalの下限が閾値を支持しない限り statistically_supported にはなりません。

静かなregressionとdrift

prompt/model/retriever移行によるmetric悪化を fslc ai regress で、本番telemetryの閾値超過とdriftを fslc ai drift で、閾値つき証拠として検出します。

Guarantee boundary

provedと言えるもの・言えないものを最初に固定する

fsl-aiの中心は、契約の種類ごとに「どう扱い、どの語彙で報告するか」を固定したguarantee boundaryです。すべての結果とfindingがこの区分を運びます。

Contract class例扱いResult vocabulary
syntactic / structural hardtool schema、authority、forbidden tool、human approval token、symbolic precondition evidence。静的検査、kernel展開、runtime guard/replay finding。verified_under_assumptions、violated、finding kind ai_hard_contract_violation。
agent structuralnested agentのgrant、visibility、delegation、review gate、tool reachability。graph解析による構造的証拠。証明ではない。agent_analyzed、finding kind agent_structural_violation。
evaluator-backedgroundedness、source support、prompt-injection判定。external evaluator evidence。kernel proofではない。evaluator_supported。proved とは決して表示しない。
statisticalaccuracy、recall、hallucination rate、sliceごとの指標。external stochastic evidence layer（fslc ai eval / regress）。statistically_supported / statistically_unsupported。
observed未宣言toolの実行、log上のschema mismatch、本番telemetryのdrift。fslc ai replay / fslc ai drift の観測証拠のみ。replay_conformant / replay_nonconformant、observed_supported / observed_mismatch、finding kind observed_contract_violation。

prompt injectionやRAG groundednessを「hard」と呼べるのは、検査される述語が構造的でguardに裏づけられている場合だけです。例えば「schema_valid=false のeventではtool callを実行しない」「citation IDが宣言済みの有限allow-listに含まれる」は hard です。「回答がsourceに支持されている」という意味論的主張はevaluatorが必要で、形式証明としては表示できません。

Hard contract

ai_component でtoolとauthorityを明示する

hard contract層は、AI componentを観測可能なtool境界だけでモデル化します。model/prompt/input/outputのmetadata、tool宣言とschema名、symbolic precondition、authority、fallback、そして check hard のrule選択です。

ai_component RefundAgentToolSafety {
  model refund_model_v1;
  prompt refund_prompt_v1;
  input RefundRequestV1;
  output RefundDecisionV1;

  tool SearchOrder {
    schema SearchOrderV1;
    precondition order_exists;
    effect read_order;
  }

  tool RefundPayment irreversible {
    schema RefundPaymentV1;
    precondition order_paid;
    precondition amount_refundable;
    effect payment_refunded;
  }

  tool DeleteCustomerData irreversible {
    schema DeleteCustomerDataV1;
    effect customer_data_deleted;
  }

  authority {
    may_execute SearchOrder;
    requires_human_approval RefundPayment;
    forbidden DeleteCustomerData;
  }

  fallback {
    when low_confidence require human_review;
    when missing_order_evidence require refuse_with_reason;
  }

  check hard {
    rule tool_authority;
    rule human_approval_required;
    rule forbidden_tool_blocked;
    rule tool_schema_declared;
    rule tool_precondition_declared;
  }
}
rule検査内容
tool_authorityforbidden toolが同時にexecutableになっていないこと。
human_approval_requiredirreversible なexecutable toolが requires_human_approval に入っていること。
forbidden_tool_blocked生成されたkernel状態機械に、forbidden toolの tool_executed をtrueにできる遷移が存在しないこと。
tool_schema_declaredexecutable toolがschema名を宣言していること。
tool_precondition_declared宣言されたsymbolic preconditionが、replay時にbusiness precondition evidenceとして照合されること（tool schema適合とは別）。

check hard を省略すると5 rule全部が検査されます（安全なdefault）。また、business/requirements dialectと違い、ai_component のどのフィールドも "description text" tagを受け付けません — すべてbareな識別子か数値です。

展開

ai_component は既存kernelへ展開される

AI dialectは、verifierに確率・percentile・evaluator semanticsを一切追加しません。hard contractは有限状態と通常の不変条件になり、fslc ai check（内部でBMCまたはk-induction）で検査されます。

有限のtool状態

Tool enumと、human_approved、tool_executed、tool_suggested の Map<Tool, Bool>、そして fallback_required: Bool が通常のkernel stateになります。

生成action

suggest_*、approve_*、execute_*、fallback_* actionが、approval tokenとguardされた実行を表します。

生成invariant

forbidden toolは決して実行されず、irreversible / approval-requiredなtoolはapproval後だけ実行できる、という不変条件が生成されます。

fallbackは構造のみ

when <reason> require <target> は fallback_<reason> actionとghost状態に展開されますが、target は自由ラベルなのでinvariantは生成されません。「fallback先が実際に実行された」ことの証明ではありません。reasonはcomponent内で一意である必要があります。

Recursive agents

入れ子agentは sub_agent ではなく、scope付きagent

agent Parent { agent Child { ... } } は Parent.Child というlexical namespaceを作ります。親のauthority/contextは暗黙継承されず、子は明示的な grant で受け取ります。grantは直接の親の境界のsubsetでなければなりません。実行時の協調は別の orchestration graphです。

agent SupportOrchestrator {
  context [CustomerTicket, ApprovedSupportDocs];
  tools [SearchDocs, CheckPolicy, SendEmail];
  authority {
    may_execute [SearchDocs, CheckPolicy];
    requires_human_approval [SendEmail];
  }
  review_gate PolicyCheckAgent;

  agent RetrievalAgent {
    trust medium;
    grant authority [SearchDocs];
    grant context [ApprovedSupportDocs];
    tools [SearchDocs];
    authority { may_execute [SearchDocs]; }
    output RetrievedSources visibility [parent, PolicyCheckAgent];
  }

  agent PolicyCheckAgent {
    trust high;
    grant authority [CheckPolicy];
    grant context [CustomerTicket, ApprovedSupportDocs];
    tools [CheckPolicy];
    authority { may_execute [CheckPolicy]; }
    contract { hard { rule PolicyDecisionCited; } }
    output PolicyDecision visibility parent;
  }

  agent SendAgent {
    trust high;
    grant authority [SendEmail];
    grant context [CustomerTicket];
    tool SendEmail irreversible {
      schema SendEmailV1;
    }
    authority { requires_human_approval [SendEmail]; }
    output SendResult visibility parent;
  }

  orchestration {
    RetrievalAgent -> PolicyCheckAgent;
    PolicyCheckAgent -> SendAgent;
  }

  failure_policy {
    when RetrievalAgent.failed -> retry up_to 2;
    when RetrievalAgent.failed_after_retry -> HumanReviewPending;
  }
}

fslc ai check はこれを agent_analyzed（formal_result: "not_run"）として解析し、graph_summary に6つのgraph — scope_tree、delegation_graph、authority_graph、information_flow_graph、tool_reachability_graph、failure_policy — を分離して返します。構造違反はfinding kind agent_structural_violation で報告されます。

violation意味
child_authority_exceeds_parent_authority子のgrant/tool/authorityが直接の親の境界を超えている。
child_context_exceeds_parent_context子のcontext使用がgrantされたcontextを超えている。
visibility_leak_across_sibling_agentsdelegation pathのないsiblingへのoutput visibility。
low_trust_agent_path_to_high_authority_tooltrust low のagentからhigh-authority toolへの経路。上のsnippetの RetrievalAgent を trust low に変えると実際にこのfindingが返ります。
irreversible_operation_without_human_approval_pathirreversible toolへの経路に requires_human_approval がない。
policy_review_bypass_in_orchestrationhigh-authority toolに到達するorchestration pathが、宣言された review_gate の子を全部すり抜けている。

正確な範囲に注意してください。trust は自由識別子で、文字列 low だけが専用チェック（low_trust_agent_path_to_high_authority_tool）を持ちます。medium / high は構文上通りますが専用チェックはまだありません。contract { hard { rule <Name>; } } はparseされて各agentの agent_ir.contracts（例: {"hard_rules": ["PolicyDecisionCited"]}）に載るだけで、まだ何とも相互検証されません — forward-declaredなmetadataです。ai_component と同様、"description text" tagはどのフィールドも受け付けません。

統計evidence

dataset / evaluator / statistical_property で品質を証拠として縛る

fslc ai eval はprecomputedなeval JSONLを読むdeterministic checkerです。AIモデルもevaluatorも実行せず、Bernoulli/proportion metric（accuracy、recall、hallucination_rate、wrong_tool_call_rate など0/1のもの）だけをWilson intervalで判定します。結果は常に formal_result: "not_run" です。

dataset SupportEvalV3 {
  source "examples/ai/support_eval_v3.jsonl"

  population {
    language in ["ja", "en"]
    ticket_type in ["refund", "bug", "urgent"]
  }

  slice JapaneseRefundTickets {
    language == "ja"
    ticket_type == "refund"
  }
}

evaluator SupportAnswerJudge {
  input question: Question
  input answer: Answer
  output grounded: Bool
  output confidence: Float

  calibration {
    dataset HumanLabeledGroundednessV2
    require agreement_with_human >= 0.90
  }
}

statistical_property LooseQuality {
  target SupportAnswerAgent
  dataset SupportEvalV3
  evaluator SupportAnswerJudge
  confidence 0.95

  require ci_lower(metric.accuracy, 0.95) >= 0.45

  slice JapaneseRefundTickets {
    require min_samples >= 5
    require ci_lower(metric.accuracy, 0.95) >= 0.35
  }
}

eval recordは1行 = 1つのBernoulli観測です（schema fsl-ai-eval-record.v0。case_id / slice / metric / boolean outcome / evaluator.id が必須で、同一 (case_id, slice, metric) の重複は dataset_invalid）。

{"schema_version":"fsl-ai-eval-record.v0","case_id":"support-001","component":"SupportAnswerAgent","dataset":"SupportEvalV3","slice":"all","metric":"accuracy","outcome":true,"evaluator":{"id":"SupportAnswerJudge","calibration_status":"trusted"}}

n 個の観測中 k 個成功なら estimate = k / n で、confidence c のWilson intervalから ci_lower / ci_upper が出ます。requireは ci_lower(metric.X, 0.95) >= T か ci_upper(metric.X, 0.95) <= T の形だけです。実際にexampleを走らせた結果:

propertyslicenestimateci_lower (95%)閾値result
LooseQualityall100.80≈ 0.490≥ 0.45statistically_supported
LooseQualityJapaneseRefundTickets51.00≈ 0.566≥ 0.35（+ min_samples >= 5）statistically_supported
StrictQualityJapaneseRefundTickets51.00≈ 0.566≥ 0.80statistically_unsupported

3行目が要点です。sliceで5/5全問正解（estimate 1.00）でも、n=5ではWilson下限は約0.566にしかならず、0.80という主張は支持されません。「全部通ったからOK」をサンプル数ごと拒否するのがこの層の仕事です。

優先順位status意味
1dataset_invalidrecord schema違反、slice/metricの重複case、必須sliceフィールド欠落。
2evaluator_untrustedevaluatorのcalibration evidenceが欠落、または宣言されたtrust閾値未満。
3insufficient_samples必須sliceのどれかが n < min_samples。
4inconclusive宣言されたmetricやinterval methodが計算できない。点推定のみのrequireもここで拒否。
5statistically_unsupportedサンプルは足りているが、Wilson boundが閾値を支持しない。
6statistically_supportedすべての必須gateとbound checkが通過。

require accuracy >= 0.92 のような点推定のみの要求は、警告で通過するのではなく、eval時に inconclusive（exit 1）として拒否されます。また各sliceは独立したgateで、全sliceが自分の min_samples とbound checkを通ったときだけ全体が supported になります。family-wiseな多重検定補正はこの層では主張されません。

Migration / drift / compat

移行のregression、本番のdrift、環境互換性も同じ規律で扱う

ai_migration はprompt/model/retriever移行の前後で「hard contractの維持」と「metricの非退行」を宣言し、observed_property は本番telemetryの閾値とdriftを宣言します。どちらも外部証拠で、formal_result: "not_run" です。

ai_migration PromptV7ToV8 {
  from SupportAnswerAgent {
    model gpt_5_5
    prompt support_answer_prompt_v7
    retriever support_docs_index_v12
  }

  to SupportAnswerAgent {
    model gpt_5_5
    prompt support_answer_prompt_v8
    retriever support_docs_index_v14
  }

  preserve {
    hard_contract SupportAnswerContract.hard

    no_regression {
      dataset SupportEvalV3
      metric accuracy drop <= 0.05
      metric hallucination_rate increase <= 0.02
    }
  }
}

observed_property SupportAgentOperationalQuality {
  target SupportAnswerAgent
  source production_logs
  window last_7_days

  require observed(metric.hallucination_rate) <= 0.30
  require drift(metric.refusal_rate) <= 0.10 compared_to previous_7_days
}

exampleのv7→v8（意図的にregressさせたデータ）を実際に走らせた結果:

コマンド検査実測判定
fslc ai regressaccuracy drop <= 0.051.00 (10/10) → 0.80 (8/10)、observed_delta 0.20 > allowed_delta 0.05passed: false → finding ai_migration_regression
fslc ai regresshallucination_rate increase <= 0.020.00 (0/5) → 0.20 (1/5)、observed_delta 0.20 > allowed_delta 0.02passed: false、全体は statistically_unsupported
fslc ai compare閾値判定なしのdelta報告accuracy delta −0.20、hallucination_rate delta +0.20result: "compared"（findingなし）
fslc ai driftobserved(metric.hallucination_rate) <= 0.30現在 0.25（n=4）passed: true
fslc ai driftdrift(metric.refusal_rate) <= 0.10baseline 0.10 → 現在 ≈0.667、drift ≈ 0.567passed: false → observed_mismatch、finding ai_observed_drift（guarantee_kind: "runtime_observed"）

環境互換性はfsl-dbと共有です。fslc ai compat --environment prod は、componentのmodel/prompt/retriever/tool/output schemaを dbsystem のartifact capability profileとして生成します（result: "compat_profile_generated"）:

artifact support_answer_agent {
  requires model.gpt_5_5, prompt.support_answer_prompt_v8,
           retriever.support_docs_index_v14, tool.CreateTicket, tool.SearchDocs;
  provides output.AnswerSchemaV2;
}

この断片を dbsystem のenvironmentに組み込み、fslc compat check --include-ai で、server/mobile/DB artifactと同じ有限snapshot上のcapability互換性として検査します。provider側の欠落は required_capability_missing になります。

Finding contract

すべてのfindingは fsl-ai-finding.v0 で同じ形をしている

hard contract違反も、agent構造違反も、統計不支持も、driftも、全部同じfinding schemaで返ります。必須フィールドは fsl、result、kind、severity、component、contract、tool、failed_rule、violation、guarantee_kind、evidence、witness、minimal_conflict_set、repair_candidates、assumptions です。repair_candidates は各候補に weakens_spec flagを持ち、「specを弱めて通す」修理を明示的に区別します。

guarantee_kind発行元意味
syntactic_hardai check / ai replay のhard-contract violation宣言境界とguard前提の上の決定的事実。
agent_structuralrecursive agent解析graph上の構造的証拠。証明ではない。
runtime_observedreplayのlog mismatch、ai drift観測証拠。logにないことは未使用の証明ではない。
evaluator_supported将来のevaluator findingevaluator証拠。proved とは表示しない。
statistically_supported / statistically_unsupportedai eval / ai regressWilson boundつき統計証拠。proved とは表示しない。

境界

検査されるものと、認識されるだけのものを混同しない

検査されないblock境界

ai_action、単独の retriever block、trust_boundary、トップレベルの authority { target ... }（exampleの ai_contract も同様）は、block境界としてのみ認識されます。parserは本文に一切降りないため、中身がgarbageでも check を通ります。raw_blocks に {kind, name} として列挙されるだけで、何も制約しません。

検査される面はこれだけ

kernel裏づけの ai_component / agent と、外部証拠の dataset / evaluator / statistical_property / ai_migration / observed_property。この外にあるものに契約を書いても効きません。

forward-declaredなmetadata

agentの contract { hard { rule ...; } } と failure_mode は、parseされて agent_ir.contracts / failure_modes に載りますが、まだevidenceと相互検証されません。追跡されるmetadataであって、検証済みの主張ではありません。

evaluatorを実行しない

fslc ai eval はprecomputed JSONLを読むだけです。LLM evaluatorの実行、provider sampling、promptの変異は行いません。boolean outcomeのみ対象で、連続scoreは対象外です。evaluatorの信頼はcalibration evidenceの問題として扱われます（欠落は evaluator_untrusted）。

formal_result: "not_run" の一貫

replay、agent解析、eval、regress、compare、drift、compatの結果はすべて formal_result: "not_run" を運びます。これらを proved / verified として表示してはいけません。kernelの形式結果を持つのは ai_component のhard-contract展開だけです。

時間・確率・LLM真偽

「7日間window」はlog選別のラベルであって実時間の証明ではなく、driftは2つの観測窓の差です。LLM出力の真偽・groundednessそのものは、この dialect のどの層でも証明されません。
