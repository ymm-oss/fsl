1. Model

FSL views an application as states plus operations

FSL does not verify arbitrary implementation code. It extracts the state relevant to the behavior you care about, the operations that move that state, and the rules that must hold.

State stateThe logical variables the verifier tracks.
Initial State initThe deterministic starting point.
Operation actionOne atomic step, enabled by requires.
Propertyinvariant, trans, reachable, or leadsTo.
CounterexampleIf a property breaks, FSL returns the shortest trace.

Small FSL state machine

spec OrderFlow {
  enum St { Draft, Submitted, Approved, Shipped }
  state { status: St, approved: Bool }
  init { status = Draft approved = false }

  action approve() {
    requires status == Submitted
    status = Approved
    approved = true
  }

  action ship() {
    requires status == Approved
    status = Shipped
  }

  invariant ShipOnlyAfterApproval {
    status == Shipped => approved
  }
}

How to read it

FSLMeaning
stateVariables tracked by the verifier.
initThe starting point for exploration.
actionA transition that may run when enabled.
requiresThe enabled condition.
invariantA rule that must hold in every reachable state.

2. Finite State Space

FSL makes the model finite so the machine can search it

Real systems are unbounded. FSL models the part that matters with bounded integer types, enums, sequence capacities, and verification bounds.

Bounded integer type

type Qty = 0..5
type UserId = 0..2

Out-of-range values are detected automatically as type_bound.

Enum

enum St { Open, Closed }

Use enums for stages, statuses, and finite categories.

Sequence capacity

const CAP = 3
state { q: Seq<JobId, CAP> }

Full push, empty pop/head, and out-of-range at are diagnosed.

Dialect bounds

verify {
  instances Claim = 3
  values Amount = 0..10
}

Business and requirements layers put entity counts and numeric ranges in verify.

3. BMC

Bounded model checking explores all paths up to K steps

BMC stands for bounded model checking. Starting at init, it explores every combination of action instances at 0, 1, 2, up to K steps. If a counterexample exists, the shortest trace is returned.

Run

$ fslc verify specs/cart_v1.fsl --depth 8
{
  "result": "verified",
  "completeness": "bounded",
  "checked_to_depth": 8
}

Meaning

ResultHow to read it
verifiedNo counterexample up to depth K. This is not an infinite proof.
violatedA counterexample exists. The returned trace is the shortest reproduction.
reachable_failedA desired state was not reached within K steps.

  1. Place one state

    First build the init state. Every invariant is checked here too.

  2. Enumerate enabled action instances

    For each action and parameter-value combination, check whether requires holds.

  3. Build the next state

    The action body is evaluated as a simultaneous assignment. Variables you do not write are preserved automatically.

  4. Check properties

    Safety rules, before/after relations, bounds, partial operations, deadlock, and liveness are all verified.

4. Induction

k-induction proves more than reachable paths

BMC checks states reachable from init. k-induction additionally asks whether every state satisfying the invariants keeps satisfying them after one more step.

Base case

Checks that reachable paths up to depth K have no counterexample.

Step case

Checks that arbitrary invariant-satisfying states remain safe after one more step.

unknown_cti means information missing for the proof

A failed step case is not necessarily a reachable bug. CTI stands for counterexample-to-induction: a candidate saying "if this state could exist, the next step would break the invariant."

{
  "result": "unknown_cti",
  "cti": {
    "states": [
      { "status": "Approved", "approved": false },
      { "status": "Shipped", "approved": false }
    ]
  }
}

In this example, an Approved order should always have approved == true. Stating that fact as an auxiliary invariant makes the proof go through.

invariant ApprovedImpliesFlag {
  status == Approved => approved
}

The decision: if the CTI start state is genuinely reachable, it is a spec or design bug. If it is unreachable but ruled out by a domain truth, add an auxiliary invariant that expresses that truth.

5. Liveness

leadsTo checks "eventually" properties

An invariant says "bad states never occur." A leadsTo property says "once P holds, Q eventually holds."

Example

fair action finish(j: JobId) {
  requires jobs[j] == Running
  jobs[j] = Done
}

leadsTo EventuallyDone {
  forall j: JobId {
    jobs[j] == Running ~> jobs[j] == Done
  }
}

Typical failures

ShapeMeaning
LassoP remains pending inside a logical loop.
StallP remains pending and no action is enabled.
Ranking failuredecreases cannot prove progress.

fair

Weak fairness. If an action stays enabled throughout a loop, it is assumed to eventually run. Mark only the actions that carry progress.

within K

P ~> within K Q is a bounded response within K steps. It is checked inside the BMC range.

decreases

A ranking measure. If induction can show a value strictly decreases at every step, liveness is proven even at unbounded depth.

7. Layers

FSL connects layer intent through refinement

Business, requirements, design, and implementation are not written at the same level. FSL lets each layer use its own vocabulary, then checks that lower layers preserve upper-layer intent.

Business business

Actors, entities, processes, policies, goals, KPIs.

↓ implements / refinement

Requirements requirements

Requirement IDs, acceptance flows, forbidden flows, NFRs, SLA.

↓ refinement mapping

Design spec

State, actions, invariants, reachability, liveness.

↓ testgen / replay

Implementation

Adapter or logs are checked against the logical state.

What the refinement check actually inspects

Each step on the concrete side maps to one step on the abstract side, or to an internal step that does nothing on the abstract side (a stutter). The check then verifies that init correspondence, transition correspondence, and the abstract enabled conditions and state all hold.

refinement CartImplRefinesCart {
  impl CartImpl
  abs  ShoppingCart

  maps auto
  map stock[i: ItemId] = impl_stock[i] - reserved[i]
  action reserve(i: ItemId) -> stutter
  action impl_checkout(u: UserId) -> checkout(u)
}

Safety is included in this inclusion: every behavior the implementation allows must be one the abstract spec already allows. Liveness is not carried down automatically unless you opt in with preserve progress.

8. Implementation

The spec becomes the oracle for conformance tests

Verification checks the spec's internal consistency. Implementation conformance is connected through runtime monitoring, scenario generation, test generation, and log replay.

Monitor

Concrete FSL executor. Steps one action without Z3 and returns state diffs.

scenarios

Generates integration-test skeletons from reachability, action coverage, and liveness.

testgen

Generates a pytest scaffold where you fill in an implementation adapter.

replay

Checks whether an observed event log conforms to the FSL state machine.

The adapter contract

class Adapter:
  def reset(self):
    ...

  def step(self, action, params):
    ...

  def observe(self):
    return {
      "status": "Submitted",
      "cart": None,
      "queue": [0, 1]
    }

observe() projects the implementation's state into the spec's logical shape: enums become their names, optional types become None or a value, sequences become lists, and maps become dicts. The verifier then compares this projection against what the spec predicts.

9. Reading Results

JSON is the handoff format an AI uses to repair

FSL output is not a log a human skims. It is structured data an AI agent reads to decide the next repair action, with the verdict, counterexample trace, and unsat-core all in machine-readable form.

ResultMeaningNext action
violatedA reachable counterexample exists.Read the last action and state diff in the returned trace, then fix the enabled condition, assignment, or property.
reachable_failedA desired state is not reached.Raise the search depth if it is too shallow; relax the constraints if they are too strong.
unknown_ctiAn auxiliary invariant is missing for induction, or a real bug.Decide whether the CTI start state can really occur.
errorSyntax, type, semantic, acceptance-flow, or forbidden-flow error.Use the location, expected syntax, and fix hint to repair it.
provedAn invariant is proven at unbounded depth.Check intent with reachability, vacuity, mutation testing, and implementation conformance.

10. Limits

What FSL guarantees, and what it does not

Guarantees

The internal consistency of the spec as written: reachable counterexamples, bounded-type violations, partial-operation safety, unbounded proofs once induction succeeds, safety inclusion under refinement, and conformance wherever an adapter or log is anchored.

Does not guarantee

That the spec captures the original intent. Liveness is not propagated down automatically; real code is unchecked until testgen or replay anchors it; probabilities, percentiles, real-time milliseconds, and UI usability are out of scope; and verification is bounded by default unless induction is used.

Practical point: before writing a spec, a human confirms the states, operations, boundary conditions, exceptions, and priorities. The verifier strongly checks what you wrote; what you should have written but omitted is covered by acceptance flows, forbidden flows, vacuity detection, mutation testing, and human review.
