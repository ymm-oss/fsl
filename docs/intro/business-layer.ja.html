業務層を書く

業務ルールを、
検証できる骨格にする。

業務層 business は、担当者、業務対象、業務プロセス、統制、ゴール、KPIを書く層です。実装の都合ではなく、業務として守るべき流れを先に固定します。

このページで決めること

1. 役割業務層に書くこと・書かないこと 2. 基本形actor・entity・process 3. 統制とゴールpolicy・goal・kpi・control・governance 4. 検証範囲verify の決め方 5. 要件層への渡し方名前をそろえて詳細化につなぐ 6. チェックリストレビュー時に見ること

ファイルを分ける理由: 業務層は「何を業務として許すか」を書きます。要件層は「どの入力・条件・受け入れ手順で満たすか」を書きます。混ぜると、業務判断と仕様実装の判断が区別しにくくなります。

1. 役割

業務層は、実装語ではなく業務語で書く

業務層は、業務としてのライフサイクルと統制を読むための層です。画面、API、キュー、DBフラグなどはまだ出しません。

書く

担当者、業務対象、ステージ、遷移、業務統制、到達したいゴール、KPI。例: 返品申請は承認または却下される、承認後だけ返金できる。

書かない

DBカラム、画面遷移、API名、非同期ジョブ、リトライ詳細、ミリ秒単位の時間、確率、組織図の細部、説明文だけの根拠。

判断基準: 業務担当者がそのまま読んで「この流れは正しい」と判断できるなら業務層です。実装者でないと意味が取れない要素は、要件層か設計層へ下げます。

2. 基本形

actor・entity・process で業務の流れを書く

最初は、業務対象がどのステージを進むかだけを書きます。FSL はこの流れから、検証できる状態機械を作ります。

business ReturnHandling {
  actor Customer, Manager
  entity Return

  process Return {
    stages Requested, Approved, Rejected, Refunded
    initial Requested
    transition approve Requested -> Approved by Manager
    transition reject  Requested -> Rejected by Manager
    transition refund  Approved  -> Refunded by Manager
  }
}

verify {
  instances Return = 3
}
構文意味書くときの目安
actor業務操作を行う担当者一覧。遷移の by に出る名前を並べます。
entity業務対象。返品、申請、注文、契約など、1件ずつ状態を持つもの。
process業務対象のステージ機械。状態名と、業務として許す遷移だけを書きます。
transition業務上の1操作。誰が、どのステージからどのステージへ進めるか。

3. 統制とゴール

守るルールは policy、到達したい状態は goal

業務層では「起きてはいけないこと」と「できるべきこと」を分けて書きます。集計したい数は kpi として置きます。

よく使う読みやすい形

kpi refunded = count Return in Refunded

policy EveryRequestDecided "すべての依頼はいずれ判断される"
  every Return in Requested must eventually be Approved or Rejected or Refunded

goal AllSettled "すべての依頼は完了状態にできる"
  all Return can be Refunded or Rejected

式で書く形

policy ApprovedEventuallyRefunded "承認済みはいずれ返金される" responds {
  forall r: Return {
    stage(r) == Approved ~> stage(r) == Refunded
  }
}

goal SomeRefunded "返金完了の例が存在する" {
  exists r: Return { stage(r) == Refunded }
}
目的構文失敗時にわかること
業務上の禁止・安全性policy ... invariant { ... }業務ルールを破る手順。
放置禁止・いつか完了policy ... every ... must eventually ...放置されるケースや進まないループ。
到達可能性goal ... can reach ...業務上できるはずの完了状態に届かない理由。
集計kpi name = count Entity in Stage説明・シナリオ用の投影。自動でルールにはなりません。

control で統制を追跡し、governance でまとめる

control は名前つきの統制メタデータ（owner・severity・applies_to）で、それ自体は性質を生みません。satisfies で検査可能なルールに結びつけると、違反時に「どの統制が破れたか」が出ます。複数の業務specで共有する統制は、独立した governance カタログにまとめます。

統制を宣言して結びつける

control CTRL-DECISION
  "すべての返品は必ず審査される"
  owner Manager
  severity high
  applies_to Return

policy EveryRequestDecided "すべての依頼はいずれ判断される"
  satisfies CTRL-DECISION
  every Return in Requested must eventually be Approved or Rejected or Refunded

共有の統制をカタログにまとめる

governance EnterpriseReturnControls {
  authority Operations owns CTRL-DECISION
  control CTRL-DECISION "すべての返品は必ず審査される"

  delegates ReturnHandling from "return_policy.fsl" {
    require CTRL-DECISION
    CTRL-DECISION is satisfied_by policy EveryRequestDecided
  }

  preservation ReturnReform {
    before AsIsReturn from "asis_return.fsl"
    after  ToBeReturn from "tobe_return.fsl"
    preserve CTRL-DECISION
    checked_by refinement "tobe_refines_asis.fsl"
  }
}
構文意味注意
control ID "..." owner ... severity ... applies_to ...統制のメタデータ。カタログの項目のみ。性質は生成しません。
policy/goal ... satisfies CTRL検査可能なルールを統制に結びつける。違反時、JSONに requirement.controls が載ります。
governance { authority ... owns ... }業務をまたぐ独立した統制カタログ。業務specの一段上で、所管（オーナー）を記録します。
delegates B from "..." { require / is satisfied_by }各統制を業務の policy/goal に対応づける。fslc check が、要求した統制をすべて満たすか検証します。
preservation { before / after / preserve / checked_by refinement }As-Is → To-Be の改革をまたいで統制を保つ。詳細化を深さ8で実行し、refines を報告します。

fslc check governance.fsl は、参照する統制・業務ファイル・満たす policy/goal を検証し、各 preservation の詳細化を実行します。結果は governance.preservations に入ります。未知の統制参照はエラー、宣言したのに使われない統制は警告です。

4. 検証範囲

verify で、何件の業務対象を同時に見るか決める

業務層の entity は、verify ブロックで件数を指定します。件数が増えるほど相互作用を見つけやすくなりますが、検証は重くなります。

単体の流れを見る

verify {
  instances Return = 1
}

1件だけで基本のステージ遷移を確認します。

複数件の相互作用を見る

verify {
  instances Return = 3
}

集計、担当者、同時進行、放置の問題を見るなら3件程度から始めます。

数値は要件層へ送る

number Amount
verify { values Amount = 0..3 }

金額や日数の境界は、入力条件を書く要件層で扱うのが基本です。

5. 要件層への渡し方

名前をそろえると、要件層が業務層を実装しやすい

要件層は implements で業務層につながります。業務層の対象名、ステージ名、遷移名を不用意に変えないと、対応関係が読みやすくなります。

業務層

business ExpenseToBe {
  actor Employee, Manager, Finance, System
  entity Claim
  process Claim {
    stages Draft, Submitted, Approved, Rejected, Paid
    initial Draft
    transition submit Draft -> Submitted by Employee
    transition approve Submitted -> Approved by Manager
    transition reject Submitted -> Rejected by Manager
    transition pay Approved -> Paid by Finance
  }
}

要件層

requirements ExpenseRequirements {
  implements ExpenseToBe from "business.fsl" { }

  number Amount
  process Claim with amount: Amount {
    stages Draft, Submitted, Approved, Rejected, Paid
    initial Draft
    transition submit Draft -> Submitted by Employee with a: Amount
      when a > 0 set amount = a
      covers REQ-1 "申請者が金額を入力して申請する"
  }
}

注意: 業務層を要件に合わせて弱めてはいけません。詳細化チェックに失敗したら、まず要件層が業務意図を落としていないかを見ます。

6. チェックリスト

業務層レビューで見ること

  1. 業務対象は1件ずつ状態を持つものか

    申請、返品、注文、契約など。単なる入力値や設定値は entity にしません。

  2. ステージ名は業務担当者の言葉か

    Queued や Retrying のような実装状態は、必要になるまで設計層へ下げます。

  3. 放置してはいけない状態に応答性があるか

    every ... must eventually ... で「申請済みのまま残らない」などを明示します。

  4. KPIをルールと混同していないか

    kpi は集計投影です。「必ずN件以下」のような制約は、別の policy や要件層で書きます。

次に、要件層で具体化する

業務層で「何を守るか」を固定したら、要件層で入力、条件、受け入れ手順、禁止手順、非機能要件を書きます。
