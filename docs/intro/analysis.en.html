Structural observation

Use fslc analyze to create a review entry point

fslc analyze turns an FSL spec into a review map: which requirements, actions, state variables, and properties are connected. Use it before or after verification to find impact scope, weakly anchored declarations, and traceability gaps quickly.

Implementation details live in DESIGN-analysis.md. A deeper, team-facing practical guide exists as GUIDE-analyze.ja.md — currently Japanese-only.

What it does

Narrow the review before reading the whole spec

The slow part of review is often not the final correctness judgment. It is discovering what to read and how an upper requirement maps to lower design. analyze does that discovery mechanically first.

See impact scope

action_state_graph lists which state variables each action reads and writes. That makes it easier to see what a changed action can affect without relying only on grep or full manual reading.

Follow requirement links

requirement_property_graph and traceability_graph connect requirement IDs, properties, scenarios, and lower-layer design actions. Use them to check where upper intent lands in design.

Surface review candidates

--profile ai-review flags review-required structures such as weakly anchored properties, unguarded updating actions, and state that is read but never written. These are prioritization signals, not violation verdicts.

Share the same view

Outputs are available as JSON, DOT, Mermaid, and JSON Schema-backed contracts. The same review view can feed PR descriptions, CI, editor diagnostics, and visualization tools.

Boundary

Analyze is not verify

fslc analyze builds deterministic structural JSON from the validated spec model. It does not call Z3, explore reachable states, replay logs, or prove refinement.

Use it for review

Find disconnected requirements, unanchored properties, graph cycles, and traceability gaps before deeper verification work.

Keep proof separate

Use verify, refine, and replay when you need formal evidence or implementation conformance.

Read the status honestly

Structural findings carry formal_status: "not_a_violation". A cycle is not automatically a bug.

AI-review

Initial findings

disconnected_requirement

A requirement is declared but has no useful structural anchor such as an action, property, scenario, KPI, or control.

unanchored_property

A user property has no requirement tag, scenario/governance anchor, or action-state anchor.

progressless_cycle

A requirement/scenario-linked multi-action cycle has no visible progress story such as leadsTo, fair action, or terminal.

unwritten_state

A state variable is initialized and read, but no action writes it. If it is fixed data, consider making that explicit; otherwise suspect missing behavior.

unread_state

A state variable is written, but no transitive relevance chain reaches a guard, property, ensures clause, or scenario.

unguarded_action

An action that writes state has no requires clause. Review whether it should really be always enabled or whether a guard was omitted.

conservation_candidate

Counter-like effects structurally preserve a weighted sum. Treat it as a candidate invariant, then prove it with verify.

tag_stale_reference / tag_formula_disjoint

Exact code-shaped tag identifiers are missing from the current spec or the tagged formal definition. This is local drift evidence, not natural-language meaning judgment.

divergent_choice

A depth-4 BMC witness shows two actions enabled in one reachable state and splitting an invariant or acceptance outcome. The finding asks which outcome is intended and remains visible even when acknowledged.

unconstrained_effect

A depth-4 BMC witness shows two actions writing different next values to state outside the contract-observation graph. It supersedes the duplicate unread_state signal, but is never suppressed by acknowledgement.

The cycle finding does not inspect English keywords in names. The underspecification findings are bounded review evidence, not proof of determinism: absence means only “not witnessed within depth 4.” An exact match with an undecided: declaration adds acknowledged:true and acknowledged_by; unmatched findings have no acknowledgement fields and remain unresolved. The tag records intentional deferral and is not a verification condition. See DESIGN-undecided.md.

Limits

Make the boundaries explicit

No natural-language inference

Requirement text is carried through the graph, but the core analyzer does not interpret its meaning.

No proof evidence

Findings, project traceability, and refinement graphs show structure only. Run verify, refine, or chain for counterexamples, reachability, or refinement results.

No automatic fixes

Candidate repairs are options. Add a guard, make a value constant, or add a requirement tag only after confirming the intended behavior.

Editor signals are opt-in

fslc-lsp surfaces analysis findings as informational diagnostics when FSLC_LSP_ANALYSIS_DIAGNOSTICS=1 is set.
