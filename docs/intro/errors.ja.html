エラー・終了コード

「反例を読んで直す」がFSLの中核ループです。ここはresult/kind/violation_kindの種類と、終了コードの対応表、そして各種の読み方索引です。

契約

JSONエンベロープと終了コード

すべてのコマンドは stdout に1つのJSONを出し、resultフィールドが終了コードに対応します。この対応（exit_code()）はCLIリファレンスにソースそのものが載っています——ここでは要点だけを表にします。

終了コード意味代表的なresult
0成功verified / proved / refines / conformant / generated / typestate / mutated / explained / ok など
1検査は完走したが不成立violated / reachable_failed / unknown_cti / nonconformant / refinement_failed など
2仕様エラーresult: "error" かつ kind が parse/name/type/semantics/io（--vacuity error時の空虚性エラーもここ）
3内部エラーresult: "error" かつ kind: "internal"

verifiedでも読むもの

resultが成功でも見るべき場所

warnings

verified/provedでもwarningsは空とは限りません。デッドロック到達可能性など、バグではないが確認すべき情報が入ります。

completeness

"bounded"は「その深さまでは反例が無い」、"unbounded"は帰納法による無限深さの証明です。仕組みと使い方で違いを詳しく説明しています。

unknown_cti

k帰納法が反例候補（CTI）を返す場合。到達可能かどうか未確定なので、補助不変条件を足すのが定石です——仕組み詳細を参照。

action_coverage

一度も有効化されなかったアクションが無いかの診断。書いたつもりのアクションが実は到達不能、というミスを見つけられます。
