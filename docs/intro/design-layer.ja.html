設計層を書く

要件契約を、
実装でき、証明できる状態にする。

設計層は普通の spec です。内部の状態・操作・キュー・アウトボックスを書き、別ファイルの refinement 対応表で「上位の要件をまだ守っていること」を証明します。同じ spec が、下位の実装適合の判定基準にもなります——このファイルそのものが設計ドキュメントであり、別に保守すべき文書は存在しません。

このページで決めること

読み方の前提: spec、refinement、map、stutter、compose などはFSLの予約語です。本文では「設計層」「詳細化対応」「内部操作」のように日本語名を併記します。

1. 役割

実装に近い構造を書く。ポリシーは書かない

設計層は、合意済みの要件契約を受け取り、実装者に必要な詳細を足す層です。内部サブステージ、キュー、二段階操作、リトライ、導出状態などを書きます。ポリシーを決め直す層ではありません。

ここに書く

内部サブステージ、キューとアウトボックス、二段階操作（提出してから確定）、リトライ、分割、導出・キャッシュ状態、そして要件層への refinement 対応表。

ここでしてはいけない

設計を通すために、要件層や業務層を弱めたり書き換えたりしないこと。上位の契約が厳しすぎる／曖昧なら、反例を示して契約の判断を仰ぎます。

設計層は spec そのものです。 business や requirements と違い、design という専用予約語はありません。実装に近い抽象度で核言語の spec を書き、対応表ファイルで上位につなぎます。business ⊒ requirements ⊒ design ⊒ implementation という関係です。

2. 基本形

内部状態と操作を持つ、普通の spec

要件のライフサイクルから出発し、実装が実際に持つ状態と操作を足します。下の内部サブステージ、件数台帳、アウトボックスは設計層だけのもので、要件層には現れません。

spec ExpenseDesign {
  type Claim = 0..2
  type Amount = 0..3
  const AUTO_LIMIT = 1
  const OUTBOX_CAP = 3
  enum DesignSt { DesignDraft, DesignAutoApproved, DesignPaymentSubmitted, DesignPaid }
  struct DesignClaim { st: DesignSt, amount: Amount }

  state { design: Map<Claim, DesignClaim>, paid_count: Int, outbox: Seq<Claim, OUTBOX_CAP> }
  init {
    forall c: Claim { design[c] = DesignClaim { st: DesignDraft, amount: 0 } }
    paid_count = 0
    outbox = Seq {}
  }

  fair action pay_submit(c: Claim) {       // 表に出る: 申請が支払いに進む
    requires design[c].st == DesignAutoApproved
    requires outbox.size() < OUTBOX_CAP
    design[c].st = DesignPaymentSubmitted
    paid_count = paid_count + 1
    outbox = outbox.push(c)
  }
  fair action pay_confirm(c: Claim) {      // 内部: 提出済みの支払いを確定する
    requires design[c].st == DesignPaymentSubmitted
    design[c].st = DesignPaid
  }
  fair action outbox_send() {             // 内部: 通知アウトボックスを掃き出す
    requires outbox.size() > 0
    outbox = outbox.pop()
  }
}
構文意味設計層での目安
enum DesignSt実装に近いサブステージ。DesignPaymentSubmitted は要件層に対応物がなく、あとで上位に畳まれます。
state { ... }内部の論理状態。要件が触れない台帳・キュー・キャッシュはここに置きます。
Seq<T, N>容量つきのキュー／アウトボックス。容量が有限なので検証器が総当たりできます。
fair action検証器が放置してはいけない操作。fair は、のちの preserve progress の土台になります。

まず設計を単体で証明します。fslc verify design.fsl --engine induction で、上につなぐ前に不変条件と部分操作の安全性を確認します。

3. 上位へ詳細化

refinement 対応表で、設計が要件を守ることを証明する

対応表は、どちらの spec も汚さない第3のファイルに書きます。すべての内部操作を「ひとつの要件操作」か「stutter（上では何も起きない）」のどちらかに対応づけ、すべての抽象状態を内部状態から取り出します。

refinement DesignRefinesRequirements {
  impl ExpenseDesign        // 詳細(設計)spec
  abs  ExpenseRequirements  // 要件spec

  map claim_stage[c: Claim] =
    if design[c].st == DesignDraft then Draft
    else if design[c].st == DesignAutoApproved then Approved
    else if design[c].st == DesignPaymentSubmitted then Paid
    else Paid
  map claim_amount[c: Claim] = design[c].amount

  action pay_submit(c)  -> pay(c)     // 表に出る操作 → 要件の操作
  action pay_confirm(c) -> stutter     // 内部: 要件層には見えない
  action outbox_send()  -> stutter
}

fslc refine design.fsl requirements.fsl mapping.fsl --depth 8 が有界の前方シミュレーションを回し、result: "refines" か、型のついた失敗を返します。

refine が見ること意味失敗が教えること
初期状態の対応対応づけた初期状態が、要件の init を満たす。abs_state_mismatch — 対応表か init のバグ。
遷移の対応すべての設計ステップが、有効な要件操作に対応し、更新も一致する。abs_requires_failed — 設計が上位の条件を迂回した。
stutterstutter にした操作が、本当に抽象側を変えない。stutter_changed_abs — 「内部だけ」という主張が嘘。
対応値の範囲対応づけた値が、抽象の型の範囲に収まる。map_out_of_bounds — 射影が抽象の範囲を超えた。

✅ 安全性は含まれる

refine が通れば、設計は要件の不変条件と条件（統制）を破れません。内部の詳細は、契約を破らない限り自由に足せます。

⚠ 応答性は——既定では含まれない

詳細化は stutter を許すため、要件が保証した応答を設計が落としても通ってしまいます。残したいなら次節の preserve progress で明示します。

内部詳細を隠す仕組みは2つ。 action ... -> stutter は内部だけの操作を隠し、map 式は内部フィールドを吸収します（例：map stock[i] = impl_stock[i] - reserved[i] で reserved を隠す）。maps auto は同名の状態・操作に恒等対応を生成し、明示の対応が優先されます。if … then … else … は対応表の式の中だけで使え、普通の spec では使えません。

4. 応答性を保つ

上位の leadsTo を、対応表ごしに引き継ぐ

要件が約束した応答がまだ重要なら、preserve progress ブロックを足します。名前のついた抽象 leadsTo を、設計の実行上で再検査します。公平性は設計自身の fair 操作から来ます。

refinement DesignKeepsLivenessProgress {
  impl DesignKeepsLiveness
  abs  ClaimPolicy

  map s = if d == DSubmitted then Submitted
          else if d == DApproved then Approved
          else if d == DRejected then Rejected
          else Paid

  action route()   -> stutter
  action approve() -> approve()
  action reject()  -> reject()
  action pay()     -> pay()

  preserve progress {
    respond EveryClaimDecided by approve, reject   // by の操作は impl の操作名。公平性はその fair から来る
  }
}
設計が…結果直し方
応答待ちのまま循環で空回りするprogress_lost（lasso 反例）進行させる操作に fair を足す／戻す。
応答する前に手詰まりになるprogress_lost（stall）応答への経路を有効にする。または下位に leadsTo … decreases … を書き --engine induction で検証する。

応答性は、頼んだところにしか来ません。 preserve progress は安全性の詳細化を変えません。そして公平性は自動継承されず、設計の fair action 宣言からのみ生まれます。

5. 合成

検証済みの部品を compose で組み立てる

大きな設計は部品から組みます。compose は検証済みの spec を名前空間つきでまとめ、同期操作で歩調をそろえ、単独で発火すべきでないステップを隠します。

compose OrderSystem {
  use ShoppingCart as cart from "cart_v1.fsl"
  use Payment      as pay  from "payment.fsl"

  state { orders_linked: Int }
  init  { orders_linked = 0 }

  fair action checkout_and_pay(u: cart.UserId, p: pay.PayId) =
      cart.checkout(u) || pay.capture(p) {     // 2つの部品が1ステップで進む
    requires pay.payments[p].st == Authorized
    orders_linked = orders_linked + 1
  }

  internal cart.checkout                       // 同期でだけ発火する
  internal pay.capture
}
構文意味注意
use A as ns from "..."部品を名前空間つきで取り込む。状態・操作は ns.x で参照します。
a.act(...) || b.act(...)同期操作：両方が一緒に進む。合わさったステップが requires／本体／ensures を共有します。
internal ns.actionその操作は単独で発火しなくなる。同期操作の一部としてのみ動きます。

同期では公平性は継承されません。 同期操作が公平なのは、そう宣言したときだけです。さもないと fair_not_inherited の警告が出ます。compose はそれ自体を fslc refine の impl 側にできるので、組み立て全体をひとつの要件契約に対して証明できます。

6. 受け渡し

設計層は、実装適合の判定基準になる

設計を証明しても、コードを証明したことにはなりません。両者をつなぎます。spec からテストを生成する、実ログを spec に再生する、実行時監視器を埋め込む。いずれも、実装状態を spec の論理形に射影する Adapter が要ります。

Monitor（監視器）

実装に fslc.runtime.Monitor を埋め込み、spec を破るステップを実行時に弾きます。

scenarios（シナリオ）

具体的なトレース（covers／受け入れの意図を含む）を生成し、実装が再現すべき手順にします。

testgen（テスト生成）

fslc testgen が、spec を判定基準にして実装をランダムウォークする pytest 雛形を出します。

replay（ログ再生）

fslc replay が有限のイベントログを spec に照合します。注意：leadsTo は有限ログからは判定できません。

アダプタの契約

生成されたテストは Adapter.observe() を呼びます。これは実装を読み、状態を spec の論理形——同じ変数、同じ範囲つき型——で返します。適合の質は、この射影の質と同じです。observe() が嘘をつけば、判定基準も嘘になります。

class ExpenseAdapter(Adapter):
    def observe(self) -> dict:
        # 実システムを spec の状態変数に射影する
        return {
            "design": {c.id: {"st": c.stage, "amount": c.amount} for c in self.repo.all()},
            "paid_count": self.ledger.count(),
            "outbox": [m.claim_id for m in self.queue.pending()],
        }

✅ 設計層が保証すること

設計自身の不変条件と部分操作の安全性（帰納法を使えば無限の深さで）。refine による要件層までの安全性の包含。preserve progress と fair を書いたところの応答性。そして実装適合は、testgen や replay を実コード・実ログにつないだところだけ。

⚠ 保証しないこと

要件そのものの正しさ（refine は abs が別途検証済みだと仮定する）。応答性の自動継承。雛形をつなぐまでの実コードの正しさ。そして核言語の外——確率、百分位、実時間ミリ秒、外部サービスの挙動。

7. チェックリスト

実装に渡す前に見ること

  1. すべての内部操作が対応づいているか

    各設計操作は、ひとつの要件操作か、根拠のある stutter のどちらかに対応します。対応漏れは type エラーです。

  2. すべての抽象変数が対応づいているか

    対応表は要件層の状態をすべて覆う必要があります。さもないと refine が対応漏れを報告します。

  3. 設計を単体で証明したか

    上につなぐ前に、verify と --engine induction で不変条件を再確認します。

  4. まだ重要な応答があるか

    あるなら preserve progress と 下位の fair 操作で覆います。安全性の詳細化だけでは保てません。

  5. 証明の種類を分けて報告したか

    設計の不変条件を証明、要件への詳細化を証明、そして実装適合は testgen か replay を実コード・実ログにつながない限り証明していないと分けます。

次は、文法を通して読む

3つの層がそろったら、文法・構文ページが具体的な書き方——型、操作、性質、詳細化、合成——をまとめてつなぎます。
