はじめる

5分で、反例を読むところまで。

インストールしてから、壊れた仕様を検証し、反例JSONを読み、直し、最後に「証明」まで進む一本道です。すべて実際にコピペして動かせます。

FSLをはじめて見る場合は、先に概念の入門ページを読むと直感がつかみやすくなります。

▽ スクロール

0. 導入

依存はlarkとz3-solverの2つだけ

C++コンパイラや別途のZ3インストールは不要です。Mac/Windows/Linuxとも pip install だけで揃います（Python 3.9+）。

git clone https://github.com/ymm-oss/fsl && cd fsl
python3 -m venv .venv && source .venv/bin/activate
pip install -e ".[dev]"   # lark, z3-solver, pytest + fslc を editable install

1. check

まず速いループ：構文と型だけを見る

fslc check はZ3を呼ばず、構文・名前・型だけを検査します。編集のたびに叩く最速のループです。

$ fslc check docs/intro/specs/order_buggy.fsl
{
  "fsl": "1.0",
  "result": "ok",
  "spec": "OrderWorkflow",
  "warnings": []
}

構文・型は正しい、という結果です。しかしこの仕様にはルール違反が仕込まれています——checkはそこまでは見ません。

2. verify — violated

次に総当たり：反例を機械に探させる

fslc verify は深さ8まで、あり得る全ての操作列をZ3で試します。ルール違反があれば、最短の反例を返します。

$ fslc verify docs/intro/specs/order_buggy.fsl --depth 8
{
  "fsl": "1.0",
  "result": "violated",
  "spec": "OrderWorkflow",
  "violation_kind": "invariant",
  "invariant": "ShipOnlyAfterApproval",
  "violated_at_step": 2,
  "last_action": { "name": "ship", "params": {} },
  "trace": [
    { "step": 0, "state": { "status": "Draft", "approved": false } },
    { "step": 1, "state": { "status": "Submitted", "approved": false },
      "action": { "name": "submit" },
      "changes": { "status": { "from": "Draft", "to": "Submitted" } } },
    { "step": 2, "state": { "status": "Shipped", "approved": false },
      "action": { "name": "ship" },
      "changes": { "status": { "from": "Submitted", "to": "Shipped" } } }
  ]
}

この反例の読み方：result: "violated" なので invariant と trace だけ見れば十分です。submit → ship の2手で、approved が false のまま出荷（ShipOnlyAfterApproval違反）に到達しています——承認を飛ばして出荷できてしまう、というバグです。

3. 直す

1行のガードを直す

ship アクションの requires が間違った状態を許していました。

  action ship() {
-   requires status == Submitted   // BUG: should require Approved
+   requires status == Approved    // FIX: ship only after approval
    status = Shipped
  }

4. verify — verified

直した仕様を、もう一度深さ8で検証する

$ fslc verify docs/intro/specs/order_fixed.fsl --depth 8
{
  "fsl": "1.0",
  "result": "verified",
  "spec": "OrderWorkflow",
  "checked_to_depth": 8,
  "completeness": "bounded",
  "invariants_checked": ["_bounds_status", "ShipOnlyAfterApproval"],
  "action_coverage": { "submit": true, "approve": true, "ship": true, "cancel": true },
  "warnings": [
    { "message": "deadlock reachable at step 3 (state: status=Shipped, approved=true)",
      "hint": "add an enabled action, declare intended stops in a terminal { } block, or use --deadlock=ignore if intentional" }
  ]
}

warningsも読む：result: "verified"になってもwarningsは空とは限りません。ここでのdeadlock警告は、Shippedが意図した終端状態であるという事実を検証器がまだ知らないだけです（terminal { }宣言で伝えられます）。バグではありません——ただし読まずに無視する癖はつけないでください。

verifiedは「深さ8まで反例が無かった」という有界の結果です。無限の深さで確かめるには、次のステップに進みます。

5. verify --engine induction

最後に証明：無限の深さへ

補助不変条件を1本足した order_proved.fsl は、k帰納法で無限の深さまで証明できます。

$ fslc verify docs/intro/specs/order_proved.fsl --engine induction
{
  "fsl": "1.0",
  "result": "proved",
  "spec": "OrderWorkflow",
  "engine": "induction",
  "completeness": "unbounded",
  "invariants_checked": ["_bounds_status", "ShipOnlyAfterApproval", "ApprovedImpliesFlag"],
  "warnings": []
}

completeness: "unbounded" — これでこの仕様は、どれだけ長い操作列を試してもShipOnlyAfterApprovalが破れないことが証明されています。仕組みと使い方で、BMCからprovedへ進む考え方を詳しく読めます。
