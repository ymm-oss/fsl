構造の観測

fslc analyze でレビューの入口を作る

fslc analyze は、FSL 仕様を「どの要件・操作・状態・性質がつながっているか」というレビュー用の地図にします。検証の前後で、影響範囲、根拠の薄い宣言、トレーサビリティの穴を短時間で見つけるためのコマンドです。

実装設計の詳細は DESIGN-analysis.md にあります。

何ができるか

仕様を読む前に、見るべき場所を絞る

レビューで時間がかかるのは、正誤判断そのものよりも「どこを読むべきか」「この要件は下位設計の何に対応するか」を探す時間です。analyze はその探索を機械的に先に済ませます。

影響範囲をつかむ

action_state_graph で、各操作が読む・書く状態変数を一覧化します。変更した操作がどの状態に効くかを、全文読解や grep だけに頼らず確認できます。

要件とのつながりを追う

requirement_property_graph と traceability_graph で、要件ID、性質、シナリオ、下位設計の操作の対応を見ます。上位要求が設計側でどこに落ちているかを確認できます。

レビュー候補を出す

--profile ai-review は、根拠の薄い性質、guard のない更新操作、一度も書かれない状態などを「要確認」として出します。これは違反判定ではなく、レビューの優先順位付けです。

共有できる形にする

JSON、DOT、Mermaid、JSON Schema で出力できます。PR説明、CI、エディタ診断、外部可視化ツールに同じ観点を渡せます。

効果

レビュー時間が減る理由

手作業で時間がかかることanalyze で先に出せるもの効果
変更した操作がどの状態に影響するか探す操作と状態の read/write グラフ影響範囲の探索を短縮し、読む対象を絞れる。
要件IDが実際の性質や設計操作に結びついているか追う要件・性質・シナリオ・refinement のグラフトレーサビリティ確認を再現可能にできる。
全宣言を均等に読んで怪しい箇所を探すai-review findings最初に見るべき候補を並べ、レビューの順序を決めやすくする。
PRコメント用に根拠を説明するwitness、candidate repair、do-not-assume 付き JSON指摘の根拠と注意点をそのまま共有できる。

analyze はレビューを置き換えるものではありません。仕様判断は人間が行います。効果が出るのは、レビュー前半の「探索」と「根拠集め」を短くできるからです。

境界

analyze は verify ではない

fslc analyze は、検証済みの spec モデルから決定的な構造 JSON を作ります。Z3、到達可能性探索、ログ replay、詳細化証明は実行しません。

レビューに使う

孤立した要件、根拠の薄い性質、グラフ上の循環、トレーサビリティの穴を深い検証の前に見つけます。

証明とは分ける

形式的な証拠や実装適合が必要なときは verify、refine、replay を使います。

status を正直に読む

構造所見は formal_status: "not_a_violation" を持ちます。循環があるだけでバグとは言いません。

コマンド

目的別に出力する

fslc analyze spec.fsl --projection tsg --format json
fslc analyze spec.fsl --projection action_state_graph --format json
fslc analyze spec.fsl --projection requirement_property_graph --format json
fslc analyze spec.fsl --projection property_state_graph --format json
fslc analyze specs/ examples/e2e/ --profile ai-review --format json
fslc analyze specs/cart_refines.fsl --projection refinement_graph --format json
fslc analyze fsl-project.toml --projection traceability_graph --format json
fslc analyze spec.fsl --projection action_state_graph --format dot
fslc analyze spec.fsl --projection action_state_graph --format mermaid
出力見えるもの主な使いどころ
tsgspec、状態、操作、ガード、効果、性質、要件、シナリオを含む Typed Semantic Graph 全体。独自ツールや下流の投影で使う元データを得る。
action_state_graph操作と、その操作が読む・書く状態変数のつながり。変更の影響範囲を確認する。
requirement_property_graph要件IDと、それが covers する操作、性質、シナリオ、KPI、control のつながり。要件の根拠や孤立を確認する。
property_state_graphユーザー定義の性質と、それが読む・検査する状態変数のつながり。性質が何を監視しているか確認する。
refinement_graph単体の mapping file に含まれる impl/abs 名、state map、action map、stutter、preserve-progress 宣言。詳細化の対応関係をレビューする。
traceability_graphproject manifest から business、requirements、design、refinement mapping を横断した構造。上位要件が下位設計へ落ちているか確認する。
--profile ai-review構造上のレビュー候補と、その witness、修正候補、早合点してはいけないこと。レビューの初動で見る順序を決める。

AIレビュー

最初の所見

disconnected_requirement

要件が宣言されているのに、操作、性質、シナリオ、KPI、control などの有用な構造アンカーにつながっていません。

unanchored_property

ユーザー定義の性質に、要件タグ、シナリオ・ガバナンス上の根拠、または action-state の根拠がありません。

progressless_cycle

要件またはシナリオにつながる複数操作の循環に、leadsTo、fair action、terminal などの進捗ストーリーが見えません。

unguarded_action

状態を書き換える操作に requires がありません。常に実行可能でよい操作なのか、guard を書き忘れていないかを確認します。

unwritten_state

初期化され、読まれている状態変数を、どの操作も書き換えていません。固定値なら const 化、そうでなければ更新操作の抜けを疑います。

cycle 所見は、名前の英単語を見ていません。要件タグやシナリオなどの構造アンカーに基づきます。日本語の要件文は保持されますが、fslc core は意味解釈しません。

スキーマ

出力の種類

出力schema version用途
Typed Semantic Graphtsg.v0下流の投影やツールが使う安定したグラフデータ。Schema は schemas/fslc/analysis/tsg.v0.schema.json。
Graph projectionsanalysis-graph.v0components、SCC、cycles、degree を含むレビュー用グラフ要約。DOT/Mermaid はこのデータの表示形式です。

限界

できないことも明確に分ける

自然言語の推論はしない

要件文はグラフに保持しますが、その意味を core analyzer が推論することはありません。

証明の証拠ではない

所見、project traceability、refinement graph は構造だけを示します。反例、到達可能性、詳細化の成否が必要な場合は verify、refine、chain を使います。

自動修正しない

candidate repair は選択肢です。guard を足す、const にする、要件タグを追加する、といった判断は仕様の意図を確認してから行います。

editor signals は opt-in

FSLC_LSP_ANALYSIS_DIAGNOSTICS=1 を設定すると、fslc-lsp が analysis findings を Information 診断として出します。
