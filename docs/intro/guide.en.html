Verification engines

Two stages: brute force first, then proof

The verifier fslc has two modes. Each is stronger than the last.

① BMC (bounded model checking)

fslc verify --depth N

Brute-forces every path within N steps from the initial state. A counterexample means a definite bug. Otherwise: "no counterexample up to depth N (verified)". Fast, and the counterexample is concrete.

② k-induction (infinite depth)

fslc verify --engine induction

Shows that from any state satisfying the rule, one step keeps the rule. Succeed and it's proved at infinite depth. Stricter than BMC.

First, BMC:

$ fslc verify order_fixed.fsl --depth 8
{ "result": "verified", "spec": "OrderWorkflow" }   // no counterexample up to depth 8

No counterexample to depth 8. But can you claim it's safe at any depth? That's where induction comes in.

Run induction

Induction asks: "what if we started from this state?"

Induction starts from every state that satisfies the rule, reachable or not. So it sometimes raises a state that never actually occurs and says "from here, one step breaks it." That's a counterexample to induction (CTI). Try it with the buttons below.

🔎 fslc verify --engine induction 
// add this one line and it becomes proved (auxiliary invariant)
invariant ApprovedImpliesFlag { status == Approved => approved }

Also based on real fslc output (unknown_cti → add auxiliary invariant → proved).

Reading a CTI

A CTI is not necessarily a real bug. It means one of two things:
(a) The invariant is too weak — a state that never actually occurs isn't ruled out by the rule → teach it "Approved ⇒ approved" with an auxiliary invariant and it's proved.
(b) A real, reachable bug — you can actually get to that state → fix the spec or the implementation.
Telling the two apart is the human's job.

The vertical thread

Three dialects, joined by refinement

Write the same domain at different granularities and connect them with refinement. fslc refine mechanically checks that the upper layer's intent is preserved below, and requirement IDs propagate into every layer's diagnostics.

consulting (business)

Rules, controls, KPIs, goals. Check business-level contradictions, forbidden states, goal reachability.

↓ refinement

requirements

Requirement IDs, acceptance criteria, forbidden flows, NFRs (permissions / audit / SLA…). Trace counterexamples back to the request.

↓ refinement

design

State machines, data, async, composition. invariant / reachable / leadsTo / mappings check the requirements are met.

↓ scenarios / testgen / replay

Implementation / QA

Generate scenarios and pytest scaffolds. Check that implementation logs conform.

fslc refine impl.fsl abs.fsl mapping.fsl checks one refinement step; fslc chain fsl-project.toml verifies the whole chain at once.

Run this in CI, not just once before shipping: every commit re-verifies the whole corpus, a change at one layer becomes a mechanical check of the layers below it, and drift between spec and implementation gets caught immediately instead of resurfacing as an incident. The same corpus is also what a new engineer — or an AI agent picking up the code — reads first to understand how the system is supposed to behave.

The toolbox

fslc command cheat sheet

All output is machine-readable JSON (for AI to read and fix). Exit codes: 0 = verified/proved/refines…, 1 = violated/counterexample, 2 = spec error.

Write & verify

check — syntax/types only (fast loop)
verify --depth N — BMC, shortest counterexample
verify --engine induction — prove at infinite depth

Measure spec quality

--vacuity error — detect vacuous properties
--strict-tags — lint untagged/unreferenced
mutate — measure constraint strength
explain — bounds & counterfactuals

Connect to implementation

scenarios — integration-test scaffold JSON
testgen -o test.py — pytest conformance scaffold
replay --trace log.json — log conformance
typestate --ts — state machine → types

Connect the layers

refine impl abs map — check refinement
chain project.toml — business→impl in one go
verify of a composed spec also works

Run it for real

Run the write → verify → repair loop

FSL doesn't assume humans memorize the grammar. AI drafts, fslc returns counterexamples, AI fixes. You own intent and the final call.

01

Give intent

State what must hold — "never ship unapproved" — in words.

02

AI drafts

The skill supplies grammar and the repair protocol; it generates the .fsl.

03

fslc counterexamples

Contradictions, loopholes, unreachability as JSON.

04

AI repairs

Reads the counterexample; repeats until nothing breaks.

What the AI reads is JSON like this (a counterexample trace):

{
  "result": "violated",
  "invariant": "ShipOnlyAfterApproval",
  "violated_at_step": 2,
  "last_action": { "name": "ship" },
  "trace": [ /* Draft → submit → Submitted → ship → Shipped */ ]
}

Setup is three commands (details in the README):

# 1. Install (gives you fslc and the skills)
$ bash install.sh

# 2. Ask the AI (via the skill)
"Use $fsl-requirements to write requirements for 'no shipping before approval;
 refunds never run twice', verify it, and repair until no counterexamples remain."

# 3. If needed, also generate test scaffolds and implementation hookup
$ fslc testgen spec.fsl -o tests/test_conformance.py

Adoption

Where do you start?

Start high-risk-first — that's a sequencing choice, not the finish line. The spec corpus is your project's documentation, not a scarce resource to spend only where failure hurts, so the target end state is broad coverage across business, requirements, and design.

🎯 High-risk core

Close to traditional formal methods.

Payments/refunds, applications/approvals, reservations/inventory allocation, permissions/audit, queues/async, SLA/timeouts. Start here — mistakes hurt most.

🧩 Small state management

Worth it because AI drafts the spec.

Screen transitions, back/cancel, confirmation modals, unsaved/saving, wizards, double-submit prevention, retry/pending/failed, role-based availability. For pre-implementation loophole checks.

The criterion isn't "how many states" but "whether some order of operations or flag combination can enter an unintended state." Even 3 states and 2 flags are worth FSL-ing if back/cancel/retry/permission branching is involved. Start with a small PoC you can finish in two weeks, then keep expanding coverage — the corpus stays the project's living documentation, not a one-off exercise.

Give it a try

Have AI write one small spec and run fslc verify on it.
