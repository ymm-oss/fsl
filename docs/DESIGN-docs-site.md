# FSL Docs Site — Information Architecture, Navigation Chrome, and Reference-Page Template

This document specifies the next iteration of the `docs/intro/` manual site's information
architecture (IA), navigation chrome, and the template for the two new generated reference
pages (`language.html`, `cli.html`). It is a **decisions artifact** — it fixes the shape of
the site, not its prose. It does not author manual content, does not implement code, and does
not touch `docs/intro/assets/site.css` / `site.js` or any existing `docs/intro/*.html` file.
A companion static wireframe (`docs-site-wireframe.html`, see §7) demonstrates the three page
states this document describes; consult it alongside this doc, not instead of it.

Status: proposed. Confidence on the chapter→category mapping (§2) is **medium** — flag before
downstream content work reassigns anything (see caveat there).

## 1. Why this shape (six decisions)

The current site already has fixed chrome (`topbar` + `docs-sidebar`, generated by
`initNav()` in `site.js`) and 12 flat chapter pages plus a hand-authored `syntax.html` /
`analysis.html`. Two problems motivate this pass: (a) 12 flat chapters with no grouping is
already at the edge of scanability and two more reference pages (`language`, `cli`) would push
it over; (b) `LANGUAGE.md` and the CLI surface (`cli.py`) have no dedicated site page at all —
readers and AI-agent HTML fetchers currently have no single-page, disclosure-friendly view of
either.

### D1 — Navigation model: always-expanded 2-level categorized sidebar

Group the (growing) chapter list under 4 fixed category headers, all chapters visible under
their header without interaction — no accordion, no dual-mode toggle.

- **Rejected: accordion.** An accordion hides sibling categories by default, which breaks the
  single behavior the current sidebar is good at: scan every chapter, in order, in one glance.
  Adding 2 reference pages does not yet justify trading that away.
- **Rejected: dual-mode toggle** (e.g. "flat list ⇄ grouped list"). Once everything is visible
  at once under D1, a mode switch is a control with no second mode worth reaching — it is
  complexity added for a choice nobody needs to make.
- Net effect: the sidebar gains 4 group labels (implemented as real `<h2>` elements, not
  visual-only text — see §4) that the chapter-link markup and its current/hover states are
  otherwise untouched by.
- **Accepted trade-off, stated explicitly (revision after critique):** on ordinary laptop
  viewport heights (~768-900px), 4 group labels + up to 14 chapter rows can exceed the
  sidebar's available height, triggering the sidebar's own `overflow:auto` (already present
  today for the flat 12-item list). "Scan in one glance" degrades to "scan with one small
  internal scroll" on such hardware — this is judged an acceptable, pre-existing-pattern
  trade-off rather than a reason to reintroduce an accordion; if it proves too cramped in
  practice, tightening `.docs-category-label`/`.chapter-link` vertical rhythm is a cheap,
  purely visual follow-up that does not change this decision.

### D2 — Orientation affordance: same sidebar everywhere + a linking breadcrumb

The categorized sidebar from D1 renders on **every** page — hub pages and chapter pages alike
— at every breakpoint: the existing ≥1180px fixed left rail, and the existing <1180px wrapping
grid (now with category sub-headers inside the grid instead of a flat list). This keeps "where
am I in the whole manual" answerable from anywhere, including the 4 new hub pages.

In addition, every chapter/reference article gets a lightweight breadcrumb at the top of the
`<article>`/first content `<section>`: `Category Name / Chapter Name`, where the category
segment is a real `<a>` back to that category's hub page.

- The breadcrumb's category link **is** the "back to category" affordance on the 12 existing
  chapter pages. A separate sticky "back to category" button was considered and rejected there:
  the topbar is already `position: sticky`, and a second sticky element competing for the same
  viewport edge is the kind of chrome-stacking that degrades mobile more than it helps
  orientation.
- The breadcrumb is scoped to chapter/reference pages, not hub pages (a hub page has no
  "chapter" segment to show).
- **Revision after critique — narrow exception for the 2 generated reference pages.** The
  sidebar's `position: sticky` rail (the thing that makes "return path always reachable" true
  on the 12 chapter pages) is itself `position: static` below the existing 1180px breakpoint,
  per the current site's own CSS. Below that breakpoint, on an ordinary chapter page this is a
  minor cost (chapters are a few screens long). But `language.html`/`cli.html` are, by D3's own
  design, single long pages that grow via nesting rather than pagination — so a reader who has
  scrolled deep into the CLI tree on a narrow viewport has no sticky sidebar *and* no sticky
  back-button, only the browser's native Back. This is fixed narrowly, not by reversing D2's
  general rule: `language.html`/`cli.html` alone additionally get a small `position: fixed`
  bottom-right "↑ Top" link (a plain in-page anchor, works with JS disabled, does not collide
  with the top-anchored sticky topbar because it sits at the opposite corner). The 12 existing
  chapter pages do not need this and do not get it — if a future chapter page grows long enough
  to need it, add the same control there rather than sitewide by default.

### D3 — Reference-page template: native `<details>`/`<summary>` disclosure tree

Both `language.html` and `cli.html` render their content as a nested tree of native HTML
disclosure elements, not a custom JS accordion/tree widget.

- **Why native elements, not JS:** content inside a closed `<details>` is still in the DOM and
  the accessibility tree, and is served as plain markup to any client that doesn't run
  JavaScript — including most AI-agent HTML fetchers, which is a primary consumer for `fslc`'s
  own docs (`CLAUDE.md`: "the JSON envelope … is a stable contract"; the same "don't hide
  the machine-readable surface behind interaction" principle applies to the human/agent-shared
  HTML surface). A custom widget that lazily injects children on click would hide content from
  exactly that audience.
- One **"Expand all / Collapse all"** control is added per page as progressive enhancement:
  plain JS that sets/clears the `open` attribute on every `<details>` in the tree. With JS
  disabled, the control simply isn't wired up and the page still works — the author decides
  per-node default `open` state (see §4) and everything is still readable by scrolling/searching.
- **Nesting depth is fixed per page, not generalized:** `language.html` is 1 level deep
  (top-level = `LANGUAGE.md`'s own section headings; each opens to that section's constructs).
  `cli.html` is 2 levels deep (top-level = subcommand groups — `verify`, `check`, `refine`,
  `chain`, `scenarios`, `replay`, `testgen`, `mutate`, `explain`, `html`, `typestate`,
  `analyze`, plus the dialect-namespaced groups `db`, `ai`, `domain`, and the
  compatibility/"compat-style" group; nested level = the sub-subcommands under the dialect
  groups, e.g. `db observe`, `db import`, `ai replay`, `domain scaffold`). `cli.html` stays a
  **single page** — disclosure via nesting is the answer to CLI surface growth, not a page
  split, so there is exactly one URL to link to for "the whole CLI."

### D4 — Provenance signal: a neutral badge, not a warning

Both generated pages carry a small persistent badge near the `<h1>` reading e.g. "Generated
from LANGUAGE.md" / "Generated from cli.py — `<version/date placeholder>`".

- Styled with the new `--ref`/`--ref-soft` pair (§5), deliberately **not** `--danger`/`--warn`.
  This is a provenance disclosure ("this text has a machine source of truth, don't hand-edit
  it"), not a caution about correctness — using the warning palette would train readers to
  associate it with something being wrong, which is the opposite message.
- **Out of scope for this document:** CI-enforced staleness detection (failing a build when
  generated content drifts from `LANGUAGE.md`/`cli.py`) is recommended for the separate
  generation-pipeline task (§8). This document only fixes what the badge looks like and where
  it sits, not how/when it is produced or verified fresh.

### D5 — Visual system: extend, don't replace, plus one new token pair

All new chrome is built from the existing token system in
`docs/intro/assets/site.css` (`--brand`/`--surface`/`--line`/`--shadow`/`--radius`/etc.), the
existing `.card`/`.panel`/`.matrix`/`.badge`/`.pill` components, and the existing light+dark
split via `prefers-color-scheme`. Exactly one new semantic token pair is added: `--ref` /
`--ref-soft`, for the "generated reference" zone only (§5). It is applied to exactly **two**
places — the provenance badge and the tree-node `<summary>` accent/hover — and nowhere else;
the other three existing template shapes (narrative/interactive like `concept.html`,
dense-reference like `syntax.html`, mid-length layer-explainer like `business-layer.html`) keep
using `--brand` exactly as today.

**Revision after critique — no bordered header-band, no mono H1/H2.** An earlier pass of this
design wrapped the badge + `<h1>` in a tinted, bordered "callout-shaped" band and set the
`<h1>`/`<h2>` themselves in `var(--mono)`. Both are dropped: (a) this site already uses the
identical pill-badge-plus-tinted/bordered-box combination, sitewide, to render live
verify-verdict state (`violated`/`cti`/`proved` on `concept.html`/`guide.html`/etc.) and to mark
`.callout.warn` cautions — stacking that same combination around a provenance note risks a
reader (who has just seen a colored verdict pill elsewhere in the manual) briefly misreading it
as a status about *this page*, the opposite of D4's "informational, not a warning" intent; (b)
`var(--mono)` is this site's sitewide, exclusive signifier for literal code/syntax
(`code`/`pre` use nothing else) — reusing it for heading tone on `language.html`/`cli.html`
repurposes an established "this is literal code" cue as a "this feels technical" cue, which can
read as the H1 itself being a command token. The replacement: a plain badge (no wrapping band)
directly under the breadcrumb, a small **mono-set kicker label** ("Generated Reference") above
the ordinary sans-serif `<h1>` — carrying the same "technical/tool" signal at kicker scale,
where it cannot be mistaken for a code span or a literal heading-as-command — and the lead
paragraph unchanged. See §4/§5 for the corrected component and token scope.

### D6 — Data model: `categoryId` on `CHAPTERS` + a new, non-growing `CATEGORIES`

`site.js`'s `CHAPTERS` array (currently 12 entries) gains one new field per entry,
`categoryId`, plus 2 new entries for `language` and `cli`. A new `CATEGORIES` array (exactly 4
entries, one per fixed Diátaxis-style category, never grows) is added alongside it and is the
only place category label/order/description lives — `CHAPTERS[i].categoryId` is a foreign key
into it, so there is exactly one nav-mapping source per concern (chapter metadata stays in
`CHAPTERS`; category metadata stays in `CATEGORIES`) rather than two overlapping ones. The 4
category **hub pages** are a separate, fixed set of 8 files (4 categories × ja/en) that are not
`CHAPTERS` entries — see §3 for the exact shape and §8 for why hub-page routing is out of
scope here.

**Revision after critique — scope of the "one-line edit" claim.** Moving a chapter to a
different category is a one-line `categoryId` edit *for membership*. It is **not** a one-line
edit for that chapter's *position within* its new category's rendered list (sidebar group and
hub-page card grid both render in `CHAPTERS` array order, filtered by `categoryId` — there is no
separate per-category order field). §2's own medium-confidence caveat already names a plausible
case where this matters (e.g. moving `guide` into `get-started`): the moved chapter would land
wherever it happens to sit in the global array relative to its new siblings, not necessarily
where the maintainer intends, requiring a second, manual array-reordering edit to place it
correctly. This is a real but narrow gap — it does not create a second nav-mapping source (the
hard constraint), it only means "one-line" undersells what re-categorizing sometimes requires.
Recorded here rather than fixed by adding a new ordering field, since no reassignment has
happened yet — add a per-chapter `order`-within-category field only if/when the maintainer
review in §2 actually reorders something.

### D7 — `docs/LANGUAGE.ja.md`: a second canonical source, section-aligned to `docs/LANGUAGE.md`

Supersedes the "no translation, no second source to keep in sync" framing in D4/the earlier
`tools/build_site_reference.py` docstring (itself attributed there to "the fable decision on the
canonical language problem"). Under that policy `docs/intro/language.ja.html`'s body reused the
English `docs/LANGUAGE.md` verbatim — only the lead paragraph and each section's one-line
`SECTION_BLURBS` description were Japanese. A user explicitly asked for the Japanese page to be
readable in Japanese, accepting the resulting maintenance cost.

- **What changed:** `docs/LANGUAGE.ja.md` is a new, hand(-and-agent)-maintained Japanese
  translation of `docs/LANGUAGE.md`, kept **section-aligned 1:1**: the same count and order of
  `## ` headings as the English file. FSL keywords, type names, `fslc` subcommands/flags, result/
  diagnostic identifiers (`verified`, `unknown_cti`, `FSL-ANNOTATION-TARGET`, ...), JSON
  keys/values, and every fenced code block are left untranslated/byte-identical to the English
  source — only heading text and prose are translated.
- **How the site stays honest about drift:** `render_language_tree("ja")` in
  `tools/build_site_reference.py` reads `docs/LANGUAGE.ja.md` and compares its section count
  against `docs/LANGUAGE.md`'s. A mismatch raises `SystemExit` at generation time rather than
  silently shipping a page with a missing or misplaced section — the same "fail loudly on drift"
  discipline D4/the original docstring already used for `SECTION_BLURBS` coverage, now also
  covering section-count parity between the two language files.
- **Anchors and blurbs stay keyed off English.** Both the `<details id="...">` slug and the
  `SECTION_BLURBS` lookup are computed from the **English** heading (via `zip(en_sections,
  render_sections)`), never the Japanese one. This keeps `#7-the-verifier-fslc`-style cross-page
  links stable regardless of which language rendered the page, and means `SECTION_BLURBS` did not
  need a second, Japanese-heading-keyed dictionary.
- **Accepted cost, not hidden:** every future `docs/LANGUAGE.md` change needs a corresponding
  `docs/LANGUAGE.ja.md` edit (human- or agent-reviewed) to stay section-aligned; this is exactly
  the "second source to keep in sync" cost D4 originally chose to avoid. It is now accepted in
  exchange for a Japanese reference readers can actually read in Japanese. `AGENTS.md`'s
  "a language feature moves all of its files together" rule lists `docs/LANGUAGE.ja.md` alongside
  `docs/LANGUAGE.md` for this reason.
- **Out of scope for this document:** any machine-translation or CI-enforced
  translation-freshness pipeline (e.g. auto-flagging which English section changed since the last
  Japanese sync) is a separate future task, not implemented here. Today the only enforcement is
  the section-count parity check above — it catches an added/removed/reordered section, not a
  paragraph that changed English wording without a matching Japanese edit.

## 3. Data model — exact shapes

### 3.1 `CATEGORIES` (new, 4 entries, fixed)

```js
// site.js — new top-level const, alongside CHAPTERS.
// Shape: { id, order, en: [label, description], ja: [label, description] }
const CATEGORIES = [
  { id: "get-started",           order: 1, en: ["Get Started", "Orientation and fit before you write anything"],
                                            ja: ["はじめる", "書く前に方向感と適合を確認する"] },
  { id: "guides",                order: 2, en: ["Guides", "Task-oriented walkthroughs, one per layer"],
                                            ja: ["ガイド", "各層のタスク志向の手順"] },
  { id: "reference",             order: 3, en: ["Reference", "Lookup-oriented syntax and tool surface"],
                                            ja: ["リファレンス", "検索用途の構文・ツール仕様"] },
  { id: "examples-background",   order: 4, en: ["Examples & Background", "Worked mechanisms and dialect deep-dives"],
                                            ja: ["実例と背景", "仕組みの実例とダイアレクト詳細"] },
];
```

`order` drives both the sidebar's category sequence and the hub-page index order; it is data,
not implied by array position, so re-ordering categories later is a one-line change.

### 3.2 `CHAPTERS` (existing array, +`categoryId` field, +2 entries)

```js
// site.js — CHAPTERS gains one field per existing entry (categoryId) and two new
// entries (language, cli). Existing per-entry shape is unchanged otherwise:
// { id, categoryId, en: [shortLabel, sidebarTitle, sidebarDescription], ja: [...] }
const CHAPTERS = [
  { id: "concept",            categoryId: "get-started",           en: [...], ja: [...] },
  { id: "when-to-use",        categoryId: "get-started",           en: [...], ja: [...] },
  { id: "guide",              categoryId: "guides",                en: [...], ja: [...] },
  { id: "business-layer",    categoryId: "guides",                en: [...], ja: [...] },
  { id: "requirements-layer", categoryId: "guides",                en: [...], ja: [...] },
  { id: "design-layer",       categoryId: "guides",                en: [...], ja: [...] },
  { id: "syntax",             categoryId: "reference",             en: [...], ja: [...] },
  { id: "analysis",           categoryId: "reference",             en: [...], ja: [...] },
  { id: "mechanism",          categoryId: "examples-background",   en: [...], ja: [...] },
  { id: "domain",             categoryId: "examples-background",   en: [...], ja: [...] },
  { id: "db",                 categoryId: "examples-background",   en: [...], ja: [...] },
  { id: "ai",                 categoryId: "examples-background",   en: [...], ja: [...] },

  // NEW — generated reference pages (D3/D4); both land in "reference".
  { id: "language", categoryId: "reference",
    en: ["Language", "Language Reference", "Generated map of LANGUAGE.md"],
    ja: ["言語仕様", "言語リファレンス", "LANGUAGE.mdから生成"] },
  { id: "cli", categoryId: "reference",
    en: ["CLI", "CLI Reference", "Generated map of every subcommand"],
    ja: ["CLI", "CLIリファレンス", "全サブコマンドを生成"] },
];
```

The 10 pre-existing entries keep their current `en`/`ja` tuples verbatim (elided above as
`[...]`) — this is a pure additive field, not a rewrite of chapter metadata (that rewrite is
out of scope; see §8).

### 3.3 Category hub pages (NOT `CHAPTERS` entries)

4 categories × {`ja`,`en`} = **8 static files**, named after the category `id`
(`get-started.en.html`, `get-started.ja.html`, `guides.en.html`, …, matching the existing
`<stem>.<lang>.html` convention). Each hub page's content is entirely derived at author-time
from two lookups, not a new data source:

- Its own header/description: `CATEGORIES.find(c => c.id === thisPage)`.
- Its chapter list: `CHAPTERS.filter(c => c.categoryId === thisPage)`, rendered with the
  existing `.chapter-card` component (already used on `index.html`), in `CHAPTERS` array order.

Because hub pages read `CATEGORIES` + a filter over `CHAPTERS` rather than owning their own
list, there is no second per-chapter nav-mapping source to keep in sync — reordering or
re-categorizing a chapter is exactly the one-line `categoryId` edit described in §1/D6.

## 4. Component descriptions (plain language)

**Categorized sidebar.** Same `<aside class="docs-sidebar" data-nav>` mount point as today.
Instead of one flat `<nav class="docs-chapters">`, it renders 4 stacked groups, each a **real
`<h2>` heading** (visually styled like the existing `.docs-sidebar-title span` kicker treatment,
but a genuine heading, not a styled `<p>`) followed by that category's own
`<nav class="docs-chapters" aria-labelledby="{categoryId}-label">`, with the `<h2>` carrying
`id="{categoryId}-label"`. The chapter belonging to the current page keeps its existing
`aria-current="page"` + `--brand`-tinted treatment; no new state is introduced for "current
category" beyond the current chapter already being visibly inside its group. ≥1180px: unchanged
fixed left rail, just with the added group headings. <1180px: unchanged wrapping grid, with each
category's chips now under its own sub-heading instead of one undivided grid.

**Revision after critique — accessibility.** The wireframe's first pass used a plain
`<p class="docs-category-label">` for each group and left the resulting 4 `<nav>` elements
unlabeled. On a screen reader, that renders as 4 identical, anonymous "navigation" landmarks
with no heading-rotor entry for "Reference"/"Guides"/etc. — the exact grouping structure D1
exists to introduce would be invisible to a non-sighted reader relying on the landmark/heading
rotor rather than linear reading. The `<h2>` + `aria-labelledby` pairing above is the fix: each
category becomes a distinct, named landmark (`aria-labelledby` points AT-agents to its heading
text as the nav's name), consistent with D3's "content must be reachable by AT/non-JS clients"
standard for the reference-tree template — the same standard now applies to the sidebar
grouping that motivated this whole redesign.

**Breadcrumb.** A one-line `<nav aria-label="Breadcrumb">` immediately inside the first
`<section>`/above the page's own kicker, rendered only on chapter/reference pages (not hubs):
`<a href="{categoryHubHref}">{Category label}</a><span> / </span><span
aria-current="page">{Chapter sidebarTitle}</span>`. Small, muted, `--muted` text with a
`--brand`-on-hover link for the category segment — visually a quieter sibling of the existing
`.kicker`, not a new visual register.

**Disclosure tree (`language.html`/`cli.html`).** A plain nested list of native
`<details><summary>…</summary>…</details>` elements (§3 in D3 above covers depth per page).
Each `<summary>` gets the `--ref` accent (text tint + hover background), so the whole tree
reads as "you are in the generated-reference zone" without a color that also means
verified/violated/CTI elsewhere on the site. One `<div class="tree-controls">` above the tree
holds "Expand all" / "Collapse all" buttons (styled with the existing `.btn`/`button.op`
classes); their only job is toggling `open` on every `<details>` — no other state.

**Provenance badge.** A small **plain** pill directly under the breadcrumb (no wrapping band,
box, or border around it or the `<h1>` — see the D5 revision above for why), visually built on
the existing `.badge` component (`display:inline-flex; border-radius: 999px`) but with
`--ref`/`--ref-soft` instead of the existing status colors, holding a fixed short string plus a
placeholder for the generation source (`Generated from LANGUAGE.md` / `Generated from cli.py —
v0.x / 2026-07-09`, the version/date being filled in by whatever the generation pipeline task
decides — this doc only fixes that a placeholder slot exists).

**Mono kicker label.** A small `.kicker`-style line ("Generated Reference"), set in `var(--mono)`,
placed above the ordinary sans-serif `<h1>` on `language.html`/`cli.html` only. This — not the
heading font — is where the "technical/tool" signal lives (see D5 revision); the `<h1>` itself
stays sans-serif like every other page's heading.

**Back-to-top control (2 reference pages only).** A small `position: fixed` pill anchored to the
bottom-right corner of the viewport, a plain `<a href="#{page}-top">↑ Top</a>` — no JS required,
native in-page anchor scrolling. Present only on `language.html`/`cli.html` (see the D2 revision
above for why only these two pages need it). Does not appear on the 12 existing chapter pages.

## 5. New CSS tokens (add to `docs/intro/assets/site.css`)

Add to the existing `:root` block (after the current `--warn-soft` line, before `--shadow`):

```css
  --ref: #0f7d82;        /* generated-reference zone (language.html / cli.html only) */
  --ref-soft: #e4f4f4;
```

Add to the existing `@media (prefers-color-scheme: dark) { :root { … } }` override block
(after the current `--warn`/`--warn-soft` dark values):

```css
    --ref: #4dd2d6; --ref-soft: #133335;
```

Rationale for the hue: teal/graphite, deliberately far from the 4 reserved semantic colors —
`--brand` indigo #4f46e5 (states/structure), `--danger` red (violation/forbidden), `--ok` green
(verified/proved), `--warn` amber (CTI/counterexample) — so it cannot be misread as any of
those meanings; it should read as "instrumentation/tooling," which matches D4's "neutral
provenance, not a warning" intent. Contrast was checked directly against the stated hex pair
(light `#0f7d82` text on `#e4f4f4` background, dark `#4dd2d6` on `#133335`) and is at least as
strong as the existing `--ok`/`--warn`/`--danger` soft-badge pairs it sits alongside. `--ref` is
applied only to the 2 places named in the D5 revision above (the plain badge, and the
tree-`<summary>` accent/hover); it must not appear in `.chapter-link`, `.docs-sidebar`, a
bordered/tinted band around any heading, or any of the other 3 template shapes.

## 6. Out of scope for this document

This document fixes IA/nav/template/visual-system decisions only. The following are downstream
implementation tasks, not covered here:

- **The generation pipeline** that produces `language.html`/`cli.html` content from
  `LANGUAGE.md`/`cli.py`, and any CI-enforced staleness check that fails a build when generated
  content drifts from its source (flagged as a recommendation in D4, not specified here).
- **Creating the 2 new generated pages** (`language.en/ja.html`, `cli.en/ja.html`) and the
  **8 new category hub pages** (`get-started.*`, `guides.*`, `reference.*`,
  `examples-background.*`, ja/en) as actual files.
- **Rewriting the 12 existing chapter pages** to add the breadcrumb markup and pick up the
  now-categorized sidebar; none of `docs/intro/*.html` is edited by this document.
- **Editing `docs/intro/assets/site.css` / `site.js` themselves** — §5/§3 specify the exact
  blocks to add, but this document does not perform that edit; a downstream task applies it
  (and updates `initNav()` to group by `categoryId` and to render hub-page nav).
- **Reconciling the chapter→category mapping** in §2 with the maintainer (medium-confidence
  caveat, restated from §2).
- **`docs/README.md` / `docs/LANGUAGE.md` cross-links** to the new pages once they exist.
