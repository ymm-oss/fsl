# FSL — `--strict-tags` lint (traceability reconciliation) implementation design

Motivation: issue #5 (category 7 of roadmap #1). Declaration tags `"REQ-n: original text"` are
implemented but optional. If the AI drops a requirement (omission) or adds a constraint not in
the NL (fabrication), nothing is currently flagged. The requirements dialect enforces tags only
on elements **inside a requirement block**; top-level elements can be untagged (a plain `spec`
has zero discipline).

## 1. CLI

`fslc check --strict-tags [--requirements ids.txt]` (the same flag also applies to verify).
Warnings are attached **only on successful results ok / verified / proved** (not emitted on
violated). The default (no flag) output is byte-for-byte as before.

## 2. Two-way reconciliation (`strict_tag_warnings(spec, requirement_ids=None)`)

1. **`untagged`**: user invariants / actions / leadsTo / reachable that have no `meta`
   (= fabrication candidates with no basis in an NL requirement). Including `MODEL: …` /
   `ASSUME-n: …`, anything with some meta counts as tagged (the lint does not do prefix
   judgment).
2. **`unreferenced_requirement`**: declared requirement IDs that appear in no declaration tag,
   acceptance ID, or forbidden ID (= omission candidates that were forgotten to be formalized).
   - **Declared** = `--requirements ids.txt` (one ID per line) ∪ requirement-block IDs of the
     requirements dialect (auto-collected via `__requirement_ids` — this also catches an
     **empty requirement block** "declared but forgotten to formalize." Collection is essential
     because the trace disappears after expansion).
   - **Referenced** = the `meta.id` of every element ∪ acceptance IDs ∪ forbidden IDs.

## 3. Excluding generated elements (mandatory)

Elements auto-generated by a dialect cannot be tagged by the user, so they are excluded. Not by
name guessing but by an **explicit marker**: dialect expansion emits `("__generated", [names…])`
→ `spec["generated_names"]`.
- `tick` (time block, `_expand_time`).
- `_deadline_*` already inherits requirement meta, a business transition is auto-tagged
  (`{id: transition name, text: "by Actor"}`), and branches splitting inherits requirement
  meta ⇒ exclusion is unnecessary.
- If a user writes `tick` in a spec with no time block, it is linted normally (a same-named
  identifier when time is present is a type error in the first place).

## 4. Ripple (verification engine and Z3 unmodified)

- model.py: `strict_tag_warnings` (~40 lines, spec-dict traversal only) + aggregation of
  `generated_names` / `requirement_ids`. dialects.py: emit `__generated` / `__requirement_ids`.
- cli.py: wire `--strict-tags` / `--requirements` into check / verify. Append
  `{kind:"untagged"|"unreferenced_requirement", element, name(display name), loc, hint}` to
  successful results.

## 5. Tests (tests/test_strict_tags.py)

Untagged per element kind (display name, loc) / MODEL and ASSUME are out of scope / generated
elements excluded (time-generated `tick` not warned) /
user `tick` with no time warns / `--requirements` unreferenced / empty requirement block →
unreferenced / acceptance and forbidden IDs count as referenced / not shown on violated /
default output unchanged.

## 6. Future work / related

`unknown_requirement_id` (ghost requirements, typos) and `--strict-tags error` (CI gate) are v2.
Limited to **existence-level** reconciliation — empty formalization that is tag-only
(**semantic level**) is handled by #6 mutate's requirement stress report
(`empty_formalization`). Roadmap #1.
