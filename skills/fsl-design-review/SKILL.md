# FSL 設計検討 — 設計判断を契約適合の検査に翻訳する手続き

本体は以下の5ステップの手続き。設計原則(SOLID 等)は手続きの出発点ではなく、
各ステップの判断レンズとして登場する。「この設計は良いか」を「この設計は凍結した
契約に適合するか」へ翻訳し、fslc に反例付きで答えさせるのがこのスキルの仕事。

**前提**: FSL の構文・検証コマンド・修復プロトコルは fsl スキル
(SKILL.md + reference.md)に従う。本スキルは手続きと結果の解釈だけを供給する。
動く実例: `specs/bank.fsl` + `bank_impl.fsl` + `bank_refines.fsl`(refinement)、
`seat_booking_*`(状態タグ依存写像)、`specs/bank_system.fsl`(compose)。

## 手続き

### Step 1 — 設計課題を契約の言葉に翻訳する

設計課題はほぼ次の4型に落ちる: 新規設計案の比較 / 変種・拡張の追加 /
既存設計の変更レビュー / 置換可能性の判定。どの型でも特定するものは同じ:

- **開く部分**: 何が変わりうるか(変種・拡張点・変更箇所)
- **閉じる部分 = 契約**: 何が壊れてはならないか
  (不変条件・ガード・外から観測可能な状態)

レンズ — OCP: 「閉じる範囲」をここで決める。DIP: 設計案たちが共通に依存すべき
抽象は何か。契約候補は箇条書きでユーザと認識合わせしてから先へ進むのが推奨
(成果物ファイルは作らない。チャット上でよい)。

### Step 2 — 契約を抽象 spec として凍結する

契約を抽象 `spec` に書き、fsl スキルの標準ワークフローで
`check` → `verify` → `--engine induction` まで通す(proved 推奨)。
契約自体が矛盾していたら設計検討は始まらない — ここで出る `violated` は
設計案ではなく契約の欠陥。

レンズ — DbC: `requires`/`ensures`/`invariant` がそのまま契約。境界や部分操作の
防御は自動検査(`type_bound`/`partial_op`)に任せ、手書きしない。

**抽象層の規律(破ると後で「閉」が崩れる)**:
- 観測可能な状態と契約だけを語る。実装都合の状態を持ち込まない
- **変種を enum で列挙しない**。変種追加のたびに抽象層を編集する羽目になる。
  変種の具体型は詳細層にだけ持たせ、写像の `if c then a else b` で畳み込む

### Step 3 — 設計案を refinement として書く

各設計案・変種を「詳細 `spec` + 写像ファイル」として記述する。抽象 spec の
ファイルは編集しない。詳細層にしかないアクションは `stutter` に対応させる。
モジュール追加型の拡張は refinement ではなく `compose`(既存ファイル無編集で
合成・同期し、横断 invariant を合成層に書く)。

レンズ — LSP: refine 検査の内容(事前条件を強めない・事後条件を弱めない・
不変条件を保つ)は LSP の定義そのもの。SRP/ISP: spec を関心事ごとに分けて
compose する分割の規律として効くが、**検査はされない**(違反しても fslc は
何も言わない)。

### Step 4 — 検査し、結果を設計判断に翻訳する

`fslc refine impl.fsl abs.fsl map.fsl`(+ 各 spec 単体の `verify`)。
結果の機械的な修復手順は fsl スキルの修復プロトコル表に従うが、設計検討では
直す前に必ず「**設計案が悪いのか、契約が過剰なのか**」を判断する — どちらの
結論も設計レビューの正当な成果。

| 検査結果 | 設計上の意味 | 原則語彙での報告 |
|---|---|---|
| refine 成功 | この案は契約を壊さず置換・拡張できる | LSP 充足。OCP の「拡張に開く」が成立 |
| `abs_requires_failed` | 詳細層が抽象層のガードを迂回(統制の近道、実質的な事前条件強化) | LSP 違反 = 置換不能。`impl_trace` が禁じ手の手順 |
| `abs_state_mismatch` | 観測可能な振る舞いが契約と食い違う | LSP 違反(事後条件相当の逸脱)。写像の解釈ミスも疑う |
| `stutter_changed_abs` | 「内部の詳細」と主張した操作が外から見える変化を起こす | カプセル化の破れ。stutter 宣言が嘘 |
| coverage `false` | 発火し得ないアクション = 死んだ拡張点・死んだ手順 | YAGNI / dead code 相当。`blocking_requires` を読む |
| `unknown_cti` | 契約は真だが暗黙の前提が要る | 暗黙の設計前提の発見。ドメインの真実なら補助 invariant として明文化 |

### Step 5 — 証明と判断を分けて報告する

報告は3区分を混ぜない:

1. **証明されたこと** — 契約適合(LSP/OCP 相当)。コマンドと結果を添える
2. **反例** — trace を設計の言葉に翻訳して提示(「○○を飛ばすと△△が成立」)。
   `requirement: {id, text}` があれば要件の意図と突き合わせる
3. **人間の判断に残ること** — SRP/ISP/DRY/デメテル等のコード構造の良し悪し、
   インターフェース設計、命名。FSL は関与しない

**「SOLID 準拠を証明した」とは書かない。** 証明できたのは契約適合のみ。

## 原則 ↔ FSL 機構の対応(参照用)

| 原則 | FSL での扱い | 強度 |
|---|---|---|
| LSP(リスコフ置換) | `fslc refine` の検査内容そのもの | 証明可能 |
| OCP(開放閉鎖) | 抽象 spec 無編集 + 詳細 spec / compose で拡張 → refine で適合検査 | 証明可能 |
| 契約による設計(DbC) | requires / ensures / invariant が言語中核 | ネイティブ |
| DIP(依存性逆転) | 層連鎖(業務 ⊒ 要件 ⊒ 設計 ⊒ 実装)で詳細が抽象に適合 | 構造で体現 |
| SRP / ISP | compose による関心事分割 + `internal` で公開範囲制御 | 規律として支援(検査不能) |
| DRY / KISS / デメテル | コード構造の話で FSL の語彙に翻訳できない | 対象外 |

## 推奨プラクティス(任意 — リスク・重要度に応じて)

- **変種マトリクス**: 変種が複数あるとき、全変種 × 抽象契約の refine を通す
  (「すべての変種が置換可能」の機械検査)。高リスクの契約のみでよい
- **変更レビューへの転用**: 変更前 spec を abs、変更後を impl として refine
  すると「この変更は旧契約を保つか」の検査になる(振る舞い互換の確認)
- **proved まで**: verified(有界)止まりでなく induction まで通すのは、契約が
  長期に安定すべき抽象層では特に価値が高い
